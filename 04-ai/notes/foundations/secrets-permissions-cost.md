# Secret、最小权限与费用上限

## 1. 概念、用途与工程边界

### 定义

Secret 是能够授予系统访问能力的敏感值，包括 API Key、访问 Token、密码、私钥和连接串。最小权限要求身份只获得完成当前任务所需的最少资源、操作和时间范围。费用上限通过账户预算、项目配额、速率限制、请求限制和应用内计量控制模型调用的财务风险。

### 为什么需要

模型 API 通常按输入、输出或工具使用计费。Secret 泄露会导致未授权访问、数据暴露和费用损失；Agent 或循环代码即使没有攻击者也可能因失控重试和长上下文快速增加费用。

### 核心特性

### Secret 生命周期

Secret 需要创建、分发、存储、使用、轮换、吊销和审计。长期静态 Key 风险高于短期凭据。Secret 不应出现在源代码、Git、URL、客户端包、截图、日志和模型上下文中。

### 最小权限

开发、测试、CI 和生产使用不同身份。只读与写入权限分开；不同项目和租户分开；能够限制模型、区域或预算时应显式限制。高风险操作需要服务端再次授权，不能相信模型生成的参数。

### 费用边界

控制分为多层：供应商账户预算与告警、项目级配额、用户/租户速率限制、单请求最大输入输出、Agent 最大步骤和总超时、队列并发、异常检测与熔断。告警不是硬限制，必须确认供应商预算功能是否会自动停止请求。

### 工程使用

上线前检查：

```text
[ ] Secret 由服务端 Secret Manager 或受控运行环境提供
[ ] 本地 Secret 文件已忽略，仓库启用 Secret 扫描
[ ] 开发、CI、生产使用不同凭据
[ ] Key 可立即吊销和轮换
[ ] 日志和错误报告会脱敏
[ ] 设置账户预算、项目告警和应用硬上限
[ ] 每请求限制输入、输出、重试、步骤和总时间
[ ] Usage 按用户、租户、模型、功能和版本记录
[ ] 费用异常能够停止队列或关闭功能
```

发生泄露时先吊销/轮换，不要先花时间清理 Git 历史。随后确认使用日志、影响范围和下游凭据，再清理历史并补检测规则。

### 常见错误与边界

- 把 Secret 放入 `.env` 后认为绝对安全；`.env` 只是本地配置方式，仍可能被误提交、备份或读取。
- 在浏览器调用模型供应商并暴露长期 Key。
- 只设费用告警，不设应用级最大 Token、步骤、并发和速率。
- 多租户共享高权限凭据且不记录租户 Usage，无法归责和隔离。
- 把 Tool 输出和外部文档当可信指令，导致模型诱导高成本或高权限调用。

### 延伸机制

费用治理也影响可靠性：限制过严会造成正常请求失败，因此应区分硬上限、软告警和降级策略。可以在达到阈值后切换低成本模型、减少上下文、禁用非必要 Tool 或转人工处理，但必须记录质量影响。

## 三类控制的执行位置

```mermaid
flowchart LR
    A["用户请求"] --> B["应用认证与授权"]
    B --> C["预算、速率、步骤上限"]
    C --> D["服务端持有 Secret 调用模型"]
    D --> E["Tool 参数再次授权"]
    E --> F["审计与 Usage 结算"]
```

Prompt 可以提示模型不要执行高风险操作，但授权、硬预算和业务不变量必须由受控系统执行。供应商“预算”可能只是告警，是否能阻断请求必须按具体产品验证。

## 生命周期与控制明细

| 对象 | 创建/使用 | 失败响应 |
| --- | --- | --- |
| Secret | 受控生成、最小范围、服务端注入 | 泄露后吊销、轮换、审计 |
| 权限 | 每个请求基于当前身份和资源检查 | 默认拒绝，不由模型补全 |
| 费用 | 预估预留，完成后按 Usage 结算 | 降级、排队、停止或人工审批 |
| Agent 步骤 | 最大步数、总时长、工具白名单 | 保存状态并返回明确原因 |

## 可验证实例

假设租户每日硬预算 10 元，当前已用 9.20 元，请求按最大输入输出估算上限 1.50 元。系统应在调用前拒绝或请求额外授权，而不是先调用再等待账单。若实际 Usage 仅 0.40 元，结算后释放预留差额；并发请求必须原子预留，避免共同越过预算。

## 验证与排错

1. 在测试环境使用无生产权限的独立凭据。
2. 用 Secret 扫描器检查仓库、构建产物和日志样本。
3. 注入 401、403、预算不足和重复 Tool 请求，确认不会自动放宽权限。
4. 对循环任务验证最大 Token、步骤、并发、时长和费用均能终止任务。

## 练习与完成标准

设计一个模型调用网关的安全清单。验收：凭据不进入客户端；开发/生产隔离；按租户授权；调用前预留预算、调用后结算；有硬停止条件；泄露演练能在不修改代码的情况下吊销凭据。

## 完整案例：多租户摘要服务

### 输入

- 租户 A 与 B 使用同一应用，但文档和预算相互隔离。
- 每个租户每日硬预算 100 元，单请求最大预留 2 元。
- 模型 Key 仅后端可读；文档 Tool 需要当前用户身份与资源授权。

### 逐步处理

1. 网关验证用户会话并解析租户，不接受请求体自行声明的租户作为授权事实。
2. 事务性检查租户剩余预算并预留 2 元；预留失败则不调用模型。
3. 后端通过受控 Secret 引用调用模型，日志只记录凭据版本，不记录值。
4. 每次 Tool 调用重新检查用户对文档的读取权限，模型生成的文档 ID 只是不可信参数。
5. 响应完成后依据 Usage 与价格版本计算实际费用，结算并释放差额。
6. 任务取消、超时或流中断仍写计量状态；无法获得最终 Usage 时标记待对账，而非假定零费用。

### 输出

```json
{
  "tenant": "tenant-a-hash",
  "reservation": 2.0,
  "actual_cost": 0.43,
  "released": 1.57,
  "credential_version": "model-prod-7",
  "authorization": "allowed"
}
```

### 验证

- 使用租户 A 身份请求租户 B 文档，必须在 Tool 执行前拒绝。
- 并发发送足以超过剩余预算的请求，只有事务性预留范围内的请求获准。
- 将 Key 放入测试日志，Secret 扫描器必须阻断流水线。
- 轮换凭据后旧版本立即不可用，应用无需重新构建。

### 失败分支

若供应商控制台预算仅发送邮件告警，应用仍可能继续消费。硬停止必须由已验证能阻断的供应商配额或应用网关执行。若日志包含完整 Prompt，还需单独评估其中的个人数据和商业秘密；隐藏 Key 不等于日志安全。

## 边界检查矩阵

1. Secret 创建后记录所有者、用途、范围和到期时间。
2. 分发使用受控引用，不通过聊天、工单和源代码传值。
3. 运行时只向需要的进程提供，不传给模型上下文。
4. 轮换同时更新消费者并验证旧值失效。
5. 授权每次基于当前身份、资源和动作重新检查。
6. Tool 参数来自模型时仍是不可信输入。
7. 租户隔离在数据库查询和缓存键中同时执行。
8. 费用预留使用最坏请求上限并处理并发竞争。
9. 最终 Usage 按价格版本结算，异常请求进入对账。
10. Agent 设置步数、Token、时间、工具和费用多重上限。
11. 告警提供可见性，硬限制负责停止。
12. 事件日志脱敏且访问受限，支持泄露调查。
