# 市场、竞品与商业系统分析

市场与商业分析要回答一条完整链路：哪些主体存在目标任务，谁参与购买，客户当前用什么完成任务，产品如何收费与交付，收入能否覆盖持续服务成本，以及优势能否在客户环境变化后继续成立。

## 一、分析边界

市场规模不是行业报告中的最大数字，竞品不是功能相似的产品清单，商业模式也不等于定价页。三者共享同一个分析单位：一类可识别客户，在明确场景中，为完成任务而付出的全部成本。

本文的前置知识是[目标用户与使用场景](../03-requirements-prioritization/01-target-user-context.md)、[产品北极星指标](../06-metrics-experiments/01-north-star-metric.md)和[相关性与因果关系](../06-metrics-experiments/09-correlation-causality.md)。

分析结果必须注明地域、客户条件、计价周期、币种和证据日期。没有这些边界，“市场有一百亿元”与“客户愿意每月付一千元”之间没有可计算关系。

## 二、从任务定义市场

### 2.1 市场的操作性定义

对产品决策而言，市场是一组同时满足以下条件的潜在交易主体：

1. 具有产品要解决的目标任务。
2. 问题频率或损失足以触发改变。
3. 拥有预算，或能从现有支出中迁移预算。
4. 能够被产品允许的渠道触达和服务。
5. 不被地区、合规、集成或交付约束排除。

“所有企业”不满足这些条件。一个只提供英文界面、必须接入 GitHub Cloud、采用信用卡自助支付的代码审查工具，不能把使用本地代码托管、必须采购招标、要求数据驻留的组织直接计入首年可服务市场。

### 2.2 TAM、SAM 与 SOM

- `TAM`：在既定问题和价值单位下，所有合格主体都采用时形成的理论年收入机会。
- `SAM`：在当前地域、渠道、产品能力和合规条件下能够服务的部分。
- `SOM`：在给定期间内，结合销售产能、转化率、上线速度和竞争条件后可实际获得的部分。

三者不是固定数据库字段。边界改变，数值就改变。计算时应同时保留客户数版本和收入版本：

```text
TAM_accounts = 所有符合问题定义的账户数
TAM_revenue  = Σ(每类账户数 × 该类可验证年价格)

SAM_accounts = TAM_accounts × 产品可用比例 × 地域覆盖比例 × 合规可服务比例
SOM_accounts = min(SAM_accounts × 可达率 × 成交率, 销售与实施年产能)
SOM_revenue  = Σ(预计成交账户 × 合同年值)
```

相乘项不是随意填写的“保守系数”。每个比例都要指向可复核数据，例如现有名单覆盖、渠道曝光、历史线索转化、实施团队每月上线数或法规排除清单。

### 2.3 自上而下与自下而上

自上而下从官方经济、企业或人口统计切出目标行业、地区和规模。它适合检查数量级，并发现被个人样本遗漏的群体。

自下而上从账户名单、目标职位、可触达渠道、价格和成交产能累加。它更接近执行，但容易受名单覆盖不足、重复账户和理想转化率影响。

两个估算应独立进行。若官方数据得到 18,000 家合格企业，而名单法得到 2,300 家，不应直接取平均。先解释差额：行业分类是否过宽、名单是否只覆盖融资企业、母子公司是否重复、地区与规模门槛是否一致。

美国 Census Business Builder 提供地区、行业、企业与人口数据，可用于构建自上而下分母；SBA 的市场研究指引则要求同时分析需求、市场规模、地点、饱和度与竞品价格。官方统计只能证明主体存在，不能证明其具有特定痛点或购买意愿。

### 2.4 用户类型与市场分群

分群变量只有在会改变问题、购买、使用或服务成本时才有意义。常见变量包括：

| 分群变量 | 可能改变的产品判断 | 不足之处 |
| --- | --- | --- |
| 行业 | 业务对象、法规和术语 | 同行业内部流程仍可能不同 |
| 组织规模 | 角色数量、审批和部署方式 | 员工数不等于目标团队规模 |
| 任务频率 | 自动化收益与付费上限 | 高频可能只是流程设计不良 |
| 技术环境 | 集成、数据驻留和实施成本 | 环境声明需要实测确认 |
| 风险等级 | 审批、审计和人工确认 | 风险等级需有一致标准 |
| 购买方式 | 自助、销售辅助或招标 | 购买方式会随合同额改变 |

人口统计特征若不能改变上述决策，不应只因容易获得就成为核心分群。

## 三、还原购买决策链

### 3.1 六类角色

- 使用者：直接完成任务，承担学习与切换成本。
- 管理员：配置账号、权限、数据、集成和生命周期。
- 发起者：把问题带入正式评估，组织试用或方案比较。
- 经济购买者：拥有预算，对投入产出和合同负责。
- 审批者：审查安全、隐私、法务、财务或架构条件。
- 决策者：在候选方案之间作最终选择；可能与经济购买者相同。

这些是决策责任，不是固定职位。小团队中一个人可承担多个角色；大客户中一个角色可由委员会承担。产品分析应记录谁能推进、谁能否决、谁承担失败后果。

### 3.2 购买链路图

```mermaid
flowchart LR
    A["使用者确认任务损失"] --> B["发起者形成候选方案"]
    B --> C["管理员验证部署与治理"]
    C --> D["安全、法务、财务审批"]
    D --> E["经济购买者确认预算"]
    E --> F["决策与合同"]
    F --> G["实施、采用与续约"]
    G -->|"价值证据"| E
```

首次购买与续约不是同一个决策。首次购买可依赖方案承诺和试用，续约应能看到采用、任务结果、风险事件、支持成本和未来替代成本。

### 3.3 购买标准矩阵

| 角色 | 关键问题 | 可验证证据 | 常见否决条件 |
| --- | --- | --- | --- |
| 使用者 | 是否更快、更准、更少返工 | 真实任务试用与完成率 | 操作变慢或破坏现有流程 |
| 管理员 | 能否批量治理与恢复 | 权限、SCIM、日志、导出实测 | 无法撤销、迁移或审计 |
| 安全/法务 | 数据如何处理 | 数据流、合同、留存与分包清单 | 处理目的或责任不清 |
| 财务 | 支出是否可预测 | 报价、上限、账单模拟 | 用量无法预算或对账 |
| 经济购买者 | 是否产生净价值 | 基线、试点结果、总拥有成本 | 收益依赖持续额外人力 |

只优化使用体验无法自动通过采购；只满足审批也不能创造采用。产品方案要使每个关键角色获得与其责任匹配的证据。

## 四、竞品与替代方案

### 4.1 直接竞品的判定

直接竞品同时争夺相近目标客户、预算和任务结果。界面相似但购买预算不同的产品可能不是直接竞品；界面完全不同但替代同一支出的方案可能是强竞争者。

比较必须固定任务。例如比较“新成员在十分钟内找到项目发布规则”，而不是比较“是否有 AI、搜索、标签”。任务基准包含：

- 入口与前置条件。
- 完成标准和允许错误。
- 主流程耗时与人工步骤。
- 权限、失败、恢复和导出。
- 采购价格与实施成本。
- 对管理员和购买者的影响。

### 4.2 间接替代与不消费

间接替代包括表格、邮件、群聊、人工外包、自建脚本、多个工具组合以及维持现状。它们可能体验不佳，却已经被预算、习惯和组织责任吸收。

“不用软件”也有成本：处理时间、等待、错误、重复工作、机会损失和风险暴露。但这些成本必须用现有记录估算，不能为了证明方案价值而全部货币化。

### 4.3 总拥有成本

```text
TCO = 订阅或许可
    + 实施与迁移
    + 集成开发
    + 管理与培训
    + 运行用量
    + 支持与故障补救
    + 退出与数据迁移
```

对自建方案还要计算维护、值班、依赖升级和关键人员风险；对人工方案要计算排队时间、质量差异和扩容速度。

### 4.4 为什么不复制竞品功能

竞品功能可能服务于不同客户、商业模式或遗留架构。复制前至少回答：

1. 它解决什么具体任务，证据是什么？
2. 它为哪个角色创造价值，谁为此付费？
3. 是否依赖竞品已有数据、网络、渠道或运营能力？
4. 会增加哪些长期权限、兼容、支持和迁移责任？
5. 删除该功能后，核心价值是否仍然成立？

若只能观察到功能存在，无法观察采用、结果和成本，应把“值得复制”标为假设。

FTC 的竞争指引强调竞争不仅来自同类公司，也来自可能约束价格、质量与选择的替代关系。产品分析中的市场边界同样不能为得到有利结论而随意缩窄。

## 五、定价与收费

### 5.1 四个不同对象

- 价值单位：客户获得价值随什么增长，例如成功处理订单数。
- 计费单位：账单按什么计量，例如席位、API 调用或存储量。
- 价格函数：单位数量如何映射到金额。
- 套餐：功能、额度、支持和治理能力如何组合。

价值单位与计费单位可以不同，但偏离越大，客户越难理解公平性，企业也越难预测收入与成本。

### 5.2 常见收费结构

| 结构 | 适用条件 | 优点 | 主要风险 |
| --- | --- | --- | --- |
| 固定订阅 | 价值和成本较稳定 | 易预算、易解释 | 重度客户可能亏损 |
| 按席位 | 价值随协作人数增长 | 可预测、易采购 | 抑制邀请和全员采用 |
| 按用量 | 用量与价值、成本相关 | 低门槛、自然扩张 | 账单波动、客户抑制使用 |
| 分层套餐 | 分群需求存在离散差异 | 自助选择与升级路径 | 人为围栏导致错配 |
| 基础费加用量 | 存在平台价值和变动成本 | 平衡承诺与扩张 | 解释、计量和对账复杂 |
| 按结果 | 结果清晰且可归因 | 价值表达强 | 归因争议与逆向选择 |

Stripe 的定价资料把价值指标、定价模型、套餐层级和计量分开，并明确按席位、按用量、混合与按结果方案的适用条件。它是支付供应商的实践资料，不构成所有产品都应采用某种模型的规则。

### 5.3 阶梯、分段与封顶

假设 API 价格如下：前 10,000 次包含在 500 元基础费中，之后至 100,000 次每次 0.02 元，超出部分每次 0.015 元。

80,000 次的分段计费为：

```text
bill = 500 + (80,000 - 10,000) × 0.02
     = 1,900 元
```

若错误地把 80,000 次全部按 0.02 元计算，会得到 2,100 元。产品文档必须写清包含额度、计量时区、去重、失败请求、迟到事件、舍入、退款与封顶。

### 5.4 定价验证

定价实验不能只看转化率。至少同时观察：

- 目标分群的付费转化与销售周期。
- 首期账单争议和退款。
- 用量分布、重度账户毛利与客户预算偏差。
- 升级、降级、扩张和取消原因。
- 被价格结构抑制的核心使用行为。

降价提高转化但降低贡献毛利，或按席位收费导致团队只买少量共享账号，都不能只用新增客户数判断成功。

## 六、成本与单位经济

### 6.1 成本分层

- 获客成本：营销、销售工资与佣金、渠道费、售前和试点。
- 交付成本：实施、迁移、培训、客户专属集成和上线支持。
- 服务成本：云资源、第三方 API、模型推理、支付费、支持和值班。
- 留存成本：客户成功、续约、折扣、补救和服务信用额。
- 固定投入：通用研发、管理与基础平台；分析边际决策时仍需防止长期忽略。

### 6.2 CAC

```text
CAC = 某获客 cohort 的销售与营销成本 / 该 cohort 新增付费客户数
```

必须明确成本归属窗口和客户成熟窗口。一个季度花费、另一个季度成交时，直接用同季度分子分母会错配。企业销售还应分别报告首签 CAC、扩张成本和渠道分成。

### 6.3 毛利与贡献毛利

```text
Gross profit = Revenue - Cost of revenue
Gross margin = Gross profit / Revenue

Contribution margin = Revenue
                    - 与该账户直接相关的服务成本
                    - 可变交付与支持成本
                    - 交易及渠道变量成本
```

会计报表中的收入成本分类与产品单位经济口径不一定相同。产品决策必须同时保留财务认可口径和操作分析口径，不能把内部估算称为法定毛利。

### 6.4 模型与基础设施成本

AI 产品的单位成本至少按请求类型拆分：输入、输出、检索、工具调用、重试、缓存未命中、审核和人工接管。平均每请求成本会掩盖长文本或代理循环的尾部损失。

```text
每成功任务成本 = (全部模型与工具成本 + 失败重试 + 人工补救) / 成功任务数
```

分母用“成功任务”而不是“请求”，才能反映为了完成一次用户目标发生的重试和失败。

## 七、订阅业务指标

### 7.1 MRR 与 ARR

`MRR` 是按月标准化的经常性合同金额。年付 12,000 元合同可标准化为 1,000 元 MRR；一次性实施费不应进入 MRR。

```text
Ending MRR = Beginning MRR
           + New MRR
           + Expansion MRR
           - Contraction MRR
           - Churned MRR
```

`ARR` 常以期末经常性金额年化，但公司定义并不统一。简单月订阅可用 `ARR = MRR × 12`；季节性、用量型、短期或非标准合同不能机械年化。

公开公司在 SEC 披露中通常同时给出自己的 ARR/NDR 定义，并警告其他公司可能采用不同计算方法。因此跨公司比较前必须先对齐：是否包含用量、是否包含已到期但在续约谈判中的合同、汇率、收购和一次性金额。

### 7.2 ARPU

```text
ARPU = 期间收入 / 期间平均活跃付费单位数
```

“单位”可以是账户、席位或用户，必须固定。期末客户数替代期间平均数会在快速增长时高估或低估。ARPU 上升可能来自涨价、客户结构变化、扩张或低价客户流失，不能单独解释为产品价值提高。

### 7.3 Churn 与留存

```text
Logo churn = 期间流失客户数 / 期初客户数
Gross revenue retention = (期初经常性收入 - 流失 - 收缩) / 期初经常性收入
Net revenue retention = (期初经常性收入 - 流失 - 收缩 + 扩张) / 期初经常性收入
```

NRR 可超过 100%，因为包含期初客户的扩张；新客户收入不能进入分子。月度 logo churn 不能直接乘 12 当作年流失，稳定月流失 `c` 下的年留存为 `(1-c)^12`。

例如月流失率 3%，年留存约为 `0.97^12 = 69.4%`，对应年流失约 30.6%，不是 36%。真实流失若随客户年龄变化，应使用 cohort 生存曲线。

### 7.4 LTV

常见稳态近似：

```text
LTV ≈ ARPA × Gross margin / Revenue churn rate
```

该式要求流失近似稳定、单位与周期一致、未来毛利可代表当前。早期产品、合同集中、负扩张、季节性和长期承诺会破坏假设。

更稳健的方法是按 cohort 预测每期贡献毛利并折现：

```text
LTV = Σ(t=1..T) [预计存续概率_t × 预计账户贡献毛利_t / (1+r)^t]
```

LTV/CAC 不是事实标签，而是模型结果。应同时报告回收期、假设区间和 cohort 实际成熟度。

### 7.5 CAC 回收期

```text
CAC payback months = CAC / 每账户月度贡献毛利
```

若 CAC 为 12,000 元，月收入 1,500 元，毛利率 80%，每月另有 200 元可变支持成本，则月贡献毛利为 `1,500×80%-200=1,000` 元，回收期 12 个月。

如果忽略支持成本会得到 10 个月。企业实施密集、AI 推理密集或渠道分成高的产品尤其需要贡献毛利版本。

## 八、差异化、护城河与迁移成本

### 8.1 差异化

差异化是目标客户能感知、愿意选择且竞争者不能低成本消除的价值差异。它要通过任务结果、购买理由或留存行为验证，而不是来自团队自我描述。

差异可以来自工作流深度、独有分发、数据反馈、集成覆盖、可靠性、合规资质、品牌信任、成本结构或生态互补。

### 8.2 可持续性检验

对每项优势询问：

- 竞争者复制界面需要多久？
- 复制结果需要哪些数据、渠道、资质、规模或组织能力？
- 优势会随使用增强，还是随客户增长变成负担？
- 技术或政策变化是否会清除优势？
- 客户能否验证差异，还是只能相信营销陈述？

功能本身通常可复制；围绕功能形成的数据质量、工作流嵌入、运营能力和分发关系可能更难复制。

### 8.3 迁移成本与不当锁定

迁移成本包括数据导出与映射、集成重建、权限重配、流程培训、并行运行、合同退出和历史审计连续性。

健康迁移成本来自产品嵌入真实工作和积累结构化资产；不健康锁定来自阻止导出、模糊格式、惩罚性条款或人为不兼容。后者会降低初次采购信任，并带来监管与声誉风险。

产品应提供可用导出、删除、权限和迁移文档，同时通过持续价值留存客户。

## 九、完整案例一：研发团队协作 SaaS

### 9.1 输入

产品为 50—300 人研发组织提供发布检查与变更协作。团队从官方企业统计切出目标行业和规模，再用 1,200 个可识别账户名单校验。现有销售每名每季度最多完成 12 个有效评估，实施团队每月最多上线 8 家。

过去两个季度的数据：160 个有效机会，40 个试点，16 个首签；平均首年合同 72,000 元；每个首签分摊销售营销成本 30,000 元；标准实施 8,000 元；月云和支持成本每账户 1,200 元。

### 9.2 市场与购买链

官方分母切分得到 9,000 家候选企业；技术与地区约束后 SAM 为 3,400 家。按销售 4 人、每人每季度 12 个有效评估，年评估产能为 192 个。

历史“有效机会→首签”转化为 10%，仅按产能估算首年约 19 家。即使 SAM 很大，SOM 仍被销售与实施能力限制。

角色链为：研发效能负责人发起，开发者使用，平台团队管理，安全与采购审批，CTO 承担预算。试点必须同时验证开发者任务耗时和管理员权限/审计，否则只能证明单用户体验。

### 9.3 竞品与替代

团队选择三个直接竞品，并加入 GitHub Action 自建脚本和人工发布清单两个替代方案。固定任务是“变更进入生产前完成四项检查并保留审计证据”。

比较后发现，自建脚本许可成本低，但维护、规则分叉和值班归研发平台团队；人工清单在低频时合理，却在多团队并行发布时出现等待和遗漏。

竞品的 AI 总结功能不在首版复制范围，因为试点主要损失来自审批等待和证据分散，且总结不能替代确定性检查。

### 9.4 定价与单位经济

团队比较按开发者席位与按月发布次数。席位容易采购但抑制全员接入；纯用量与发布次数相关，却让客户预算波动。

最终试点采用“平台基础费 + 活跃开发者分层”的混合结构，并设置年度用量回顾，不对偶发峰值即时加费。

首年收入 72,000 元；估计服务成本 `1,200×12=14,400` 元；标准实施 8,000 元。首年贡献毛利为 49,600 元，首签 CAC 30,000 元，可在首年覆盖，但尚未计入高风险客户的专属集成。

### 9.5 输出与验证

首年目标不是“获得 SAM 的 1%”，而是完成 18—20 家可持续首签，并证明标准实施不超过 10 个工作日、90 日核心团队采用达到门槛、首年贡献毛利为正。

失败分支：若客户必须定制审批引擎，实施成本从 8,000 元升至 60,000 元，原价格和产能模型失效。团队要么提高企业套餐价格并限制定制，要么排除该分群，不能用未来续约假设掩盖首年亏损。

## 十、完整案例二：生成式 AI 客服产品

### 10.1 输入

产品为电商客服生成回复草稿。候选客户按月工单量、语言、现有客服系统和人工审核能力分群。价值单位是“达到质量标准并被客服采用的回复”，模型成本却按输入输出 Token、检索和工具调用产生。

一个月试点包含 100,000 张工单：60% 可自动生成草稿；草稿采纳率 70%；每次生成平均模型与检索成本 0.08 元；质量审核与失败补救月成本 8,000 元。

### 10.2 有效任务与成本

生成调用数为 60,000，模型与检索成本 4,800 元。被采纳草稿为 42,000 条。

```text
每采纳草稿成本 = (4,800 + 8,000) / 42,000
               ≈ 0.305 元
```

若用全部工单做分母，会得到 0.128 元并严重低估有效任务成本。尚未生成、被拒绝和需要人工补救的路径必须保留。

### 10.3 定价比较

按客服席位收费易预算，但一个重度客服的工单量可能是另一个的十倍；按 Token 收费与客户价值难以对应；按“生成草稿”收费会把低质量输出也计费。

团队选择基础订阅加“被采纳草稿”用量，并约定采纳定义、撤回窗口、重复工单和批量操作。高风险退款、法律投诉不进入自动草稿范围。

### 10.4 商业指标

试点月收入 30,000 元，直接模型、检索、审核和支持成本 16,000 元，贡献毛利率约 46.7%。若只计算模型 API 的 4,800 元，会错误报告 84% 的近似毛利率。

产品计划通过缓存和更小模型降低成本，但不能牺牲严重错误率。成本优化要在同一质量门槛下比较；质量下降导致更多人工补救时，总成本可能上升。

### 10.5 差异化与失败分支

通用模型能力不是稳定差异化。该产品可能形成的优势是客服系统集成、商家政策结构化、逐条证据、审核工作流和真实纠错数据。

失败分支：若客户降低人工审核以节省成本，严重错误的后果会改变产品风险边界。团队必须在系统内限制高风险动作并提供人工确认，不能把合同中的“客户负责审核”当成唯一控制。

## 十一、调试与验证

### 市场估算检查

- 随机抽取 20 个计入 SAM 的账户，检查行业、规模、地区和技术条件。
- 对比官方分母与名单分母，解释差异而非取平均。
- 将转化率降低一半、销售周期延长一倍，检查现金与产能边界。
- 把实施产能加入 SOM 上限，避免只有销售没有交付。

### 指标检查

- MRR 是否排除一次性实施费、税和非经常项目。
- NRR 分子是否只包含期初 cohort，是否错误加入新客户。
- ARPU 的单位和平均客户数口径是否一致。
- CAC 是否考虑成交滞后、销售工资、佣金和渠道分成。
- LTV 是否报告毛利、流失稳定性和预测期限。

### 定价检查

- 用低、中、高三种用量生成真实账单。
- 测试时区边界、重复事件、失败事件、退款和封顶。
- 检查价格是否抑制产品必须发生的协作或采用。
- 将重度客户的服务成本放入最坏情形，不只看平均值。

### 竞争检查

- 每个结论标注来自公开文档、实际试用、合同报价还是推断。
- 固定任务、账号级别、地区和观察日期。
- 把人工、自建、组合工具和不改变纳入比较。
- 对“不可复制”写出所需时间、资产和证据，不用形容词代替。

## 十二、综合练习

选择一个 SaaS 或 AI 产品，提交市场与商业决策包。

### 交付物

1. 目标任务、客户条件和排除条件。
2. 独立的自上而下与自下而上 TAM/SAM/SOM，附每个输入来源。
3. 使用者、管理员、发起者、经济购买者和审批者矩阵。
4. 三个直接竞品、两个替代方案和“不改变”基线。
5. 固定任务下的 TCO 比较，包含迁移和退出成本。
6. 至少三种定价结构及三档真实账单。
7. MRR、ARR、ARPU、CAC、LTV、Churn、毛利的口径表和算例。
8. 一项差异化的可持续性检验与一项不应复制的竞品功能。
9. 两个失败分支：一个成本越界，一个价值或风险越界。

### 验收标准

- [ ] 市场分母来自可追溯统计或账户清单，不来自无边界行业总额。
- [ ] SOM 受渠道、转化、销售和实施产能约束。
- [ ] 每个购买角色都有目标、证据和否决条件。
- [ ] 竞品比较固定同一任务、环境和完成标准。
- [ ] 定价单位、价值单位与成本单位分别定义。
- [ ] 订阅指标注明分母、周期、币种、是否包含扩张与一次性费用。
- [ ] LTV 和 CAC 回收期展示假设与敏感性，不只给单一比值。
- [ ] 差异化能够被客户观察，并说明竞争者复制所需资产。
- [ ] 迁移成本没有依赖阻止导出或人为制造不兼容。
- [ ] 结论同时通过用户价值、商业价值、服务能力和风险护栏。
