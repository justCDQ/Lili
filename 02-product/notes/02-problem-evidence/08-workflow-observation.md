# 工作流观察：还原当前任务

工作流观察记录一个真实结果如何通过人员、纸张、消息、表格和多个系统完成。它从触发开始，以可验证业务结果结束，关注动作、交接、等待、返工、例外和责任。目的不是尽快找到软件功能，而是判断哪些步骤创造价值、哪些步骤只在弥补流程缺口，以及哪些风险不能自动化。

## 一、观察对象是端到端任务

“填写报销表”只是动作，“员工提交有效费用并完成入账”才是任务。若只观察软件页面，会漏掉收集票据、询问项目编号、主管审批、财务核对、退回补充和归档。

任务边界应包含：

- 真实触发：什么事件让流程开始；
- 目标角色：谁对结果负责；
- 输入：材料、权限和前置状态；
- 完成：什么产物或状态可证明成功；
- 失败：什么条件算阻断、退回或放弃；
- 下游：谁使用结果，错误如何暴露。

GOV.UK 的上下文观察指南强调在日常环境中使用真实数据、文档和设备观察任务，这有助于看到工具之外的阻碍和变通。

## 二、正式流程、描述流程与实际流程

### 正式流程

制度、SOP 或帮助文档要求怎样执行。它能证明规范，不证明每次实际如此。

### 描述流程

执行者回忆自己通常怎样做。记忆会省略习惯动作、等待和例外。

### 实际流程

在具体任务中观察到的动作、状态和产物。观察也会受研究者在场影响，不能当不受干扰的完整现实。

三者应并列。实际绕过正式步骤可能是违规、规则过时、工具不支持或特殊个案；观察者不能只凭动作判断原因。

## 三、观察、推断与解释请求分开

高质量记录先写可见事实：

```text
观察：财务复制发票上的项目编号，在表格和财务系统各输入一次。
推断：重复输入可能增加耗时和转录错误。
未知：两个系统是否允许安全集成，第二次输入是否承担独立复核。
```

GOV.UK 的记录指南要求笔记尽量写看到和听到的内容，而不是个人解释。需要理解原因时，可以在不打断关键动作的时点询问；回答属于参与者陈述，与行为观察分开。

## 四、观察方式及其影响

### 静默观察

最少打断自然流程，但容易误解不可见判断。适合先记录动作和环境。

### 偶尔追问

在交接或任务结束后询问“刚才为什么切换到表格”，能补充解释，但问题会影响后续行为。

### 边做边说

参与者解释每一步，可获得深层规则，但任务会比平时更慢、更显性，时间数据不能直接当自然基线。

### 自我观察

个人学习者可观察自己的任务。优点是权限清楚、成本低；缺点是无法同时自然执行和完整记录，且自己的经验不代表其他角色。可用屏幕事件日志、产物时间戳和任务后立即复盘降低遗漏。

研究方法、干预时点和已知影响必须写入记录。

## 五、知情同意、权限与最小记录

观察他人前明确：研究目的、观察范围、记录方式、谁可访问、保存多久、如何撤回。后来加入现场的人也需要单独确认。照片、录屏、音频和文档副本需要对应许可，不能因为参与者同意被观察就默认同意所有记录。

敏感工作流可能暴露客户、健康、财务、员工绩效、密码和商业秘密。采用：

- 不记录与问题无关的正文；
- 用角色和任务编号代替身份；
- 截图前遮盖账户和业务数据；
- 不收集密码、令牌和私密消息；
- 将原始记录与公开分析分开；
- 到期删除，并能定位参与者材料以执行撤回；
- 无授权时改为观察自己的任务或公开演示。

观察不能妨碍安全、照护、交易或关键业务。出现紧急情况时停止研究，按现场流程处理。

## 六、设计观察计划

### 研究问题

问题要能由行为回答，例如“审批总历时主要消耗在哪些等待与返工”，而不是“应该做什么功能”。

### 任务变体

至少考虑正常、边界和失败：低/高金额、材料齐全/缺失、普通/管理员、首次/熟练、单人/跨角色。只看一个成功案例无法理解恢复。

### 记录模板

```json
{
  "workflow_instance": "expense-07",
  "observed_on": "2026-07-17",
  "method": "自我观察加产物时间戳",
  "trigger": "收到需报销的有效发票",
  "completion": "费用进入已入账状态",
  "steps": [
    {
      "seq": 1,
      "role": "申请人",
      "action": "从邮件下载发票",
      "tool": "邮件",
      "active_seconds": 42,
      "wait_seconds": 0,
      "output": "本地PDF",
      "evidence": "文件创建时间"
    }
  ],
  "data_classification": "去标识教学数据"
}
```

## 七、时间、交接与返工的口径

### 总历时

从触发到完成的日历时间，包含夜间、排队和等待。

### 主动处理时间

角色实际执行动作的时间。边做边说会放大该指标，应注明观察方式。

### 等待时间

任务已准备但等待人员、系统、材料或时间窗口。区分必要等待与无明确负责人等待。

### 交接

责任或信息从一个角色、工具或组织转移。记录输入合同、输出格式、接收确认和失败责任。

### 返工

为纠正、补齐或重复执行而返回已完成步骤。重复录入不一定是返工，可能是故意双重控制；必须验证目的。

```text
总历时 = 完成时间 - 触发时间
触达效率 = 主动处理时间 / 总历时
返工率 = 含返工的任务实例 / 符合条件的任务实例
```

指标只在相同任务边界、工作日规则和完成定义下比较。

### 时间证据与测量误差

手工秒表、系统时间戳、邮件时间和参与者回忆具有不同精度。优先使用动作产物和系统事件重建关键节点；无法观察的等待标为区间，不用伪精确秒数。跨系统时先确认时区和时钟偏差。

观察者记录可能漏掉并行动作。例如参与者等待导出时处理另一项工作，这段时间既是任务等待，也不一定是人员闲置。流程成本应从任务历时和人员投入两个视角分别报告。

### 记录一致性

多人观察前用同一短视频或公开任务校准：分别记录动作、等待、交接和解释，再讨论差异并修订编码词典。正式分析时保留原始记录，不在达成共识后覆盖不一致。若两个观察者对“返工”判断不同，应回到预先定义和产物证据，而不是按职位决定。

自我观察无法做观察者一致性检查，可以在任务结束后用屏幕事件、文件版本和消息时间戳核对回忆，并让另一位读者仅根据去标识记录重建流程。重建失败说明证据不足，不表示流程本身不存在。

## 八、交接合同

每个交接点检查：

| 字段 | 问题 |
| --- | --- |
| 发送者 | 谁对输出正确负责 |
| 接收者 | 谁确认收到并继续 |
| 输入 | 哪些字段、格式与证据必需 |
| 完成信号 | 发送、送达还是接收确认 |
| 时限 | 何时应处理，超时如何升级 |
| 权限 | 谁能查看和修改 |
| 异常 | 缺失、冲突、重复如何处理 |

大量流程问题发生在“已发送但没人确认”。只画页面步骤会漏掉责任状态。

## 九、判断自动化候选

观察到重复动作后，先问：

1. 这一步是否仍有必要；
2. 它是否承担独立检查、合规或风险控制；
3. 输入规则是否稳定且可机器读取；
4. 例外能否识别并安全转人工；
5. 自动化失败是否可发现、恢复和审计；
6. 数据权限是否允许跨系统流动；
7. 减少的人工成本是否转成维护与监控成本。

删除不必要步骤优先于自动化。把坏流程原样软件化，会让错误更快、更难发现。

## 十、完整案例：跨工具报销

### 1. 输入与边界

任务从员工收到有效发票开始，到费用进入财务系统“已入账”结束。观察自己的 8 次低风险报销，并经授权查看财务退回通知；不记录同事身份和真实金额。

任务变体：6 次材料齐全，2 次缺项目编号；其中 1 次金额进入额外审批。正式 SOP、表单字段、邮件时间戳和财务状态构成证据。

### 2. 观察结果

以下是教学场景数据：

| 阶段 | 角色 | 主动时间中位数 | 等待中位数 | 异常 |
| --- | --- | ---: | ---: | --- |
| 收集票据 | 申请人 | 4 分钟 | 0 | 发票分散在邮件 |
| 填表 | 申请人 | 8 分钟 | 0 | 项目编号需另查 |
| 主管审批 | 主管 | 3 分钟 | 1.2 天 | 无到期提醒 |
| 财务复核 | 财务 | 6 分钟 | 0.8 天 | 两次退回 |
| 录入系统 | 财务 | 5 分钟 | 0 | 金额与编号重复输入 |
| 入账 | 系统 | 0 | 0.5 天 | 批处理等待 |

### 3. 事实与推断

**事实**：总主动处理约 26 分钟，等待远大于主动处理；8 次中 2 次因项目编号缺失退回；金额和编号在表格与财务系统重复输入。

**推断**：项目编号可发现性和审批责任状态可能是主要摩擦。重复输入可能增加成本，但尚不知道是否承担复核控制。

**假设**：在提交前验证项目编号、显示审批负责人和到期状态，能减少退回与无主等待。

### 4. 不直接自动化的步骤

高金额审批涉及预算责任，不能因耗时高就删除。财务复核可能承担欺诈和合规控制；需要确认控制目标，再判断是否由规则预检查辅助，而不是完全自动批准。

### 5. 验证

1. 再观察 4 次材料缺失和高金额案例；
2. 用正式 SOP 确认重复输入是否为独立复核；
3. 统计项目编号缺失的符合条件任务分母；
4. 用原型在提交前验证编号，不改变真实审批；
5. 比较退回、主动时间和错误，同时监控错误通过；
6. 检查跨系统传输的权限、审计和删除。

### 6. 失败分支

- 若观察者在场使主管立即审批，等待时间标记为受干预，不作自然基线；
- 若只看到正式表单而未看到邮件补材料，继续观察到真实完成；
- 若重复输入承担独立核对，不能简单删除；
- 若缺失项目编号只发生在一种项目类型，结论按类型限定；
- 若无法获得他人记录许可，改用自己的流程与去标识产物；
- 若原型验证阻断合法例外，提供人工继续路径并停止扩展。

## 十一、从流程图到证据结论

流程图只是表达。每个节点应链接观察实例、时间戳或文档；未知节点显式标记。BPMN 可用于表达事件、任务、网关和参与者，但使用标准符号不能让未经验证的流程变真实。

输出建议包含：当前流程、正式与实际差异、时间分解、异常清单、交接合同、不可自动化控制、三个待验证机会。不要从一次观察直接写最终产品需求。

## 十二、完成检查与练习

- 从真实触发到可验证结果；
- 正式、描述和实际流程分开；
- 方法及观察者影响已记录；
- 动作、状态、推断和参与者解释分开；
- 主动、等待、交接和返工口径一致；
- 正常、边界与失败实例均有覆盖；
- 同意、记录权限、脱敏、保留和撤回明确；
- 自动化前确认步骤目的和人工恢复；
- 结论限定样本和环境。

练习：观察一个真实跨工具任务至少 5 次，包含一次失败。输出端到端图、原始步骤表、时间分解和交接合同。完成标准是读者能从证据复算总历时，并指出哪个步骤不应直接自动化。
