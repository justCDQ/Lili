# 多来源交叉验证

交叉验证是用不同来源、方法、时间或环境检验同一个可证伪主张。目标不是堆积链接或多数表决，而是识别每类证据的盲区、共同来源、支持边界和冲突。多个来源一致可以提高可信度，但只有在它们真正独立、口径对齐且直接支持主张时才有意义。

## 一、先把结论改写成主张

“导入有问题”无法验证。主张至少限定：

- 角色和任务；
- 产品、版本、套餐与平台；
- 输入和必要环境；
- 可观察结果；
- 时间窗；
- 对照或基线；
- 什么证据会推翻它。

示例：

> 在 2026-07-01 至 07-14，使用桌面端 v3.2 导入包含空日期字段的 CSV 时，管理员任务失败率高于 v3.1；若固定输入与环境的对照没有差异，则该主张不成立。

把“v3.2 解析器改坏了”当初始结论会把根因混入问题主张。

## 二、证据来源与方法不是一回事

来源是评论、Issue、文档、日志或状态页；方法是公开陈述分析、任务观察、受控复现、行为统计或系统诊断。两篇文章引用同一个 Issue，来源仍只有一个；日志图和基于同一日志的报告也不独立。

常见证据角色：

| 证据 | 擅长支持 | 主要盲区 |
| --- | --- | --- |
| 评论/社区 | 症状语言、环境、后果 | 自选择、无总体分母 |
| Issue | 复现、维护者状态、版本链 | 仓库用户偏差 |
| 官方文档 | 期望规则、限制、发布声明 | 不证明实际成功 |
| 行为分析 | 事件规模、分段、变化 | 埋点缺失、难解释意图 |
| 日志 | 系统状态和机制 | 未记录事实不可见 |
| 受控复现 | 指定条件下因果线索 | 环境覆盖有限 |
| 工作流观察 | 上下文、变通和交接 | 样本小、观察者影响 |

交叉验证通常需要不同盲区的组合，而不是同类材料数量。

## 三、判断独立性

### 原始来源

追溯评论、新闻、报告和摘要的最早证据。十条转载同一公告只能支持公告被传播。

### 数据生成过程

两个看板可能来自同一事件表；客服摘要可能引用同一状态事故；应用商店评论可能复制社区原帖。记录 lineage。

### 人与事件

同一组织的销售、实施和支持记录可形成生命周期链，但不算三个独立组织。同一全局事故中的十个报告者可证明影响多个任务，却不是十种原因。

### 分析独立性

两个分析者使用同一数据和相同代码不构成来源独立，但可以发现分析错误。独立复算与独立数据要分开记录。

## 四、对齐口径后再比较

证据冲突常来自不同：

- 事件定义和分母；
- 产品版本、渠道和功能开关；
- 角色、套餐与权限；
- 设备、地区、语言和网络；
- 时间窗、时区和延迟；
- 样本选择与去重；
- “发布”“可用”“启用”“任务成功”的状态。

先做对齐表。无法对齐的证据标记为不直接可比，不能为得到一致而删除。

## 五、支持、反证、冲突与无关

每项证据相对于具体主张分类：

```text
support：若可信，主张更可能成立
counter：若可信，主张更可能不成立或范围更窄
conflict：与另一证据在同一口径下不一致
neutral：相关但不能区分主张真假
```

发行说明称“更新解析器”对“v3.2 失败率更高”是相关中性证据；它不报告结果。旧版用户成功也可能是对照支持，而不是反证。分类依赖主张。

## 六、建立证据矩阵

```json
{
  "claim_id": "CLM-IMPORT-032",
  "claim": "v3.2对空日期CSV的导入失败率高于v3.1",
  "scope": "桌面端管理员，2026-07-01至07-14",
  "falsifier": "固定输入与环境下版本对照无差异",
  "evidence": [
    {
      "id": "EV-01",
      "method": "行为分析",
      "stance": "support",
      "independence_group": "analytics-import-events-v2",
      "directness": "direct",
      "limitations": ["3.1样本较少"]
    },
    {
      "id": "EV-02",
      "method": "受控复现",
      "stance": "support",
      "independence_group": "local-fixture-A",
      "directness": "direct",
      "limitations": ["只测试Windows"]
    }
  ],
  "current_confidence": "medium",
  "unresolved": ["macOS和网页端是否受影响"]
}
```

`independence_group` 用于避免把同一底层数据重复计权。

### 建立证据依赖图

矩阵适合阅读，依赖图适合发现伪独立。节点包括原始数据、转换活动、报告与主张；边表示 derived_from、supports、counters 和 conflicts。W3C PROV 提供实体、活动和责任主体的来源表达，个人项目无需实现全部标准，但应能回答“这张图由哪张表、哪个查询和哪个版本生成”。

```text
raw_import_events_v2
├── query_failure_rate_202607.sql
│   ├── dashboard_import_health
│   └── EV-ANALYTICS-07
└── support_export_joined
    └── EV-SUPPORT-CORRELATION-03
```

后两项都依赖同一事件表，不能当完全独立。若事件 schema 发现错误，依赖图能定位需撤回的证据和主张。

### 权威性与主张类型

官方文档对“设计规则是什么”通常直接，对“用户是否成功”则间接；用户行为对任务结果直接，对内部根因通常间接。不存在对所有主张都最高等级的来源。应根据主张类型评价直接性，而不是用固定权威排名。

### 验证停止规则

继续收集证据也有成本。可在以下条件停止当前轮次：主张已被足够直接、独立证据限定；剩余不确定性不会改变可逆决策；风险有守护和回滚；下一证据成本明显高于信息价值。

高风险、不可逆或涉及权利的决定不能仅因时间紧而降低证据要求。停止意味着按当前边界行动并设复审，不是宣称绝对确定。

### 负结果和未观察到

受控测试未复现，只能在测试条件下形成反证。日志没有记录可能来自事件未发生，也可能是漏报。负结果必须附检测能力：样本、覆盖、埋点质量和可观察窗口。没有检测能力的“未发现”接近未知，而不是强反证。

## 七、可信度来自推理链

提高可信度的因素：直接、可复现、来源清楚、事件定义稳定、有合适分母、环境匹配、数据质量通过、含反证、时效适用。

降低可信度的因素：转载、选择性样本、共同数据、版本未知、权限不明、日志漏报、主张过宽、冲突未解释。

不建议将证据数量相加成分数。可以输出：

```text
问题存在：高可信（日志和复现一致）
范围为全部平台：低可信（只测Windows）
根因为解析器变化：中可信（版本对照支持，机制日志不完整）
```

同一个问题的不同子主张有不同可信度。

## 八、冲突是结果，不是噪声

处理冲突：

1. 确认是否同一主张；
2. 对齐时间、版本、角色和事件；
3. 追踪共同来源；
4. 检查数据质量和记录权限；
5. 寻找能产生不同预测的条件；
6. 设计成本最低且安全的区分测试；
7. 无法解决时保留冲突和决策限制。

例如评论说失败增加、总体日志稳定，可能因为评论集中在小版本、埋点漏报或评论样本偏差。任何一种都需要验证。

## 九、时间上的交叉验证

发布前后变化可以支持关联，但同时可能有流量、价格、营销、政策和季节变化。使用：

- 同长度对照窗口；
- 相同任务和分段；
- 版本而非日历日期；
- 发布前趋势；
- 未受影响对照任务；
- 回滚或修复后的反向变化。

时间接近不自动证明因果。

## 十、完整案例：v3.2 CSV 导入

### 1. 主张

在桌面端 v3.2，管理员导入含空日期字段的 CSV 时失败率高于 v3.1，问题与日期解析变化相关。

这里包含两个主张：结果差异和根因，需要分别验证。

### 2. 教学场景证据

| ID | 证据 | 观察 | 独立组 |
| --- | --- | --- | --- |
| E1 | 40条评论 | 12条提到v3.2日期错误 | store-public |
| E2 | Issue #81 | 最小CSV可复现，维护者确认 | repo-81 |
| E3 | 行为事件 | v3.2失败6%，v3.1失败1.2% | analytics-v2 |
| E4 | 服务日志 | 错误码集中`invalid_date` | service-log-A |
| E5 | Release Notes | v3.2更新日期解析 | official-release |
| E6 | 本地对照 | 固定CSV在3.2失败、3.1成功 | local-fixture |
| E7 | 网页端日志 | 同输入失败率未变 | analytics-web |
| E8 | 无空日期CSV | 两版本无明显差异 | analytics-control |

E3、E4、E8 若来自同一事件流水线，不完全独立，但回答不同字段。E5 对根因相关，不能证明失败。

### 3. 对齐

评论只有 9 条含平台，其中 8 条为 Windows；行为数据包含 Windows 和 macOS。分段后差异主要在 Windows。网页端使用不同解析路径，因此 E7 限定范围，不推翻桌面端主张。

### 4. 支持与反证

对“v3.2 空日期失败更高”：E2、E3、E4、E6 支持；E8 是有价值对照；E7 限定平台；E1 是线索；E5 中性偏支持机制。

反证：macOS 分段差异小，说明“全部桌面端”过宽。修订主张为 Windows 桌面端。

### 5. 根因推理

固定输入的版本对照和错误码支持解析变化。进一步检查发现 v3.2 将空字符串送入严格解析器，而 v3.1 先转为缺失值。故障注入重复症状；恢复旧规范化后原复现通过。

根因可信度从中提高到高，但只适用于此错误簇。

### 6. 当前结论

事实：Windows v3.2、空日期输入的任务失败率上升，并可固定复现。根因：规范化顺序变化使空值进入严格解析。范围不包括网页端，macOS证据不足。

### 7. 验证与失败分支

1. 修复后重跑固定CSV、非空对照和不同区域日期格式；
2. 用相同事件定义监控发布后两周；
3. 若行为失败下降但评论继续，检查版本采用和旧评论；
4. 若日志错误下降但任务完成不升，寻找下一个失败环节；
5. 若E3埋点版本字段错误，撤回相关量化结论；
6. 若来源权限失效，删除受限派生数据并调整可信度。

## 十一、选择下一项证据

优先选择最能区分竞争解释、成本低、风险小的证据。已有多个评论时，再收集评论收益低；更有价值的可能是固定复现或行为分母。已有总量差异但范围不清时，补版本和环境字段。

停止条件也重要：证据已足够支持可逆小改时，不必追求绝对确定；高风险不可逆决策需要更强验证和守护。

## 十二、个人学习路径

1. 写一个可证伪且限定范围的主张；
2. 找至少三种方法，不只三条链接；
3. 为每项标记原始来源和独立组；
4. 对齐版本、角色、时间与事件；
5. 分类支持、反证、冲突和无关；
6. 保留一个竞争解释；
7. 选择能最大降低不确定性的下一步。

## 十三、完成检查与练习

- 主张能被证伪且范围明确；
- 方法、来源和独立性分开；
- 转载和共同数据未重复计权；
- 证据在版本、环境、时间和口径上对齐；
- 支持、反证、冲突和无关均保存；
- 多数一致没有代替机制验证；
- 子主张分别给可信度；
- 冲突产生了区分测试或明确未知；
- 权限和时效进入可信度判断。

练习：为一个问题建立至少 6 项证据矩阵，覆盖三种方法和一个反证。完成标准是删除重复来源后推理仍成立，并能说明哪条新证据最可能改变决策。
