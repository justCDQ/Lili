# 用户目标、场景、痛点、需求、方案与功能

产品问题分析需要把用户希望得到的结果、结果发生的上下文、当前损失、必须满足的条件、可选实现机制和最终产品能力分开。六个概念混在一起时，团队容易把熟悉的功能请求直接当成需要解决的问题，也无法比较更低成本的替代方案。

## 六个概念的核心区别

| 概念 | 回答的问题 | 合格表达 | 不合格表达 |
| --- | --- | --- | --- |
| 目标 | 用户最终想得到什么结果？ | 在截止前提交准确报表 | 使用报表系统 |
| 场景 | 谁在什么条件和约束下完成目标？ | 财务每周五合并三个地区模板 | 办公场景 |
| 痛点 | 当前做法造成什么可观察损失？ | 每次耗时 40 分钟，近 8 次有 3 次漏列 | 操作很麻烦 |
| 需求 | 为达到目标必须满足什么条件？ | 能识别模板差异并指出缺失字段 | 需要一个合并按钮 |
| 方案 | 用什么机制满足需求？ | 模板校验、自动映射或人工代办 | 做得更智能 |
| 功能 | 产品实现方案中的什么具体能力？ | 上传三个 CSV 后生成差异报告 | AI 功能 |

推荐链路是：

```mermaid
flowchart LR
    A["场景与触发"] --> B["用户目标"]
    B --> C["当前做法"]
    C --> D["痛点证据"]
    D --> E["方案中立需求"]
    E --> F["候选方案"]
    F --> G["产品功能"]
    G --> H["用户结果验证"]
```

链路不是线性文档流程。验证功能后若目标结果没有改善，需要回到痛点、需求或场景重新检查，而不是继续增加功能。

## 用户目标

目标描述用户在产品之外也成立的结果。合格目标包含动作对象和完成状态，例如“确认本月所有异常账单已处理”，而不是“查看账单页面”。

目标有不同层级：

- 终局目标：为什么要做整项工作，例如按时完成月度关账；
- 任务目标：当前要完成的可观察结果，例如核对所有付款记录；
- 操作目标：完成局部动作，例如筛选未匹配交易。

分析时要明确层级。只写终局目标会过于宽泛，只写点击按钮又会把当前界面当成问题边界。通常用任务目标定义产品改进，再说明它怎样支持终局目标。

目标不应包含产品名、控件或预设技术。删除这些词后目标仍成立，说明表达更接近用户结果。

## 场景

同一个目标在不同场景中可能需要不同方案。场景至少包含：

| 要素 | 要回答的问题 |
| --- | --- |
| 角色 | 谁执行，谁受到结果影响？ |
| 触发 | 什么事件使任务现在发生？ |
| 时间 | 截止、频率、持续时长是什么？ |
| 环境 | 使用什么设备、地点、渠道和网络？ |
| 前置状态 | 已具备什么数据、权限和知识？ |
| 当前做法 | 现在按什么步骤完成？ |
| 约束 | 预算、权限、政策、技术和协作限制是什么？ |
| 完成标准 | 用户怎样知道任务结束且正确？ |

“移动端场景”不够具体。可验证表达是：“仓库人员在收货区单手操作手机，网络间歇可用，需要在司机离开前核对 20 件商品并保存异常照片。”这个场景能直接约束离线、输入方式、时限与错误恢复。

## 痛点

痛点是当前流程中可观察的损失，不只是负面情绪。常见损失维度包括：

- 时间：等待、重复输入、切换和返工；
- 质量：错误、遗漏、不一致和不可验证；
- 金钱：直接支出、罚款、库存或收入损失；
- 风险：越权、隐私、安全、合规和不可恢复操作；
- 能力：当前方法根本无法完成任务；
- 认知：必须记忆规则、比较大量选项或理解不必要术语；
- 协作：等待他人、责任不清、版本冲突和交接丢失。

痛点记录应包含发生条件、频率、后果、受影响角色和证据。例如“近四周 32 次导入中 9 次因列名变化失败，每次平均返工 18 分钟”比“导入体验差”更能用于优先级和验证。

情绪可以作为线索，但需要连接行为或后果。“用户很生气”可能来自错误金额、等待时间、无法恢复或预期不一致；不同根因需要不同方案。

## 需求

需求描述为了达到目标必须满足的条件，保持方案中立。常用结构是：

> 在某场景中，某角色需要能够完成某结果，以便达到某目标；即使出现某边界，也必须满足某条件。

例如：

> 在每周汇总不同地区报表时，财务人员需要识别并处理字段差异，以便在截止前提交可核对的总表；任一来源缺少必填列时不得静默合并。

这里没有规定使用 AI、按钮、自动化或人工服务。它允许比较规则校验、标准模板、自动映射、流程约束和人工代办。

需求至少有四类：

- 结果需求：最终必须得到什么；
- 信息需求：做决定需要看到什么；
- 控制需求：用户需要撤销、确认、权限或选择什么；
- 质量与约束需求：准确、时效、可访问性、安全和合规达到什么条件。

需求不是用户说出的原句。功能请求、评论和支持工单是证据输入，需要结合当前流程和后果还原需求。

## 方案

方案是一组满足需求的机制，可以是产品功能、流程变化、内容、人工服务、规则或组合。至少从不同机制生成候选，而不是只画同一页面的三个版本。

针对“可靠合并不同模板数据”，候选包括：

1. 统一模板并在提交端拒绝不合格文件；
2. 上传后自动检测列并要求人工确认映射；
3. 提供差异报告，由财务修正来源；
4. 由运营人员代为清洗并返回结果；
5. 不合并文件，改用共享数据入口直接采集。

方案比较要检查价值、覆盖、成本、风险、依赖和可逆性。最自动化的方案不一定最好；低频高风险任务可能适合人工确认，稳定规则任务可能不需要模型。

## 功能

功能是产品实现方案的具体能力。合格定义包含使用者、触发、输入、行为、输出、权限、失败和价值连接：

| 字段 | 示例 |
| --- | --- |
| 使用者 | 有报表编辑权限的财务人员 |
| 触发 | 上传第二个及后续来源文件 |
| 输入 | CSV、模板版本和当前字段映射 |
| 行为 | 检测缺列、未知列和类型冲突 |
| 输出 | 差异报告与待确认映射 |
| 权限 | 只能读取当前组织被授权文件 |
| 失败 | 编码错误、重复 ID、文件过大、映射冲突 |
| 价值 | 减少静默遗漏并缩短合并时间 |

“增加 AI”“做一个 Dashboard”或“支持导出”都不是完整功能定义。必须说明对什么对象执行什么行为并产生什么可观察结果。

## 六层之间的多对多关系

一个痛点可能对应多个需求。例如导入失败既需要解释错误，也需要保留已完成进度和支持重试。一个需求也可能由多个功能共同满足，例如可靠退款需要资格查询、金额计算、确认、幂等写入和通知。

同一功能还可能服务多个目标。搜索既可帮助找到文档，也可支持客服定位订单。分析时保存映射，避免因为功能使用量高就误认为所有目标都得到改善。

## 不依赖用户访谈的证据路径

### 任务复现与行为观察

亲自按公开文档完成任务，录制步骤、等待、错误、切换和恢复。对没有权限的系统，只分析公开演示、帮助文档和自己控制的账号，不绕过访问限制。

### 产品分析与日志

在有授权的数据中查看漏斗、错误码、耗时、重试、撤销、搜索后无点击和支持入口。行为说明发生了什么，不自动说明动机；需要和场景、内容或其他证据交叉验证。

### 客服、评论和公开社区

FAQ、已知问题、状态页、应用商店评论、GitHub Issue 和论坛可以发现重复失败、变通办法和功能请求。记录版本、日期、任务、后果和反例，不把声量直接当频率。

### 竞品与非软件替代

拆解竞品入口、流程、限制、失败恢复、定价和文档，再观察用户当前是否用表格、邮件、人工服务或不处理。替代方案揭示真正竞争的是哪种当前做法。

### 公开行业与政策材料

法规、标准、行业报告和公开数据可确认硬约束和问题规模。宏观趋势不能直接证明某个具体用户场景，需要与行为证据连接。

## 证据卡

```json
{
  "evidence_id": "csv-import-014",
  "observed_fact": "32 次导入中有 9 次因列名不一致失败",
  "source": "authorized_product_log",
  "scenario": "weekly-regional-report",
  "affected_role": "finance-analyst",
  "consequence": "median_rework_minutes=18",
  "supports": ["pain:field-mismatch", "need:detect-before-merge"],
  "does_not_prove": ["preferred_solution", "all_users_have_same_problem"],
  "confidence": "high"
}
```

证据卡强制区分观察事实、解释和方案。每条需求应能追溯到一条或多条证据；每个功能也应追溯到需求，而不是只追溯到某人的请求。

## 完整案例：每周合并地区报表

### 输入与证据

目标任务是财务人员每周五把三个地区的 CSV 合并为一份总表。现有材料：

- 最近 8 周任务记录显示每次平均 41 分钟；
- 24 个来源文件中 7 个出现列名或类型差异；
- 8 次汇总中 3 次在复核时发现漏列；
- 帮助中心要求用户手工统一模板，但没有列出差异；
- 公开评论中多次出现“导入成功后才发现字段为空”；
- 竞品 A 在上传前验证模板，竞品 B 允许手工映射；
- 当前替代做法是 Excel 复制粘贴和人工抽查。

### 步骤一：写目标和场景

目标是“在周五截止前得到完整、可核对的地区汇总”，不是“使用自动导入”。场景包含三位地区提交者、一个财务汇总者、不同模板版本、固定截止和敏感财务数据权限。

### 步骤二：重建当前流程

下载三个文件 → 打开并检查列名 → 手工复制到总表 → 调整日期与金额格式 → 查找缺失行 → 发送来源方确认 → 提交。等待来源方和检查差异占总时间大部分。

### 步骤三：定义痛点

核心痛点不是“没有合并按钮”，而是字段差异在合并后才暴露，导致漏列和返工。证据是 7/24 文件出现差异、3/8 汇总发现漏列、平均 41 分钟。风险是错误报表进入后续决策。

### 步骤四：写方案中立需求

财务人员需要在合并前识别字段缺失、未知列和类型冲突，并能确认处理方式；系统不能在必填字段缺失时静默产生完整状态。输出需要保留来源和处理记录以便复核。

### 步骤五：比较候选方案

| 方案 | 覆盖 | 成本 | 主要风险 |
| --- | --- | --- | --- |
| 强制所有地区使用单一模板 | 高 | 流程与迁移成本中 | 旧流程继续提交 |
| 上传前规则校验并返回差异 | 高 | 开发成本低至中 | 规则需维护 |
| 自动推断字段映射并人工确认 | 高 | 开发与评测成本中 | 错误建议被直接接受 |
| 财务运营人工代办 | 中 | 持续成本高 | 不可规模化、隐私 |
| 建统一数据录入入口 | 最高 | 迁移成本高 | 项目周期长 |

选择“规则校验 + 明确映射确认”作为第一阶段，因为它直接处理已知字段差异，可确定验证，且不要求立即迁移所有来源。

### 输出

产品功能定义为：有权限的财务人员上传 CSV 后，系统使用版本化规则检查编码、必填列、类型和重复 ID；发现差异时生成逐文件报告，用户确认合法映射后才合并；无法确认时保持未完成状态。

同时输出非目标：不自动修正原始文件、不猜测缺失金额、不替代来源方确认、不跨组织读取文件。

### 验证

用 24 个历史去标识 Fixture 离线回放，要求所有 7 个已知差异被检出，正常文件不被错误阻断超过 1 个；再用 5 个新边界样例测试编码、重复 ID、空文件、超大文件和同名不同类型。任务指标为中位完成时间、漏列率和需要人工返工次数，不能只看校验按钮点击。

若候选结果为差异检出 `7/7`、正常通过 `16/17`、漏列从 `3/8` 降到 `0/8`、中位时间从 41 分钟降到 22 分钟，则达到第一阶段门槛。仍需观察误报和来源方等待是否成为新瓶颈。

### 失败分支

- 日志缺少字段差异原因：先补错误分类，不宣称根因已确定；
- 评论与日志冲突：按版本、套餐和场景切片，保留两个结论；
- 规则无法处理模糊映射：要求人工确认，不自动选择；
- 合并更快但漏列未下降：功能没有实现主要用户价值，停止扩大；
- 来源模板已能统一：重新比较流程约束，可能不需要复杂功能；
- 敏感数据无法用于测试：使用结构等价的合成 Fixture，并降低对真实分布的置信度。

## 验证六层链路

逐层提问：

1. 删除产品名后，目标是否仍成立？
2. 场景是否足以解释约束与差异？
3. 痛点是否有频率、后果或风险证据？
4. 需求是否不包含界面和技术实现？
5. 是否至少存在三个机制不同的方案？
6. 功能是否说明输入、行为、输出、权限和失败？
7. 指标是否验证用户结果而不只是功能使用？
8. 失败后能否回到被证据推翻的层级？

## 常见错误

- 把“希望有某功能”直接写成需求；
- 把目标写成使用产品；
- 用“麻烦、低效”代替损失证据；
- 场景只有人口属性，没有触发与约束；
- 三个候选只是同一界面的不同布局；
- 竞品有某功能就推断自己也需要；
- 上线后只看采用率，不看任务结果和负面影响；
- 证据不支持时仍把方案当确定结论。

## 练习

### 练习一：拆解功能请求

选择一条公开功能请求，建立六层链路并附至少三种来源：公开 Issue 或评论、帮助文档、任务复现或竞品流程。

验收：目标不含产品名；痛点有可观察后果；需求方案中立；至少三个机制不同的方案；功能包含失败状态；每项推断标明证据限制。

### 练习二：验证非软件方案

选择一个低频任务，同时设计流程调整、内容指导、人工服务和软件功能四种方案。

验收：使用同一需求和指标比较；计算一次性与持续成本；明确什么证据会使软件方案失去优势。
