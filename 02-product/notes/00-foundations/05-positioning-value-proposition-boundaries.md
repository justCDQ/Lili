# 产品定位、价值主张、核心能力、辅助能力与产品边界

产品定位是一组选择：为哪些人在什么场景解决什么问题，通过什么机制产生什么结果，以及相对当前替代方案为什么值得选择。价值主张把这组选择转成用户能够理解和验证的结果。核心能力直接构成主要价值链，辅助能力降低获得价值的成本或风险，产品边界则明确不服务谁、不处理什么场景和不承担什么责任。

## 五个概念

| 概念 | 回答的问题 | 主要产物 |
| --- | --- | --- |
| 定位 | 产品选择在哪个问题空间竞争？ | 定位卡与取舍原则 |
| 价值主张 | 目标用户能得到什么结果，凭什么相信？ | 结果、证据和适用条件 |
| 核心能力 | 哪些能力直接完成主要价值链？ | 核心任务与能力映射 |
| 辅助能力 | 哪些能力降低使用成本、风险和阻力？ | 支持、治理与基础能力 |
| 产品边界 | 哪些对象、场景、能力和责任不在范围？ | 非目标与转介方案 |

定位不是宣传语，也不是现有功能的概括。它约束路线图、内容、定价、渠道和运营。如果任何相关请求都能进入范围，定位没有形成取舍。

## 定位的六个字段

### 目标用户

目标用户由任务、场景、约束和行为差异定义，不只由年龄、行业或公司规模定义。“中小企业”仍过宽；“每周需要汇总多渠道订单、没有专职数据团队的零售运营人员”更能指导产品选择。

目标用户还要区分使用者、购买者、管理员和被影响者。B2B 产品的购买定位与使用定位可能不同，但二者不能互相冲突。

### 场景

场景说明触发、频率、环境、前置条件和时间压力。同一个人日常记录与季度审计的需求不同。定位必须说明主要发生场景，而不是列举产品能覆盖的所有情形。

### 问题

问题是当前完成目标时反复出现且有后果的阻碍。它应有行为、数据、公开材料、评论、客服或替代做法证据。功能请求可以提供线索，但不能直接充当问题定义。

### 核心机制

核心机制说明产品主要如何产生结果，例如“把多个账户交易按规则归类并保留可追溯来源”，而不是“使用 AI”。机制要足够具体，可以与替代方案比较，但不必锁死到单个界面。

### 可验证结果

结果必须相对基线，例如“将每周对账中位时间从 90 分钟降到 45 分钟，同时不提高漏记率”。“更简单、领先、智能、一站式”无法独立验证。

### 替代方案与选择理由

替代方案包括竞品、表格、邮件、人工服务、内部工具、维持现状和不做任务。选择理由要说明在目标场景中的差异，例如更低迁移成本、更强本地控制或更完整失败恢复。

完整定位假设可写为：

> 为 `[目标用户]`，在 `[场景]`，解决 `[问题]`；产品通过 `[核心机制]` 提供 `[可验证结果]`。相对 `[当前替代方案]`，选择理由是 `[有证据的差异]`。不服务 `[非目标用户/场景/责任]`。

模板只组织假设，不证明定位正确。每个字段都要附证据与反证。

## 价值主张

价值主张包含三部分：结果、成本和可信理由。

- 结果：用户具体获得什么改善；
- 成本：需要支付、学习、迁移、配置、核验和承担什么风险；
- 可信理由：产品行为、公开文档、测量结果或可试用体验如何证明主张。

净价值不是功能数量。一个功能节省 10 分钟但增加 15 分钟核验，价值主张不成立。免费产品也有数据、注意力、学习和切换成本。

### 主张层级

| 层级 | 示例 | 需要的证据 |
| --- | --- | --- |
| 能力主张 | 支持离线记录 | 文档与实际行为 |
| 结果主张 | 无网时仍能完成记录 | 场景任务验证 |
| 比较主张 | 比竞品少两个步骤 | 同条件竞品拆解 |
| 量化主张 | 中位录入时间降低 40% | 固定基线与样本数据 |

主张越强，证据要求越高。产品首页能证明组织公开说了什么，不能证明所有用户都得到结果。

## 核心能力

核心能力直接完成主要价值链。使用删除测试：如果删除该能力，目标用户在主要场景中还能否获得定位承诺的结果？若不能，通常是核心能力。

例如个人离线记账产品的核心链可能是：快速记录交易 → 本地保存 → 分类与查询 → 核对余额。若删除记录或本地保存，价值链断裂。

核心能力应满足：

- 直接连接主要任务和结果；
- 具有明确输入、输出和失败状态；
- 能被结果指标验证；
- 在目标场景中优于或不同于替代方案；
- 得到优先可靠性、可访问性和支持投入。

“AI”“平台”“数据”不是能力定义。要说明它对什么对象执行什么行为。

## 辅助能力

辅助能力不直接产生主要结果，但降低获取价值的成本、风险或阻力：

- 账号、登录、通知和偏好设置；
- 导入、迁移、帮助和模板；
- 权限、审计、账单和管理员控制；
- 可访问性、国际化和多设备同步；
- 监控、恢复、支持与数据导出。

辅助不等于不重要。企业 SSO 可能不产生协作价值，却是企业采用的硬门槛；无障碍支持可能决定一部分用户能否使用核心能力。应分别记录“价值核心度”和“采用/风险必要性”。

## 产品边界

边界至少包含：

| 边界 | 示例 |
| --- | --- |
| 非目标用户 | 不服务需要多人审批的企业财务团队 |
| 非目标场景 | 不处理高频证券交易账目 |
| 非目标能力 | 不提供发票报销审批 |
| 数据边界 | 不读取未授权银行账户 |
| 责任边界 | 分类建议不替代税务判断 |
| 渠道边界 | 首版只支持移动端离线记录 |
| 服务水平 | 不承诺实时跨设备同步 |
| 地域与政策 | 只支持明确列出的币种与地区规则 |

边界不是一句“暂不支持”。它应说明原因、替代路径、何时重新评估以及越界请求如何处理。高风险任务需要清楚转介人工或专业服务。

## 定位证据

个人可以用以下路径验证，不把用户访谈设为必经步骤：

### 官网、帮助与定价

首页和发布材料显示公开主张；帮助文档显示实际能力与限制；定价页显示目标购买单位、套餐边界和货币化选择。三者不一致本身就是定位风险。

### 产品行为观察

使用自己有权控制的账号完成注册、核心任务、失败、导出和删除，记录入口、步骤、默认值、权限与恢复。实际行为比宣传词更能证明能力边界。

### 评论、客服与社区

公开评论、Issue、FAQ 和已知问题显示用户用什么语言描述目标、哪些场景失败、采用阻力是什么。保留日期、版本和反例，不把单条意见当总体需求。

### 行为数据

若有授权数据，观察目标用户的激活、核心任务完成、重复使用、失败和退出。使用高不等于价值高，需要连接最终结果。

### 竞品和非软件替代

在同一任务与场景下比较步骤、结果、价格、限制和迁移成本。不能把竞品功能清单当定位；真正替代品可能是 Excel、邮件或人工服务。

## 定位证据卡

```json
{
  "field": "choice_reason",
  "claim": "无网络时可以完成单笔记录并在恢复网络后同步",
  "target_scenario": "mobile-daily-expense",
  "evidence": [
    "official-offline-help@2026-07-17",
    "controlled-airplane-mode-test-07"
  ],
  "counterevidence": ["附件离线同步失败"],
  "confidence": "medium",
  "boundary": "只保证文本交易，不保证附件"
}
```

定位卡要同时保存支持与反证。没有证据的字段保持假设，不用流畅措辞掩盖未知。

## 完整案例：个人离线记账工具

### 输入与证据

分析目标是一款个人记账工具。材料包括：

1. 任务观察显示通勤中记录一笔现金支出平均需要 35 秒；
2. 20 次弱网测试中现有网页工具有 6 次保存失败；
3. 应用商店近三个月 42 条相关评论中，12 条提到录入步骤多，7 条提到网络失败；
4. 两个竞品分别采用离线优先和云端自动同步；
5. 定价页显示家庭协作和企业报销属于高级或其他产品范围；
6. 公开帮助说明离线时附件无法同步；
7. 当前非软件替代是手机备忘录，之后再录入表格。

评论只能说明存在问题线索，不能把 `12/42` 当所有用户发生率。弱网测试和任务计时提供可复算行为证据。

### 步骤一：写定位假设

目标用户是需要在移动和弱网场景中每日记录个人支出、愿意稍后核对的个人。问题是即时记录步骤长且网络失败导致遗漏。核心机制是本地先写入最小交易记录，再在联网后同步。结果目标是单笔记录中位时间低于 15 秒，离线保存成功率 100%。

替代方案是备忘录后补、表格和云端记账工具。选择理由假设为离线可靠、输入最少且数据可导出。非目标是家庭共同预算、企业报销审批、税务建议和高频投资交易。

### 步骤二：区分能力

| 能力 | 分类 | 依据 |
| --- | --- | --- |
| 金额与类别快速记录 | 核心 | 删除后无法完成主要任务 |
| 本地持久化和同步队列 | 核心 | 弱网价值链依赖 |
| 日常查询和核对 | 核心 | 记录需形成可用结果 |
| CSV 导出 | 辅助且为退出保障 | 降低锁定和迁移风险 |
| 云同步 | 辅助/采用必要 | 核心离线记录仍存在 |
| 登录 | 辅助 | 提供同步身份，不产生记录价值 |
| 企业审批 | 边界外 | 不属于个人即时记录场景 |

### 步骤三：检查价值主张

价值主张为“在无网或弱网时，15 秒内保存一笔可稍后核对的个人支出”。可信理由是本地写入状态、同步队列和可见失败恢复。不能写“永不丢失”，因为设备损坏、未备份和同步冲突仍可能造成损失。

### 输出

输出包含定位卡、核心价值链、能力分类、非目标、主张证据和三项发布门槛：20 次飞行模式测试全部本地保存；50 笔任务中位时间不超过 15 秒；同步冲突不得静默覆盖。

### 验证

用固定 50 笔支出任务比较备忘录、现有网页工具和原型。记录完成时间、漏记、重复、恢复和导出正确率。再进行断网、进程终止、磁盘不足、重复提交、跨设备冲突测试。

若原型中位时间 13 秒、离线保存 `20/20`，但跨设备冲突有 `2/10` 静默覆盖，则核心场景部分通过，数据可靠性门槛失败，不能扩大同步功能。

### 失败分支

- 目标用户大多自动导入银行卡：即时手工记录的重要性下降，重新评估定位；
- 离线快但核对错误增加：价值主张未成立，补充可见状态或缩小字段；
- 附件不支持离线：明确边界，不用“完全离线”主张；
- 企业请求增长：先验证是否形成不同产品与角色链，不直接扩展当前边界；
- 竞品新增同等能力：重新验证选择理由，而不是增加无关功能；
- 评论集中于旧版本：按版本切片，降低当前问题置信度。

## 定位检查清单

- 目标用户是否由任务和场景定义；
- 问题是否有行为和后果证据；
- 核心机制是否具体到可比较；
- 结果是否有基线、单位和门槛；
- 是否包含软件与非软件替代；
- 选择理由是否有直接证据和反证；
- 核心能力是否通过删除测试；
- 辅助能力是否标明采用或风险必要性；
- 非目标是否覆盖用户、场景、能力和责任；
- 公开主张是否与实际行为、帮助和定价一致。

## 常见错误

- “面向所有人”导致没有优先场景；
- 用形容词代替结果；
- 把现有功能清单倒推成定位；
- 把所有重要能力都称为核心；
- 认为辅助能力可以低质量；
- 只列竞品，不比较当前非软件做法；
- 把官网主张当作结果证据；
- 为留住所有请求而取消边界；
- 边界外任务没有明确失败和转介方式。

## 练习

### 练习一：定位三角核对

选择一个产品，对照首页、帮助文档和定价页建立定位卡，再用实际任务行为核对。

验收：六个字段都有来源；至少找到一处主张、能力或商业边界不一致；明确事实、观察与推断。

### 练习二：能力删除测试

列出 10 项能力，分别判断删除后主要价值、采用门槛和风险控制发生什么变化。

验收：核心与辅助不按使用频率分类；至少识别一个辅助但不可缺少的治理能力；每个非目标写出转介或失败行为。
