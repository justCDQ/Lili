# GitHub Issue、Discussion、公开 Roadmap 与 Release Notes

## 是什么

Issue 常记录可复现缺陷或任务；Discussion 适合开放问题与社区讨论；公开 Roadmap 表达计划或状态；Release Notes 记录实际发布变化。计划不是承诺，发布记录也不证明问题已对所有人解决。

## 为什么需要

这些资料把用户报告、维护者判断、计划与实际发布串成时间线，适合验证问题是否被确认以及哪个版本包含变化。不同资料的状态含义不同，联合阅读可避免把请求、计划和完成混为一谈。

## 关键特性或判断规则

- Issue 关闭原因必须查看，不默认等于已修复。
- Discussion 的点赞和回复反映社区互动，不是总体需求比例。
- Roadmap 是可变计划；Release Notes 是发布记录，但仍需核对版本和平台。
- 只有独立复现、维护者确认或正式发布能提升对应主张的可信度。

## 实际怎么使用

搜索同义词与错误文本，记录仓库、编号、日期、版本、环境、复现步骤、维护者回应、标签和状态。将重复 Issue 关联到主条目。再对照提交、正式发布与文档，区分“有人提出”“维护者确认”“已经修复”“版本已发布”。

证据表：`主张｜Issue/讨论｜影响版本｜复现质量｜维护者确认｜发布版本｜当前状态｜访问日期`。

## 常见错误与边界

关闭可能表示修复、重复、无计划或无法复现；点赞数不是用户占比；Roadmap 时间会变；开源仓库用户不代表全部市场。遵守仓库规则，不发布敏感数据。

## 补充知识

GitHub Issue 可以转换为 Discussion，子 Issue 与依赖可表达工作分解。研究时保存编号和永久链接，不复制可能后续修改的状态文本作为唯一证据。
