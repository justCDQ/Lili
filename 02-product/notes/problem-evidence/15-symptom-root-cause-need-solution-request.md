# 症状、根因、需求表达与解决方案请求

## 是什么

症状是可观察异常或损失；根因是导致症状并能解释其发生机制的条件；需求表达说明需要达到的结果或条件；解决方案请求提出特定实现。它们对应不同证据，不能相互替代。

## 为什么需要

直接实现解决方案请求可能只掩盖症状，且增加新复杂度。分层后可以比较修复根因、降低发生概率、改善恢复和改变流程等不同干预方式。

## 关键特性或判断规则

- 症状记录时间、环境、步骤、期望与实际结果。
- 根因必须有机制证据和可证伪测试，未验证时称原因假设。
- 需求表达保持结果导向，解决方案请求原样保留为线索。
- 多因素系统允许多个必要条件，不强求唯一根因。

## 实际怎么使用

1. 用时间、步骤、环境和结果记录症状。
2. 收集日志、流程和规则，列多个原因假设。
3. 用复现、对照或额外数据排除假设；未验证前只称“可能原因”。
4. 将“增加按钮/提醒/AI”改写成结果需求。
5. 比较修根因、缓解症状、改善恢复和不改产品等方案。

## 常见错误与边界

“五个为什么”不能自动证明因果；最近发生的变化未必是根因；用户提出的功能可提供重要线索但不应直接等同需求；复杂系统可能有多个必要条件。安全与生产事故的根因分析应无责并由足够专业人员完成。

## 补充知识

根因分析的目标是设计有效控制，不是寻找个人责任。对于无法彻底消除的故障，还应同时设计检测、隔离、恢复和复盘机制。
