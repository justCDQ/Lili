问题发现中的 JTBD、当前工作流与替代方案 ​

JTBD 用于描述人在特定情境中想取得的进展；当前工作流说明这个结果现在如何通过人员、工具和规则实现；替代方案说明用户实际可以选择的其他路径。三者必须联合：JTBD 约束结果，工作流提供事实，替代方案揭示选择与切换成本。

一、JTBD 不是用户画像或功能需求 ​

Christensen Institute 将 JTBD 描述为理解情境与驱动力如何推动人和组织作出选择的视角。产品研究中可以把它写成：

text
当 [触发与约束] 出现时，
[承担任务的角色] 想要 [完成可验证的进展]，
以便 [取得更高层结果]，
同时避免 [不可接受的风险或损失]。
1
2
3
4

示例：

当月末需要核对多个账户且原始数据持续变化时，财务负责人想形成可追溯的差异清单，以便在关账前修正异常，同时避免覆盖原始记录。

它不写“想要一个 AI 对账按钮”。按钮是方案；差异清单和可追溯关账才是结果。

JTBD 也不等于人口属性。年龄、行业和公司规模只有在改变触发、约束、风险或工作流时才进入分析。

二、拆解任务层级 ​

同一个陈述可能过宽或过窄：

层级示例用途
高层进展完成月度财务关账确定完整结果
核心任务找出并解释账户差异比较替代路径
子任务导入流水、匹配记录、标记异常观察工作流
操作点击上传、选择日期设计交互

若把操作当 JTBD，产品变化后任务会失效；若只写“经营得更好”，又无法验证。合适粒度通常能定义触发、产物、完成与失败，并允许多种方案竞争。

三、从证据构建，而不是凭空编写 ​

GOV.UK 指南建议同时了解谁在使用服务、想做什么、当前如何完成、遇到什么问题，以及需要什么结果；已有分析、搜索日志、支持数据和过去研究都可以作为证据，用户访谈不是唯一入口。

个人学习者可使用：

  • 真实任务日志和产物；
  • 当前流程观察；
  • 帮助文档、FAQ 和已知限制；
  • Issue、公开社区和评论中的触发与后果；
  • 行为数据中的开始、完成、回退和放弃；
  • 非软件方案的实际执行记录。

每个 JTBD 元素链接证据：触发可能来自日历与任务日志，结果来自下游产物，风险来自错误记录，替代来自实际工具。没有证据的部分标为假设。

四、还原当前工作流 ​

工作流从真实触发到结果，不从产品首页到产品成功页。记录：

  1. 触发事件与责任人；
  2. 每一步角色、工具、输入和输出；
  3. 主动处理、等待和交接；
  4. 规则、权限和审批；
  5. 错误、返工和变通；
  6. 最终产物与下游使用；
  7. 放弃或不行动的结果。

正式 SOP 与实际动作分开。使用消息补材料、把数据复制到私人表格或等待某位专家，可能揭示工具边界，也可能带来合规风险。

{
  "job_id": "JOB-RECON-01",
  "trigger": "月末第一个工作日",
  "actor": "财务负责人",
  "desired_progress": "形成已解释且可追溯的差异清单",
  "completion": "所有重大差异有证据、负责人和状态",

  "evidence": ["task-log-18", "artifact-diff-07"],

五、替代方案的完整集合 ​

直接软件替代 ​

相近产品以相近机制完成任务。比较结果、恢复、权限和总成本，不按功能数量排名。

间接工具与组合方案 ​

表格 + 消息 + 脚本 + 人工复核可能比单一产品更真实。组合方案的优势常是可控、可修改和已被组织批准。

人工与专业服务 ​

人工处理成本随任务增长，但在低频、高风险或例外多的场景可能更合适。

推迟、不做与改变目标 ​

用户可能接受差异、降低精度、延后关账或取消任务。若不行动成本低，软件机会也可能很弱。

自建与继续维护 ​

已有脚本或内部流程有维护成本，也有数据控制、定制和沉没学习。不能只比较订阅价格。

六、替代方案的选择力量 ​

选择不只由效率决定。记录：

  • 现状推力：什么后果让人考虑改变；
  • 新方案拉力：预期得到什么；
  • 现状习惯：熟悉度、数据、审批和团队默契；
  • 新方案焦虑：迁移、隐私、可靠性、锁定和失败；
  • 触发事件：价格变化、事故、人员离开或新规则。

“现有表格步骤多”不表示会迁移。如果表格透明、可审计且团队熟悉，其净价值可能仍高于新工具。

七、计算切换的净收益 ​

text
净收益 = 目标结果改善
       - 学习与配置成本
       - 数据迁移与双轨成本
       - 新的运营和供应商风险
       - 退出与锁定成本

不同成本需统一任务和时间窗。一次节省 5 分钟但每月只发生一次，可能不足以覆盖迁移；低频但避免重大数据损失，则不能只看时间。

八、事实、推断与请求 ​

text
事实：本月三次对账都把两个系统导出的CSV复制到同一表格。
事实：其中一次因列顺序变化导致公式引用错误。
推断：当前工作流对输入格式变化敏感。
用户请求：希望增加自动导入。
需求表达：需要在输入格式变化时仍能发现并安全处理差异。

自动导入可能解决复制成本，也可能把错误静默放大。需求应保留结果和风险，使规则校验、映射、人工确认或文档改进都能参与比较。

八点一、区分任务承担者、决策者和受益者 ​

组织任务常由不同角色共同完成：财务专员执行对账，负责人批准差异，审计人员使用证据，采购决定工具。只记录采购请求会漏掉执行成本，只记录操作者又会漏掉风险责任。

为每个角色写：触发、动作、结果、承担的失败后果和对方案的否决条件。角色目标冲突也是事实，例如执行者希望少填字段，审计者需要完整来源。产品机会不是选择某一方偏好，而是找到满足任务结果与必要控制的路径。

任务的稳定部分与变化部分 ​

高层结果可能稳定，具体工作流会随工具和政策变化。证据卡分别版本化：

text
稳定任务：在关账前解释重大差异并保留审计证据
当前工作流 v1：CSV + 表格公式 + 邮件审批
当前工作流 v2：数据仓库 + 规则匹配 + 工单审批
1
2
3

如果触发、责任或完成结果也变化，应新建任务版本，不把新工作流硬套旧 JTBD。方案上线后重新观察实际流程，因为用户可能新增变通，且旧步骤不一定自动消失。

证据不足时的表达 ​

只有公开请求时，可以写“有人请求自动排会”，不能补出其触发和结果。先建立问题假设和证据缺口，使用公开任务文档、真实替代执行或行为结果验证。为了让模板完整而虚构情境，会让后续方案建立在不可追溯的前提上。

九、完整案例：五人跨时区排会 ​

1. 初始请求 ​

观察到的解决请求是“做一个自动找共同空闲时间的功能”。先不接受该方案，重建任务。

2. 证据输入 ​

教学场景包括 12 次真实排会日志、日历事件、邮件线程和公开预约工具文档：

  • 触发：项目进入需要同步决策的阶段；
  • 角色：项目负责人对会议成立负责；
  • 完成：五人收到时区正确、无已知冲突且包含议程的事件；
  • 当前路径：群消息询问 → 表格收集 → 人工比较 → 创建事件；
  • 12 次中 4 次有人迟回复，3 次确认后改期；
  • 主动协调中位 18 分钟，总历时中位 26 小时；
  • 2 次会议最终改为异步文档，不再需要排会。

3. JTBD ​

当跨时区成员需要对一个不能异步解决的决策同步时，项目负责人想在截止日期前得到所有必要参与者确认的可行时间，以便完成决策，同时避免暴露无关日历信息和反复改期。

“不能异步解决”是需要验证的约束，不是默认事实。

4. 当前工作流 ​

text
负责人发起并给候选范围
→ 参与者分别回复偏好
→ 负责人人工合并时区和忙闲
→ 缺回复时提醒
→ 建立事件与议程
→ 冲突后重新收集
→ 五人确认

主要损失位于等待和改期，不只是寻找空闲时间。议程缺失也可能造成会议成立但任务失败。

5. 替代方案 ​

方案结果覆盖优势成本与风险
自动日历匹配找共同时间主动时间低需授权日历，跨组织限制
候选投票收集偏好无需完整日历权限仍需等待与改期
主持人指定快速建立事件步骤少冲突与参与公平性
人工表格全面可控熟悉、透明等待与重复劳动
异步文档避免会议无排期争议可能不收敛
不做无协调成本适合低价值议题决策延期或缺失

6. 推理结论 ​

事实：当前样本的等待远高于主动时间，且部分任务可由异步方案完成。

推断：把所有日历自动匹配并非完整机会；先判断是否需要同步，再为真正会议减少收集和改期，可能更接近目标。

假设：明确议题是否需要同步、设置回复截止与自动失效候选，能减少总历时；需用新任务验证。

7. 验证 ​

  1. 对未来 6 次任务先做同步必要性检查；
  2. 其中符合会议条件的任务采用候选投票；
  3. 分别记录主动时间、总历时、改期和隐私授权；
  4. 检查事件是否含议程并最终完成决策；
  5. 与旧路径和异步方案比较同一完成标准；
  6. 保留成员拒绝授权、未回复和临时取消的失败分支。

8. 失败分支 ​

  • 若参与者不愿共享日历，自动匹配方案不满足风险边界；
  • 若异步文档未能形成决策，不能用“没有会议”当成功；
  • 若指定时段很快但关键成员缺席，结果不合格；
  • 若只有负责人的任务日志，结论限定组织者视角；
  • 若低频任务的迁移成本高于节省，保留现状可能合理。

十、验证三者是否一致 ​

检查：JTBD 的每个结果是否在工作流中有产物；工作流问题是否影响该结果；替代方案是否完成相同边界；不行动是否被考虑；切换成本是否来自证据。

如果“更高效”没有单位、基线和结果，它不是可验证进展。如果方案只能完成子任务，应说明剩余工作由谁承担。

十一、个人学习路径 ​

  1. 选择一个自己每月真实执行的任务；
  2. 保存三次当前工作流与产物；
  3. 从触发、结果和风险写 JTBD；
  4. 去掉产品名和特定界面；
  5. 实际执行一个软件、一个人工和一个不做/延后替代；
  6. 比较完成、时间、控制、恢复和切换；
  7. 将事实、推断和方案请求分开。

不需要先进行用户访谈，但自我样本不能外推其他角色。

十二、完成检查与练习 ​

  • JTBD 有触发、进展、结果和风险边界；
  • 工作流来自真实任务而非理想制度；
  • 从端到端结果而非产品入口开始；
  • 包含软件、组合工具、人工、推迟和不做；
  • 记录现状优势而不只列缺点；
  • 切换成本与净收益可复核；
  • 请求与需求结果分开；
  • 案例有反证和失败分支；
  • 结论限定角色、场景和证据。

练习：完成一个 JTBD—工作流—替代三联分析，实际执行至少三种替代。完成标准是任何指定产品被移除后，任务仍可理解并能比较结果。
