# 目标导向 Roadmap、产品推进与持续迭代

产品推进把问题证据变成可运行变化，再用上线后的真实信号决定扩大、调整、回退或停止。完整闭环包括目标 Roadmap、负责人、依赖、评审、灰度、回滚、通知、观察与复盘。

## 一、Roadmap 的决策对象

Roadmap 不是未来功能合同，也不是迭代任务的集合。它表达在一段时间内要改变哪些用户或业务结果、优先处理哪些机会、为什么选择这些投入，以及哪些条件会改变计划。

Atlassian 将敏捷 Roadmap 描述为随信息演化的计划，并建议用较粗的月份、季度或 Now/Next/Later 表达方向。Scrum Guide 则把 Product Goal 定义为产品的长期目标，Product Backlog 是实现该目标的涌现式有序工作清单。二者都不支持把远期功能和日期写成不可变承诺。

### 1.1 四层结构

```text
战略约束：服务谁、创造什么价值、接受什么风险
结果目标：要改变的用户/业务结果及时间窗
机会：阻碍目标的用户问题或系统条件
选项：可验证的产品、运营或技术方案
```

任务属于交付层，不应直接占据战略 Roadmap。一个功能只有能指向机会和结果，才进入 Roadmap 讨论。

### 1.2 目标的完整口径

目标至少包含：目标总体、当前基线、预期方向、最小有意义变化、成熟窗口、护栏和截止复核点。

```text
在 2026 Q4 结束前，
将首次创建工作区的 5—50 人团队中，
7 日内完成“邀请成员并发布第一个共享项目”的比例
从 32% 提高到至少 42%，
同时退款率不增加超过 0.3 个百分点，
严重权限事件为 0。
```

“上线引导 2.0”不是目标；“提高活跃”没有总体、动作和窗口，无法验收。

### 1.3 Now/Next/Later 的含义

- Now：证据和方案已足以投入，存在负责人、容量和退出条件。
- Next：机会优先级较高，但方案或依赖仍需验证。
- Later：值得持续观察，尚未承诺范围、日期或资源。

远期项目的不确定性更高，因此信息粒度应更粗。将 Later 写成完整 PRD 会制造沉没成本，也让利益相关者误解为承诺。

## 二、从目标到机会和选项

### 2.1 机会不是功能

“增加批量导入”是选项；“管理员逐个创建成员平均耗时 46 分钟，且 18% 因字段错误返工”是机会。前者关闭了解题空间，后者允许比较 CSV 导入、目录同步、模板或实施服务。

机会证据应包含发生对象、频率、结果损失和当前替代。只记录请求次数会重复计算同一组织，也会让大声客户主导排序。

### 2.2 选项组合

对一个机会至少保留三类选项：

1. 产品流程变化，例如减少步骤或改变默认值。
2. 运营或文档变化，例如模板、支持和实施。
3. 技术或平台变化，例如性能、可靠性或数据能力。

选项不一定都交付。先用成本最低的方式区分关键假设，再决定进入正式开发。

### 2.3 Roadmap 条目模板

```yaml
outcome: 7日核心任务完成率从32%提高至42%
population: 首次创建工作区的5至50人团队
opportunity: 邀请与权限配置导致中断
evidence:
  - 任务漏斗
  - 失败日志
  - 支持工单
options:
  - 邀请模板
  - 目录同步向导
  - 实施清单
owner: onboarding-product
dependencies:
  - identity-platform
guardrails:
  - 严重权限事件为0
review_at: 2026-10-15
```

## 三、负责人、依赖与风险

### 3.1 负责人不是执行名单

负责人拥有决策权并对结果闭环负责：维护目标口径、推动依赖决策、在风险越界时停止、组织上线观察和复盘。所有参与者都列为负责人会导致无人能够作最终取舍。

可以分别设置：

- Outcome owner：对目标结果负责。
- Delivery lead：对交付计划与协调负责。
- Technical owner：对系统实现、运行和回滚负责。
- Decision approver：对安全、合规或商业例外有批准权。

职责必须写成动作和决策，不写“支持”“关注”。

### 3.2 依赖契约

跨团队依赖要记录提供物、消费者、接口、截止条件、失败行为和变更通知。

| 字段 | 示例 |
| --- | --- |
| 提供物 | 身份平台返回用户目录同步状态 |
| 版本 | SCIM sync-status v2 |
| 消费者 | 工作区邀请向导 |
| 就绪条件 | 沙箱通过 20 个边界样本 |
| 超时行为 | 显示可恢复等待，不创建半成品成员 |
| 变更通知 | 破坏性变更提前两个迭代评审 |

只有日期没有接口和失败行为的“依赖”仍不可执行。

### 3.3 风险登记

风险写作：原因事件可能导致结果。例：“目录同步返回部分成功且客户端重试，可能重复创建成员并错误占用席位。”

每项风险包含发生可能性、影响、暴露窗口、预防、检测、响应、负责人和触发阈值。风险分数只用于排序，严重权限或不可逆数据损失不能因概率低而无控制。

## 四、需求、设计与上线评审

评审的目标是发现假设差异并作可追溯决策。会议不是必要条件；输入不完整时开会不会自动补全信息。

### 4.1 需求评审

检查问题证据、目标、范围、非目标、成功指标、规则、依赖与最大风险。关键问题是“如果不做会怎样”“什么证据会使我们停止”。

### 4.2 设计评审

覆盖主流程与加载、空、错误、权限、冲突、撤销、恢复、键盘和窄屏状态。设计还要说明哪些规则由服务端强制，不能用隐藏按钮代替授权。

### 4.3 技术与数据评审

检查数据模型、兼容、容量、迁移、观测、隐私、审计和回滚。埋点在上线前评审，事件名、主体、属性、触发点和去重规则都要确定。

### 4.4 上线评审

上线就绪不等于测试通过。还需确认：

- 构建与配置可追溯。
- 灰度单元和目标人群明确。
- 指标分版本可观察。
- 停止、回滚和数据修复已演练。
- 支持、销售和受影响用户收到匹配信息。
- 值守人、决策人和升级路径可联系。

评审结论为通过、有条件通过或阻塞。每项条件有负责人和截止时间。

## 五、灰度与回滚

Google SRE 将 Canary 定义为把变更在部分生产环境中限时部署并评估，以决定是否继续。小流量只有在按版本/分组观测时才有意义；总体错误率会稀释 Canary 的异常。

### 5.1 发布与启用分离

- Deploy：代码或配置进入环境。
- Release：能力可被部分用户使用。
- Launch：对目标用户公开并配套传播。

Feature flag 可以将二进制发布和功能启用分开，但 flag 本身是生产配置，需要权限、审计、默认值、失效日期和故障策略。

### 5.2 灰度单元

按请求随机不适合有持久状态的功能。同一用户、工作区或订单跨新旧逻辑会产生污染。灰度单元应匹配业务一致性边界：用户、工作区、组织、地区或数据分片。

### 5.3 阶段设计

```text
内部账户 → 1%低风险账户 → 5% → 25% → 50% → 100%
```

每阶段写清最短观察时间、最小样本、绝对健康阈值、与控制组差异阈值、人工检查和继续授权人。

早期小样本适合发现崩溃、权限绕过和明显延迟，不足以估计低频业务结果。后续阶段扩大样本，再评估任务完成和留存。

### 5.4 回滚类型

- 代码回滚：恢复上一构建。
- 配置回滚：关闭 flag 或恢复配置。
- 流量回滚：将流量切回稳定版本。
- 数据回滚：恢复数据；常因新写入和外部副作用而不可直接完成。
- 业务补偿：退款、撤销、重放或人工修复。

数据库变更优先使用 expand/contract：先增加兼容字段并双读/双写，迁移旧数据，确认所有消费者切换后再删除旧结构。回滚代码不等于回滚已产生的数据。

### 5.5 停止条件

示例：

```text
立即停止：任何跨租户数据暴露、重复扣款或不可恢复删除
自动回退：5分钟窗口错误率高于控制组2个百分点且至少100次请求
暂停扩大：p95延迟连续15分钟超过800ms
人工评审：核心任务完成下降超过3个百分点
```

统计阈值不适用于零容忍安全事件。低频严重事件按发生即停处理。

## 六、通知策略

通知对象包括直接用户、管理员、支持、销售、实施、值守和依赖团队。内容与时机按责任不同：

- 用户需要变化、时间、操作和恢复方法。
- 管理员需要配置、权限、数据和审计影响。
- 支持需要已知问题、识别方法和升级入口。
- 值守需要看板、阈值、开关和运行手册。

静默灰度适用于不会改变用户预期的内部实现；改变价格、权限、数据用途或不可逆流程时，不能用“灰度”规避必要告知。

## 七、上线后观察

### 7.1 四类信号

1. 系统健康：错误、延迟、容量、依赖和队列。
2. 产品使用：曝光、开始、成功、失败、恢复和重复。
3. 用户结果：任务耗时、质量、完成与持续采用。
4. 负面影响：投诉、退款、权限事件、支持量和人工补救。

指标必须按版本和灰度单元切分，同时保留绝对 SLO。仅与控制组比较可能在共享依赖故障时让两组一起变差。

### 7.2 观察窗口

技术指标可分钟级成熟，账单争议可能需要一个计费周期，留存需要完整 cohort。上线当天不能用尚未成熟的留存判断成功。

观察计划为每项指标写 owner、查询、刷新频率、成熟条件、阈值和动作。

### 7.3 反馈处理

反馈记录场景、版本、任务、影响、频率和是否可复现。把单条反馈视为故障线索，不自动外推到总体；严重安全或数据问题即使只有一条也进入事件流程。

## 八、无责复盘

Google SRE 的无责复盘关注促成事件的系统因素，并要求行动项可跟踪。无责不等于无人负责，也不免除故意违规；它要求在当时信息和约束下还原决策，而不是事后用结果责备个人。

### 8.1 复盘结构

```text
事件摘要与用户影响
检测、缓解、恢复时间线
哪些机制正常工作
哪些条件促成事件
为什么现有保护没有阻断
行动项、负责人、截止日期和验证方式
```

“操作失误”不是根因。继续追问为何单个操作能产生该范围影响、为何默认值危险、为何缺少预览/审批/灰度、为何监控未及时发现。

### 8.2 时间指标

- TTD：异常发生到检测。
- TTA：检测到有人开始处理。
- TTM：开始处理到影响停止扩大。
- TTR：恢复到目标服务状态。

这些指标用于识别系统改进点，不用于个人绩效排名。

### 8.3 行动项质量

“加强意识”“以后小心”不可验证。合格行动项写成系统变化：为生产 flag 增加双人审批；在灰度看板按版本显示错误；删除操作增加服务端不可逆确认；每季度演练回滚。

行动项按降低发生概率、缩小影响、加快检测和加快恢复分类。每项指定验证证据；完成代码不等于风险已降低。

## 九、案例一：工作区邀请改版

### 输入

目标是把 7 日核心协作任务完成率从 32% 提高至 42%。证据显示邀请环节失败占流失的 28%，主要问题是域限制、重复成员和权限模板不清。

身份平台团队负责目录和域验证，计费团队负责席位占用，产品团队负责流程，安全团队审批跨域策略。

### 推进

Roadmap 条目围绕“降低邀请失败并正确配置权限”，候选为错误前置校验、邀请模板和目录同步。首轮选择前置校验与模板，目录同步留在 Next，因为依赖和实施成本尚未确认。

评审覆盖重复邀请、已停用账号、域变更、席位不足、部分成功、管理员权限撤销和重试幂等。

灰度以工作区为单位：内部 20 个、1%小团队、10%、50%、100%。零容忍事件为越权加入或错误占用付费席位；错误率与完成率分别设阈值。

### 输出与验证

10% 阶段邀请成功由 71% 升至 83%，但席位不足错误上升。日志发现新流程比旧流程更早预占席位，重试会保持占用。

团队暂停扩大，不因主要指标改善而继续。计费团队修正为确认成员创建后幂等占用，并补偿 37 个受影响工作区。重新灰度后错误恢复基线，才扩大到 50%。

### 复盘

促成条件包括跨服务事务边界未显式建模、测试只覆盖成功和完全失败、Canary 看板未按错误类型显示。行动项分别增加状态机契约、部分成功测试和席位占用告警。

## 十、案例二：搜索排序模型上线

### 输入

目标是减少知识库查找时间。离线评估显示新模型 NDCG 提高，但线上可能改变点击、延迟和权限过滤。新旧模型共享索引服务。

### 推进与灰度

评审明确权限过滤在检索前由确定性服务执行，模型不能看到无权文档。部署与启用分离，按用户固定分桶。

1% 阶段只观察错误、权限和 p95 延迟；10% 阶段加入零结果率、重写查询率；25% 后观察任务成功和回访。

### 失败分支

新模型 p95 延迟从 420ms 升至 780ms，控制组也升至 650ms。若只比较两组会低估共享索引故障。绝对 SLO 触发暂停，值守发现重建索引占用资源。

团队先恢复索引容量，再重新开始同时间并行 Canary，避免 before/after 的时间混杂。

### 输出

两周成熟窗口后，查找任务成功提高 5.2 个百分点，权限事件为 0，延迟在 SLO 内。团队扩大并保留旧模型快速切换 30 天，随后清理 flag 和运行手册。

## 十一、综合练习

为一个中等复杂功能完成从 Roadmap 到复盘的交付包。

### 验收标准

- [ ] Roadmap 条目有结果、总体、基线、窗口、机会和护栏。
- [ ] Now/Next/Later 的证据与承诺粒度不同。
- [ ] Outcome、交付、技术与审批责任清楚。
- [ ] 每个依赖有接口、就绪、失败和变更约定。
- [ ] 评审覆盖权限、数据、失败、恢复、埋点与上线运行。
- [ ] 灰度单元匹配业务一致性边界。
- [ ] 每阶段有观察时间、阈值、决策人和停止动作。
- [ ] 代码、配置、流量、数据和业务补偿分别设计。
- [ ] 上线观察含健康、使用、结果和负面影响。
- [ ] 复盘行动项可验证并跟踪到关闭。
