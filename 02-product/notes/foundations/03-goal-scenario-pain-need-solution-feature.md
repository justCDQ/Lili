# 用户目标、场景、痛点、需求、方案与功能

## 是什么

- **目标**：用户希望达到的结果。
- **场景**：目标发生时的人、时间、地点、触发条件、约束和上下文。
- **痛点**：当前完成任务时可观察的时间、质量、成本或风险损失。
- **需求**：为达成结果必须满足的条件，不预设实现形式。
- **方案**：满足需求的一组机制。
- **功能**：产品实现方案的具体能力。

链路应为：`场景触发 → 用户目标 → 当前做法 → 问题证据 → 需求 → 候选方案 → 功能`。

## 为什么需要

用户常用熟悉的功能表达需求，例如“加一个导出按钮”。直接照做会跳过真实目标、替代方案和限制条件。问题稳定，方案可变；保持分层才能比较多个方案。

## 关键特性或判断规则

- 目标描述结果，不能包含产品名或界面控件。
- 场景必须包含触发、角色、当前做法、约束和完成标准。
- 痛点必须能观察到后果；只有主观形容词不构成充分证据。
- 需求保持方案中立；方案和功能必须能追溯到同一需求。
- 一个功能可能服务多个需求，一个需求也可能由多个功能共同满足。

## 实际怎么使用

1. 用一句动词加结果写目标。
2. 补全触发、频率、环境、约束与完成标准。
3. 记录当前工作流及可量化损失。
4. 将功能请求改写为“需要能够……以便……”。
5. 至少列三个候选方案，再按价值、成本和风险筛选。

案例：事实是用户每周手工合并三份表耗时 40 分钟；目标是按时提交准确报表；需求是可靠地合并同结构数据；自动导入、模板校验和人工代办是不同方案，“合并按钮”只是功能。

## 常见错误与边界

- 情绪不等于痛点证据；应记录发生条件与后果。
- 需求不能写成界面控件。
- 目标不能写成“使用我们的产品”。
- 同一需求可能无须软件解决。

## 补充知识

GOV.UK 将用户需要写成“需要做什么，以便得到什么结果”，并要求基于证据持续校正。功能请求可以作为问题线索保存，但进入优先级之前应先还原需求和现有替代方案。
