# 产品、项目、功能、服务与平台

## 是什么

- **产品**：持续为一类用户解决问题、创造价值，并通过反馈反复演进的价值载体。
- **项目**：为得到唯一结果而开展的临时性工作，有明确开始、结束、范围和交付物。
- **功能**：产品中支持某项用户任务的能力。功能必须从属于问题和产品目标。
- **服务**：用户为取得结果而经历的端到端过程，可能跨软件、人员、规则和线下渠道。
- **平台**：向多类参与者提供共享能力、规则和连接机制的产品；只有出现复用或多边互动，才应称平台。

## 为什么需要

五者的管理对象不同：产品看长期结果，项目看按约束交付，功能看任务能力，服务看全程体验，平台看生态规则。把项目完成当成产品成功，会遗漏采用、留存和实际效果；把单一后台称为平台，会掩盖其复用性不足。

## 关键特性或判断规则

1. 写下对象是否有明确终止日期；有则更接近项目。
2. 写下用户最终要完成的事；覆盖完整结果的是服务，只完成其中一步的是功能。
3. 检查是否有多个参与方、共享基础能力以及治理规则；三者同时存在才按平台分析。
4. 用一张卡记录：目标用户、持续问题、核心能力、生命周期指标、当前项目、功能清单、服务触点。

## 实际怎么使用

选择一个正在使用的软件，分别写出其持续产品目标、当前有终点的项目、支撑用户任务的功能、端到端服务触点，以及是否具备平台参与方和治理机制。每项都附一个公开证据。无法从公开材料确认的平台机制标为推断。

案例：一次“上线订阅支付”是项目；订阅管理是产品能力；修改付款方式是功能；从选购到退款是服务；连接商家、消费者和开发者并提供支付规则与 API 才构成平台。

## 常见错误与边界

- 功能数量不是产品价值；应验证任务完成率和结果。
- 项目按期结束不等于产品被采用。
- “平台”不是技术架构的同义词；内部公共库可以是基础设施，但未必是产品平台。
- 同一对象可同时具有多重身份，应明确当前分析层级。

## 补充知识

PMI 将项目定义为创造独特产品、服务或结果的临时性工作。GOV.UK 服务标准要求理解用户完整情境，并跨渠道解决完整问题。这两条可用来分别校验“临时交付”和“持续服务”的边界。

软件可以同时是产品、服务和平台，分类取决于当前决策对象。拆解前写清分析层级，避免把服务体验问题误归为单个界面功能。
