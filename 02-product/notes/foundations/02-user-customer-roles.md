# 用户、客户、使用者、购买者、决策者与管理员

产品中的“人”不能只用一个用户标签表示。一个人可能直接操作产品、批准采购、支付费用或配置权限；同一个任务也可能由多人共同完成。角色分析的目标是明确谁参与价值获得、采用、付费、治理和风险承担，并把每个角色的任务与证据分开记录。

## 六个基础角色

| 角色 | 定义 | 主要行为 | 不能默认推出 |
| --- | --- | --- | --- |
| 用户 | 从产品或服务获得价值，或直接受到结果影响的人 | 完成目标、接受结果、承担影响 | 一定亲自操作界面 |
| 使用者 | 直接操作产品、API、设备或流程的人 | 输入、配置局部选项、读取输出 | 一定付款或决定采购 |
| 客户 | 与提供方形成采购、合同或服务关系的个人或组织 | 购买、接受服务、承担合同责任 | 组织内每个人目标一致 |
| 购买者 | 执行付款、采购、合同或订阅管理的人 | 询价、审批单据、付款、续费操作 | 拥有最终采用决定权 |
| 决策者 | 有权决定采用、扩大、续约、限制或淘汰的人 | 比较方案、承担决策后果、批准选择 | 直接使用产品或处理付款 |
| 管理员 | 配置账号、权限、策略、集成、数据和运行规则的人 | 开通、治理、审计、停用与恢复 | 是组织最高决策者 |

这些角色是行为责任，不是固定职位。财务负责人可能是购买者，采购专员也可能只是执行已批准的合同；部门负责人可能决定续约，却从不登录产品。个人产品中，同一个人常同时是用户、使用者、购买者和决策者。

## 用户与使用者的区别

使用者强调直接交互，用户强调价值和影响范围。例如家长操作学校缴费系统，学生可能是被服务的用户；客服操作后台为消费者退款，客服是后台使用者，消费者是退款服务用户；自动化程序调用 API 时，程序是技术调用方，最终价值仍由人或组织获得。

区分二者可以发现三类遗漏：

1. 被结果影响但没有界面入口的人；
2. 代表他人操作、需要代理权限的人；
3. 为完成服务提供帮助的内部或第三方使用者。

不能因为某类人不登录，就把其准确性、可申诉性、可访问性和风险排除在产品要求之外。

## 客户、购买者与决策者的区别

客户是关系主体，购买者是采购动作角色，决策者是选择权角色。在组织采购中，公司可以是客户，采购人员执行合同，部门负责人批准预算，IT 和安全团队拥有否决或限制条件，员工完成日常使用。

一次购买链通常包含：

```mermaid
flowchart LR
    A["发现问题或发起试用"] --> B["评估任务价值"]
    B --> C["安全、技术和合规审查"]
    C --> D["预算与方案决定"]
    D --> E["采购与付款"]
    E --> F["部署和配置"]
    F --> G["日常使用"]
    G --> H["续约、扩大或淘汰"]
```

每一步可能由不同角色承担。购买完成只证明交易发生，不证明部署成功、使用者采用或用户获得结果。试用发起者也不一定有权把产品引入正式环境。

## 管理员不是一个统一角色

成熟组织常按最小权限拆分管理责任：

- 组织管理员：管理组织级设置与成员；
- 用户管理员：创建、停用账号和重置访问；
- 账单管理员：管理付款方式、发票和订阅；
- 安全管理员：查看安全告警并配置安全策略；
- 内容或社区管理员：治理内容、互动和违规；
- 集成管理员：配置应用、API、域名或身份系统；
- 审计查看者：只读访问日志和报告。

GitHub 与 Microsoft 365 的公开权限文档都展示了角色与权限集合的分离。产品分析时不能把“管理员”写成拥有所有能力的人，否则会漏掉权限不足、职责冲突和高权限风险。

管理员的核心任务通常包括加入、变更和离开三个阶段：

| 阶段 | 任务 | 需要验证的结果 |
| --- | --- | --- |
| 加入 | 建组织、验证域名、配置登录、分配许可 | 正确人员获得最小必要访问 |
| 变更 | 调整角色、策略、集成和账单 | 变更可审计且不破坏现有工作 |
| 离开 | 停用账号、转移资产、撤销 Token | 数据和权限没有遗留风险 |

## 还需要检查的参与角色

六个基础角色不能覆盖所有产品。按实际流程检查：

- 发起者：首先提出问题或开始试用；
- 影响者：提供建议或改变决策，但没有最终批准权；
- 审批者：批准预算、权限、内容或业务动作；
- 实施者：迁移数据、配置集成、培训和上线；
- 支持者：帮助他人完成任务，例如客服、家属或代理人；
- 审计者：检查行为、记录和控制是否符合要求；
- 被服务对象：获得结果但不直接操作；
- 被影响者：承担自动决策、排序或政策结果；
- 供应方：在平台上提供商品、内容、服务或数据。

只增加角色名称没有价值。每个角色都要连接到具体任务、权限、成功证据和失败后果。

## 角色矩阵

角色矩阵用于把角色定义转成可检查的产品输入：

| 字段 | 要回答的问题 |
| --- | --- |
| 角色 | 该角色由什么行为定义？ |
| 场景与触发 | 在什么条件下进入流程？ |
| 任务 | 需要完成什么结果？ |
| 价值 | 完成后获得什么可观察改善？ |
| 成本 | 付出多少时间、费用、学习或切换成本？ |
| 权限 | 可以读、写、批准、配置和删除什么？ |
| 风险 | 错误、滥用或失败会影响谁？ |
| 采用影响 | 能发起、阻止、扩大或停止采用吗？ |
| 成功证据 | 用什么行为、状态或指标判断成功？ |
| 证据与置信度 | 结论来自哪里，确定程度如何？ |

角色矩阵描述当前分析对象，不是永久组织图。产品版本、组织规模、合同与治理方式变化后需要更新。

## 不依赖用户访谈的证据来源

个人学习可以从公开材料与可观察行为建立初始角色假设：

### 权限与管理文档

帮助中心、权限表、管理员指南和 API 权限范围可以确认系统实际支持哪些角色、每个角色能做什么。权限文档证明的是产品设计与公开规则，不证明真实组织一定按该方式分工。

### 定价、账单与采购材料

定价页、发票说明、企业采购页、许可规则和合同条款能显示谁付费、计费单位、账单权限和采购门槛。它们不能单独证明谁日常使用或谁拥有组织内部最终决定权。

### 产品行为观察

亲自完成注册、邀请、授权、账单、成员移除和资产转移流程，记录每一步需要的身份、权限、确认与失败状态。测试只能使用自己有权控制的账号和数据。

### 数据与日志

若拥有合规访问，可以按角色查看邀请接受率、首次配置成功率、活跃使用、权限错误、支持请求和续约前使用。数据能显示行为差异，但职位或意图需要其他证据解释。

### 公开评论、客服材料和社区问题

应用商店评论、公开 Issue、论坛、FAQ、状态页和已知问题能暴露谁遇到什么阻塞。例如“需要管理员开启”“账单联系人无法改设置”可提示角色分离。单条评论不能代表总体频率，要记录日期、版本和重复证据。

### 竞品拆解

比较同一任务在不同产品中的角色、权限和采购路径，可以发现可替代的治理方式。竞品实现证明一种可行选择，不自动证明适合当前用户和风险。

## 证据记录格式

```json
{
  "claim": "账单管理员可以管理付款，但不能访问项目内容",
  "role": "billing_admin",
  "source_type": "official_permission_document",
  "source_ref": "billing-roles@2026-07-17",
  "observed_behavior": "账单页面可访问，内容页面被拒绝",
  "confidence": "high",
  "counterevidence": [],
  "next_check": "验证企业套餐是否相同"
}
```

事实、观察和推断要分开：文档写明的权限是公开事实；在自己的测试组织中得到的结果是一次观察；“多数企业由财务承担此角色”若没有数据，只能是待验证推断。

## 完整案例：团队协作 SaaS 的角色分析

### 输入与证据

分析对象是一款按席位收费的团队协作 SaaS。可用证据如下：

1. 定价页说明组织按成员席位订阅；
2. 权限文档列出组织所有者、普通成员和账单管理员；
3. SSO 部署指南要求组织管理员配置域名和身份提供方；
4. 帮助中心说明移除成员前需要转移其资源；
5. 公开社区中重复出现“成员无法安装集成，需要管理员批准”；
6. 自建测试组织中，普通成员可编辑项目，但无法访问账单或组织安全设置。

这些材料不要求直接联系用户，也足以建立第一版可验证角色模型。

### 步骤一：按行为识别角色

- 普通成员直接编辑和协作，是使用者，也是获得任务价值的用户；
- 组织所有者决定是否正式采用和扩大席位，是决策者；
- 账单管理员维护付款与发票，是购买流程角色；
- IT 管理员配置 SSO、成员和安全策略，是管理员与实施者；
- 签订订阅关系的公司是客户；
- 项目外部审批人只查看或批准内容，是受限协作者；
- 被移除成员的资源接收者承担离职交接任务。

### 步骤二：建立任务与权限矩阵

| 角色 | 关键任务 | 最小权限 | 成功证据 | 失败后果 |
| --- | --- | --- | --- | --- |
| 普通成员 | 创建、编辑和共享项目 | 被授权项目读写 | 首个项目完成并持续协作 | 无法采用或误共享 |
| 组织所有者 | 决定采用、分配治理责任 | 组织配置与角色委派 | 正确授权且风险可控 | 高权限集中或采购停止 |
| 账单管理员 | 管理付款、发票和订阅 | 账单读写，不读内容 | 付款成功且发票可获取 | 服务中断或数据暴露 |
| IT 管理员 | 配置身份、策略和离职流程 | 身份、安全和成员管理 | 正确人员获得正确访问 | 越权、锁定或资产遗失 |
| 外部审批人 | 查看并批准指定内容 | 指定资源的受限访问 | 审批完成且无额外披露 | 流程阻塞或越权读取 |

### 步骤三：绘制采用链

成员发起试用不代表正式采购。采用链为：成员证明协作价值 → 部门负责人决定继续 → IT 完成安全和身份检查 → 采购或账单角色付款 → 管理员部署 → 成员日常使用 → 决策者结合使用与风险决定续约。

### 输出

输出不是一组 Persona，而是三个可执行产物：

1. 角色矩阵及证据引用；
2. 采用、部署、使用、续约的责任图；
3. 产品要求清单：分离账单与内容权限、支持角色委派、提供成员生命周期和资产转移、记录管理员审计事件。

### 验证

使用测试组织逐项验证普通成员、账单管理员和所有者的页面与动作权限；对照公开文档核对差异；在帮助中心和社区中搜索每个关键失败；检查每项产品要求能否追溯到角色任务或风险。

完成标准：所有关键流程都有负责角色；没有把付款成功当成使用成功；管理员权限符合最小权限；被服务者和外部协作者没有被遗漏；事实与推断分别标记。

### 失败分支

若公开文档没有说明谁决定续约，只能记录“决策者未知”，不能根据职位猜测。若测试行为与文档冲突，应记录产品版本、套餐和账号状态，并把结论降为待确认。若一个人承担所有角色，仍保留角色分栏，因为其任务、权限与指标不同。若评论只出现一次，不以此推断普遍问题，而是作为需要更多证据的线索。

## 常见分析错误

- 用“企业用户”“普通用户”替代任务和权限；
- 直接把职位映射为角色；
- 只分析购买者，不分析日常使用和管理；
- 只分析登录者，漏掉被服务和被影响的人；
- 把管理员设计为默认全权角色；
- 把公开定价主张当作真实采用证据；
- 把评论数量直接当发生率；
- 建立角色后没有连接产品要求与指标。

## 练习

### 练习一：B2C 角色合并

选择一个个人订阅产品，分别填写用户、使用者、购买者和决策者。即使由同一人承担，也要写出各角色的任务、成功证据和退出条件。

验收：能说明角色为何合并；至少识别一个其他角色，如家庭成员、内容供应者或客服；结论附公开材料或实际行为证据。

### 练习二：B2B 角色分离

选择一个有公开权限和定价文档的组织产品，制作至少五角色矩阵并重建采用链。

验收：每个角色包含任务、价值、权限、风险和证据；购买者与决策者不被默认合并；至少验证一个权限失败分支；明确仍未知的组织内部角色。
