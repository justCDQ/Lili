# 目标用户、JTBD、使用频率与替代方案

功能拆解需要回答：谁在什么情境下试图取得什么进展，这项任务自然发生多频繁，当前用什么方式完成，以及软件为什么可能比替代方案更合适。目标用户、JTBD、频率和替代方案共同限定结论，不能单独依靠人口画像或产品使用量。

## 目标用户

目标用户是一组在任务、触发、约束和结果要求上具有重要共同点的人。产品拆解中的分组要能解释行为差异。

“年轻白领”“中小企业”“所有创作者”通常过宽。更可执行的描述是：“每年自行完成一次个人申报、收入来源较简单、需要在截止前核对材料的人”。

目标用户至少包含：

| 字段 | 要回答的问题 |
| --- | --- |
| 角色 | 谁执行，谁购买、管理或受影响？ |
| 触发 | 什么事件使任务现在发生？ |
| 当前做法 | 不使用目标产品时怎样完成？ |
| 约束 | 时间、权限、知识、设备和风险是什么？ |
| 完成结果 | 怎样判断任务正确结束？ |
| 差异 | 为什么这一组需要不同方案？ |

人口属性只有在它改变上述字段时才有分析意义。地区可能改变政策，年龄本身不能直接解释某个界面选择。

## JTBD

JTBD 用情境中的进展组织任务。常用结构：

> 当 `[触发与场景]` 时，我想 `[完成任务]`，以便 `[获得结果或进展]`。

例如：

> 当年度申报窗口开启时，我想确认所需材料完整，以便在截止前提交且不遗漏。

JTBD 不应包含产品名、按钮或指定方案。“当我打开应用时，我想点击 AI 按钮”只是产品操作。

### JTBD 的组成

- 情境：任务为什么现在发生；
- 任务：用户试图完成的动作与对象；
- 进展：相对当前状态希望发生什么变化；
- 完成：什么证据表明任务结束；
- 阻碍：当前替代为何不足；
- 取舍：用户愿意付出什么、不愿承担什么。

JTBD 是分析表达，不是天然事实。每个组成需要证据，缺少证据时保持为推断或假设。

## 事实、推断与假设

| 类型 | 报税案例 | 能否直接作为结论 |
| --- | --- | --- |
| 观察事实 | 官方公布截止日期和材料要求 | 可在限定地区与日期下使用 |
| 观察事实 | 测试者用清单完成固定任务 | 只证明该测试环境的行为 |
| 推断 | 间隔一年会降低操作熟悉度 | 需要行为或错误证据支持 |
| 假设 | 分阶段清单会减少材料遗漏 | 需要对比验证 |

不能把自己写出的 JTBD 当成用户原话，也不能把一次个人经历外推为目标市场分布。

## 自然频率

自然频率是用户任务在产品之外发生的频率，不是打开应用、收到通知或点击功能的次数。

记录四项：

- 周期：每天、每周、每年或事件驱动；
- 峰值：月末、节日、事故或发布前集中；
- 截止：是否存在时间窗口和逾期后果；
- 变异：不同角色、地区和规模是否不同。

### 频率与错误代价

| 频率 | 错误代价低 | 错误代价高 |
| --- | --- | --- |
| 高频 | 操作效率、快捷和批量 | 自动化加校验、撤销和监控 |
| 低频 | 清晰引导、低学习成本 | 资格说明、确认、审计和人工支持 |

高频不自动等于高价值。年度报税、灾难恢复、账户继承和紧急申诉频率低，但错误影响可能高。

低频任务还要检查重新学习成本、规则变化、遗忘和旧数据迁移。追求每日活跃可能与任务本身冲突。

## 替代方案

用户选择的是完成任务的方式，不是功能表。替代至少覆盖：

### 直接软件竞品

使用相近机制解决相近任务。比较时固定用户、任务、地区、套餐和数据。

### 间接软件

日历、表格、邮件、文档和通用自动化可能组合完成同一结果。

### 人工与专业服务

会计、客服、顾问、同事和家属可能提供判断、代理和风险承担。人工成本高，但在低频高风险任务中可能更可靠。

### 纸笔和线下流程

纸质清单、打印材料和线下窗口具有不同控制、访问和隐私特征。

### 延后与不行动

用户可能等待更多信息、接近截止再做，或接受当前损失。原因可能是价值不足、切换成本、风险或任务优先级低。

“没有竞品”通常只是没有识别当前工作流和不行动。

## 替代方案比较字段

| 字段 | 说明 |
| --- | --- |
| 任务覆盖 | 能否得到相同完成结果 |
| 总时间 | 准备、执行、等待和返工 |
| 金钱 | 购买、服务和持续费用 |
| 质量 | 正确、完整和可核对 |
| 控制 | 确认、撤销、导出和隐私 |
| 风险 | 失败后果、责任和恢复 |
| 学习 | 首次和长期不用后的重新学习 |
| 切换 | 数据、流程、审批和历史迁移 |

不能只用功能数量或订阅价格排名。

## 个人可执行证据

- 官方政策、截止和流程确认硬条件；
- 帮助、FAQ、定价和权限文档确认产品公开边界；
- 公开评论、Issue 和客服材料发现失败与变通办法；
- 自己控制的任务复现记录步骤、时间和错误；
- 竞品演示与试用比较同一任务；
- 表格、纸质清单或人工服务作为非软件基线；
- 有授权时使用行为、错误和支持数据验证频率与结果。

宏观材料不能直接证明某个产品用户的行为，评论数量也不能直接当发生率。

## 完整案例：年度报税材料准备

### 输入与证据

限定对象为某一地区、收入来源简单、准备自行申报的个人。使用公开申报窗口、官方材料清单、税务软件帮助页、应用商店评论、会计服务说明和一组虚构申报材料。所有测试不使用真实身份证、税号或财务数据。

观察事实：任务一年一次；官方列出明确材料；逾期有后果；软件提供材料清单。评论中存在“忘记材料”和“规则难懂”的报告，但不能代表总体频率。

推断：低频导致熟悉度下降。假设：按来源组织的渐进清单和状态保存能减少遗漏。

### 步骤一：限定用户

排除复杂企业、多地区收入和需要专业判断的场景。软件不应把简单个人流程的结论外推到高风险对象。

### 步骤二：写 JTBD

“当年度申报窗口开启时，我想确认所需材料完整，以便在截止前提交且不遗漏。”完成标准是材料集合满足官方要求并能指出缺失，不是打开提醒或浏览清单。

### 步骤三：记录频率和风险

自然频率一年一次，峰值在截止前，规则可能按年度变化。遗漏会造成返工、延迟或错误申报，因此设计优先清晰、来源、保存、复核和转介，不追求每日使用。

### 步骤四：比较替代

| 方案 | 优势 | 成本与风险 |
| --- | --- | --- |
| 官方清单 + 日历 | 免费、权威来源直接 | 状态和材料需自己管理 |
| 表格清单 | 可定制、可保存 | 规则更新和版本责任在个人 |
| 税务软件 | 流程集成、可校验 | 费用、数据和适用边界 |
| 会计服务 | 专业判断和责任支持 | 金钱、等待和材料共享 |
| 延后处理 | 当前无需投入 | 截止风险和集中压力 |

### 输出

输出 JTBD 卡、目标用户边界、频率/风险矩阵、替代方案表和功能判断：产品需要帮助恢复上下文、显示材料来源与版本、保存进度、指出缺失并在超出边界时转介。

“提高日活”不适合作为主要目标。任务完成、遗漏、复核时间和边界转介更接近价值。

### 验证

构造 12 个固定任务：正常、缺一项、重复、旧年度、地区不适用和需要专业判断。对官方清单、表格和软件使用同一材料集。

记录材料识别正确数、遗漏、完成时间、错误自信、恢复和转介。若软件 `12` 个任务中 `10` 个正确，2 个复杂任务给出确定答案而非转介，则高风险边界失败，不能以平均完成率宣布成功。

### 失败分支

- 政策因地区或年度不同：限定版本并阻止旧规则静默使用；
- 评论来自复杂申报者：不能支持简单个人定位；
- 提醒打开率高但材料仍缺：过程指标没有形成结果；
- 会计服务成本高但错误显著低：高风险用户可能应选人工；
- 大多数目标用户已有可靠清单：增量价值不足；
- 无真实频率数据：使用官方周期，不推断访问分布；
- 产品无法判断适用边界：明确停止并转介，不生成确定结论。

## 拆解检查表

- 目标用户是否由任务和约束定义；
- JTBD 是否含触发、任务和进展；
- 删除产品名后是否仍成立；
- 每项是否标记事实、推断或假设；
- 频率是否为任务自然频率；
- 是否同时检查错误代价和峰值；
- 替代是否含竞品、组合工具、人工和不行动；
- 所有替代是否使用同一完成标准；
- 个人测试结论是否限定环境；
- 高风险场景是否有转介和停止。

## 结果与采用指标

JTBD 是否成立不能只看功能访问。可使用：

| 指标 | 定义 | 误读风险 |
| --- | --- | --- |
| 目标任务完成率 | 达到预设完成标准的任务 / 已开始任务 | 标准过松会制造成功 |
| 首次恢复时间 | 长期未使用后从进入到正确完成的时长 | 熟练用户不能代表低频用户 |
| 遗漏率 | 缺少必要对象的任务 / 已完成任务 | 参考规则必须版本一致 |
| 边界转介准确率 | 应转介且正确停止的任务 / 应转介任务 | 不能用少答替代正确边界 |
| 替代切换率 | 任务中转向人工或其他工具的比例 | 转向可能是合理选择 |
| 重复任务频率 | 同一用户在自然周期内的任务次数 | 不等于产品打开次数 |

若没有真实产品数据，报告固定任务原始结果，例如 12 个任务中 10 个正确、2 个应转介但错误继续。受控比例用于比较方案和发现风险，不能声称代表市场发生率。

频率分析还应检查任务间隔分布。平均一年一次可能掩盖少数人月度处理和多数人从不处理；产品应按实际切片判断是否需要快捷操作、批量、重新引导或人工支持。

## 练习

分别选择一个高频低风险任务和低频高风险任务，完成两张 JTBD 卡和替代方案矩阵。

验收：每张卡有目标用户、触发、任务、进展、完成、自然频率、错误后果；至少两个软件和两个非软件替代；公开事实、个人观察、推断和假设分开；固定输入完成一次可复现比较；结论明确适合与不适合的人。
