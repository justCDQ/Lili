# 从功能描述改写用户问题

功能描述规定“做什么”，用户问题说明“谁在什么条件下无法取得什么进展，以及造成什么结果”。把预设功能还原为用户问题，可以重新打开方案空间，并让优先级、验证和验收建立在结果而非交付物上。

## 前置知识与能力边界

先掌握：

- [目标、场景、痛点、需求、方案与功能](../00-foundations/03-goal-scenario-pain-need-solution-feature.md)；
- [症状、根因、需求表达与解决方案请求](../02-problem-evidence/15-symptom-root-cause-need-solution-request.md)；
- [多来源交叉验证](../02-problem-evidence/16-cross-validation.md)；
- [目标用户与使用场景](01-target-user-context.md)。

问题改写用于判断是否值得解决、边界在哪里和哪些方案可比较。它不能代替技术可行性、商业目标、合规要求或后续原型验证。

## 1. 识别五个不同层次

| 层次 | 定义 | 示例 |
|---|---|---|
| 现象 | 可观察到的事件 | 用户重复点击“导出” |
| 症状 | 用户或系统感受到的异常 | 页面长时间没有反馈 |
| 用户问题 | 阻碍任务进展并产生结果的条件 | 月末大文件生成期间，财务人员无法判断任务是否仍在进行，因而重复提交并延误核对 |
| 根因 | 导致问题的机制 | 同步请求超时、任务状态未持久化、无进度入口 |
| 方案 | 改变现状的一种手段 | 异步任务中心、邮件通知、分批导出 |

一条证据可以支持多个层次，但不能跳步。例如“客户要求邮件通知”是方案请求；它可能来自“后台任务完成后用户不知道结果”，也可能来自“用户离开页面后需要在其他渠道接收状态”。两种问题对应的方案范围不同。

## 2. 功能描述为什么会锁死决策

常见输入包括：

- “增加 AI 摘要”；
- “做一个消息中心”；
- “支持批量审批”；
- “把按钮放大”；
- “增加导出 Excel”；
- “复制竞品的看板视图”。

这些句子省略了目标用户、触发、当前做法、约束、结果和证据。团队只能讨论排期和界面，无法判断不做会怎样、是否有更低成本方案、上线后测什么。

```mermaid
flowchart LR
    A["功能请求"] --> B["追问目标用户与触发"]
    B --> C["还原当前工作流"]
    C --> D["定位阻碍与结果"]
    D --> E["检查证据和根因"]
    E --> F["形成问题陈述"]
    F --> G["发散多个方案"]
    G --> H["验证关键假设"]
```

改写不是把功能换成更柔和的措辞，而是恢复这条推理链。

## 3. 可操作的问题陈述

使用以下结构：

```text
当【目标用户】处于【触发和具体场景】时，
为了【完成任务或取得进展】，
目前使用【现有做法或替代方案】，
但受到【可观察阻碍】影响，
导致【时间、质量、成本、风险或机会结果】。
已有证据：【事实、样本、时间窗】。
仍待验证：【根因或价值假设】。
```

每个字段承担不同决策作用：

| 字段 | 决策用途 | 缺失后果 |
|---|---|---|
| 目标用户 | 限定服务对象和 Reach 分母 | 范围无限扩张 |
| 触发与场景 | 确定需求何时成立 | 设计成脱离工作流的孤立功能 |
| 用户进展 | 定义最终价值 | 把交付功能误当成功 |
| 当前做法 | 识别替代方案和转换成本 | 假设用户在等待新产品 |
| 阻碍 | 确定需要改变的机制 | 只描述情绪，不可行动 |
| 结果 | 判断严重度与业务影响 | 无法排序和测量 |
| 证据 | 区分事实与意见 | 用声音大小代替可信度 |
| 待验证 | 暴露不确定性 | 把推断写成事实 |

## 4. 从功能到问题的七步方法

### 4.1 原样记录功能请求

保留请求者、日期、原话、来源和上下文，避免在转述时丢失限制。但原话不直接进入开发范围。

```text
请求：给任务列表增加“批量删除”。
来源：12 条工单中的 4 条；运营同事在周会再次提出。
时间窗：过去 60 天。
```

### 4.2 找到动作背后的目标

连续追问：删除后用户要获得什么进展？为什么现在发生？不删除会怎样？

可能得到：运营人员在活动结束后需要清理测试任务，否则它们干扰真实任务的统计和搜索结果。

### 4.3 还原当前工作流

记录入口、前置条件、正常步骤、绕路、失败和完成标准。当前工作流可能已经通过筛选、归档、标签或脚本解决部分问题。

### 4.4 分离现象、问题与根因

“列表里任务很多”是现象；“测试任务干扰运营查看真实任务”是问题；“测试和真实任务没有环境/类型隔离”可能是根因；“批量删除”只是一个方案。

### 4.5 量化结果

可使用：额外步骤、耗时、失败率、重复操作、支持成本、资金损失、合规风险和机会损失。没有可靠数字时标注估计区间和依据，不制造精确值。

### 4.6 写出至少一个反例

若任务数量多但筛选准确、统计排除测试数据，则“多”本身不是问题。反例帮助识别陈述是否把表面特征误当因果。

### 4.7 去掉方案词再检查

把“按钮、页面、AI、通知、看板、导出、自动化”等词删除后，问题仍应成立。若删除后句子没有意义，说明仍在描述方案。

## 5. 判断问题陈述是否合格

### 5.1 可证伪

“体验不好”无法明确证伪；“首次提交者中，无法在 5 分钟内完成身份校验的人占 31%，主要停在证件要求理解”可以用新数据推翻或修正。

### 5.2 不预设唯一方案

“用户收不到邮件提醒”仍可能锁定渠道。更开放的表达是“异步审核完成后，已离开页面的申请人无法及时获知结果和下一步”。方案可以是站内任务、推送、短信、邮件或流程调整。

### 5.3 覆盖完整结果

问题结果应指向用户进展，并记录业务影响。用户等待时间增加是直接结果；重复联系客服产生支持成本是业务结果。不能只写“影响体验”。

### 5.4 范围适中

“企业协作效率低”过宽；“团队成员在文档被外部修改后无法判断哪些段落变化，导致发布前重复核对”可进入方案探索。过窄的“diff 按钮颜色不明显”则可能只覆盖一个症状。

## 6. 应用案例一：把“增加批量删除”还原为数据隔离问题

### 6.1 输入证据

- 过去 60 天有 12 个相关工单，其中 8 个来自运营；
- 运营每周创建约 500 条测试任务；
- 真实任务列表默认混入测试任务；
- 分析报表有 6% 的事件来自测试任务；
- 任务已参与审计记录，物理删除会破坏追踪；
- 运营当前逐条加“测试”标签，但存在漏标。

### 6.2 初始功能请求

```text
在任务列表增加全选和批量删除，让运营快速清理测试数据。
```

### 6.3 分层分析

| 层次 | 结论 |
|---|---|
| 现象 | 列表存在大量测试任务 |
| 症状 | 搜索和统计混入测试数据 |
| 用户问题 | 运营无法稳定区分测试与真实任务，导致日常处理和指标失真 |
| 候选根因 | 环境/任务类型没有可靠标识，筛选默认值错误 |
| 请求方案 | 批量删除 |

### 6.4 改写后的问题

```text
当运营在活动配置和验收期间创建测试任务时，
为了在上线后只处理和统计真实任务，
目前依赖人工标签并在列表中逐条识别，
但测试身份不是系统级字段且报表默认不排除，
导致真实工作流被干扰、分析口径失真并产生人工清理成本。
```

### 6.5 方案比较

| 方案 | 解决范围 | 风险 |
|---|---|---|
| 批量物理删除 | 清理已有数据 | 破坏审计和关联记录；未来继续产生 |
| 软删除/归档 | 降低列表干扰 | 报表仍可能误算；权限和恢复复杂 |
| 独立测试环境 | 从来源隔离 | 建设和数据同步成本高 |
| 系统级 test 标识 + 默认过滤 | 同时修正列表与统计 | 需要改写产生链路和历史回填 |

当前证据指向“系统级标识 + 默认过滤 + 有权限的归档”，而不是物理删除。功能改写改变了工程不变量和数据方案。

### 6.6 验证与失败分支

回填历史数据前抽样 500 条，要求分类精确率达到约定门槛；灰度后监控真实任务误标率、报表差异和运营清理时间。如果标识依赖人工选择且漏标没有下降，说明根因尚未解决，需要从测试任务创建入口自动确定环境。

## 7. 应用案例二：把“增加 AI 摘要”还原为信息定位问题

### 7.1 输入证据

- 客服处理升级工单时平均阅读 18 条消息；
- 处理时长中位数 11 分钟，长线程为 24 分钟；
- 关键字段包括当前状态、已尝试操作、承诺时间和待办责任人；
- 工单正文可能包含错误陈述和敏感信息；
- 现有搜索能定位关键词，但不能稳定还原事件顺序。

### 7.2 初始请求

```text
在工单侧边栏自动生成 AI 摘要。
```

### 7.3 问题改写

```text
当二线客服接手长线程升级工单时，
为了在回复前确认当前状态、已执行动作和未完成承诺，
目前需要顺序阅读消息并手工记录，
但信息分散、重复且可能冲突，
导致接手时间增加，并可能重复操作或遗漏承诺。
```

该问题允许比较结构化字段、时间线、人工交接模板、规则抽取和模型摘要，而不是默认模型是唯一方案。

### 7.4 三种候选方案

| 方案 | 优点 | 成本与边界 |
|---|---|---|
| 强制交接字段 | 可验证、可审计 | 增加填写成本，历史工单缺失 |
| 事件时间线 + 搜索 | 保留原文和顺序 | 仍需要人工综合 |
| 模型生成结构化摘要并附引用 | 降低阅读量 | 可能遗漏、错误；需引用、权限与人工确认 |

可以先做时间线和四个结构化字段，再用离线样例评估模型补全。涉及退款、承诺或账户操作的事实必须回到原始消息确认，不能让摘要直接触发写操作。

### 7.5 验收指标

- 接手到首次正确动作的时间；
- 关键事实召回率和引用准确率；
- 重复操作与承诺遗漏率；
- 人工展开原文的比例；
- 模型延迟、失败率和单工单成本。

失败分支：阅读时间下降但重复操作上升，说明摘要压缩了信息却损害正确性，应停止扩大流量，检查关键事实定义、引用和人工确认，而不是只调短输出。

## 8. 方案请求仍然有价值

功能请求可能包含重要线索：

- 请求者知道某个约束；
- 竞品方案建立了用户预期；
- 方案可能是低成本修复；
- 法规或合同可能直接规定输出形式；
- 已有平台组件让某方案成本显著更低。

因此不要丢弃原请求。把它作为候选方案，与问题证据分开保存，并记录它试图解决哪一部分。

## 9. 调试问题陈述

当团队仍然争论时逐项检查：

1. 是否出现具体目标用户而不是“用户”；
2. 是否有触发、当前做法和完成标准；
3. 阻碍能否被观察，而非只有“体验差”；
4. 结果是否可度量或至少可验证；
5. 是否把请求者意见当作普遍事实；
6. 根因是否只是未经验证的推断；
7. 删除方案名后问题是否仍成立；
8. 是否包含一个明确非目标；
9. 解决问题是否推进用户任务，而非只改善局部点击；
10. 是否存在更大端到端流程中的真正瓶颈。

### 证据矩阵

```text
结论｜事实来源｜样本/时间窗｜支持强度｜反例｜待验证。
```

如果每条关键结论只有同一个来源，优先补交叉证据。行为数据能说明发生了什么，流程材料能解释约束，公开反馈能发现重复模式，原型实验能验证候选方案。

## 10. 优先级和工程边界

问题陈述进入优先级时还要结合：

- 用户结果与业务目标是否一致；
- 影响范围、频率、严重度和可信度；
- 法律、安全、可访问性等硬约束；
- 依赖、工作量和机会成本；
- 是否能在当前阶段验证关键假设。

工程不应从问题陈述直接推导实现。产品负责明确问题和结果；交互发散流程；工程验证机制、可靠性和安全边界；数据定义口径；相关角色共同决定方案。

## 12. 综合练习

选择三个真实功能请求，例如“批量操作”“消息通知”“AI 助手”。对每个请求：

1. 保存原始上下文；
2. 列出现象、症状、用户问题、候选根因和方案；
3. 用至少两类证据支持问题；
4. 写出一个反例；
5. 去掉方案词后重写；
6. 提出至少三个机制不同的方案；
7. 定义用户结果、成功指标和守护指标；
8. 设计一个会推翻当前判断的失败条件。

验收标准：问题陈述不包含唯一实现，其他人能据此提出不同方案；证据、推断和假设明确分开；失败条件能够真的停止或改变方案，而不是上线后只观察使用量。
