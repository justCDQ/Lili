# MVP 的核心用户与核心场景：建立最短而完整的价值闭环

MVP 的核心用户是首个版本明确服务、具有同类任务条件的一组主体；核心场景是这组主体在具体触发、约束和数据条件下，从当前状态走到可验证结果的端到端任务。选择核心用户与场景不是忽略其他人，而是使首个版本能够完整交付一种价值、暴露关键风险并产生可解释证据。

- [目标用户、场景与限制](../requirements-prioritization/01-target-user-context.md)；
- [从功能回到用户问题](../requirements-prioritization/02-feature-to-user-problem.md)；
- [范围与非目标](../requirements-prioritization/05-scope-non-goals.md)；
- [最关键且最不确定的假设](03-critical-assumption.md)；

本文不把“核心用户”当作一个虚构人物画像，也不要求用用户访谈作为唯一证据。日志、支持记录、任务观察、公开材料、现有研究、可运行验证和产品数据都可以用于确定范围，但必须说明适用边界。

## 1. MVP 不是最少功能

MVP 至少同时满足：

```text
明确主体
+ 真实触发
+ 完整任务
+ 可用结果
+ 必要失败恢复
+ 可测证据
+ 可运营边界
```

如果只交付入口、半个流程或不能使用的演示，它是原型或组件，不是可产生最小价值的产品版本。

```mermaid
flowchart LR
    A["触发前事实"] --> B["发现与进入"]
    B --> C["提供输入/作出选择"]
    C --> D["系统与人工处理"]
    D --> E["结果确认"]
    E --> F["恢复、后续与衡量"]
```

核心场景必须跨过整条链路，但只覆盖一组清楚的条件。

## 2. 核心用户不是人口统计标签

“年轻人”“中小企业”“所有管理员”通常不足以决定产品行为。

核心用户应由任务相关属性定义：

| 属性 | 示例 |
|---|---|
| 要完成的工作 | 每周批量导入 500 条以上资产 |
| 触发频率 | 至少每周一次 |
| 权限 | 可以创建资产但不能改系统字段 |
| 数据条件 | 使用支持模板，500–5,000 行 |
| 现有能力 | 能编辑 CSV，不会编写 API 程序 |
| 时间压力 | 需在营业前完成 |
| 风险 | 错误写入会污染库存 |
| 平台 | 桌面浏览器与键盘 |
| 支持条件 | 可以访问组织帮助中心 |

合格表达：

```text
在 20–200 人团队中，
每周至少一次从已支持来源迁移 500–5,000 条资产，
拥有资产创建权限、能准备 CSV，
但没有工程支持的工作区管理员。
```

## 3. 使用者、决定者和受影响者

一个 MVP 可能涉及：

- 操作者：直接执行任务；
- 受益者：获得结果；
- 管理员：配置和授权；
- 付款者：承担费用；
- 审核者：确认高风险结果；
- 支持者：处理失败；
- 被影响者：数据或权利受到改变。

核心用户通常指主要操作者或受益者，但 MVP 必须处理其他角色的必要边界。

例如 AI 回复草稿的操作者是一线人员，受益者是请求帮助的客户，管理员决定数据源，审核者承担发送责任，安全团队约束正文处理。只优化操作者速度可能损害客户正确性。

## 4. 选择核心用户的标准

### 4.1 问题强度

- 阻碍频繁；
- 当前成本高；
- 失败后果明确；
- 已有替代方式不足；
- 结果变化可以观察。

### 4.2 方案匹配

- 候选机制直接作用于阻碍；
- 所需输入和权限可获得；
- 用户能承担迁移和学习；
- 目标平台可实现；
- 不依赖尚不存在的组织能力。

### 4.3 可安全服务

- 数据使用有依据；
- 权限可以正确执行；
- 失败能够恢复；
- 支持路径存在；
- 不把高风险对象作为未经控制的第一批。

### 4.4 学习代表性

首批用户不需要代表全部市场，但应能回答下一决定：

- 核心机制是否有价值；
- 哪些边界需要扩大；
- 哪类异常最常见；
- 单位成本如何；
- 需要什么运营能力。

只选择内部专家可能使结果偏乐观；只选择最复杂客户可能使范围无法闭合。

## 5. 排除条件

排除不是永久拒绝。每项排除需要原因和重开条件：

```yaml
excluded_group:
  condition: "超过 50,000 行或含自定义脚本列"
  reason: "当前解析、任务进度和回滚没有容量证据"
  safe_alternative: "迁移服务"
  review_trigger: "5,000 行范围达到质量门槛且完成容量测试"
```

常见排除：

- 未支持数据格式；
- 没有必要权限；
- 需要专业责任判断；
- 网络或设备条件不满足；
- 受特定监管约束；
- 任务频率太低，迁移成本无法回收；
- 异常需要尚未建立的人工支持。

入口必须能判断适用性并提供替代，不能让用户完成全部输入后才发现不支持。

## 6. 核心场景的组成

完整场景包含：

```yaml
scenario:
  actor: "支持模板范围内的工作区管理员"
  trigger: "需要从旧系统导入第一批资产"
  start_state:
    - "已登录"
    - "拥有 assets.create"
    - "有一个 UTF-8 CSV"
  desired_result: "资产被正确创建且批次可审计、可撤销"
  constraints:
    - "500–5,000 行"
    - "桌面浏览器"
    - "错误行不得静默跳过"
  path:
    - "选择文件"
    - "解析与映射"
    - "dry run"
    - "修正或确认"
    - "提交"
    - "查看逐项结果"
  recovery:
    - "解析失败保留文件摘要"
    - "提交未知时按 batchId 查询"
    - "部分失败可导出错误"
    - "整批新建可撤销"
  completion:
    - "权威数据库达到预期"
    - "结果与预览一致"
    - "有批次审计记录"
```

## 7. 触发比入口更重要

入口是“点击导入”；触发是“用户有一批旧系统数据需要迁移”。同一入口可能被好奇、测试或误触打开。

记录：

- 触发前发生什么；
- 用户从哪里取得输入；
- 为什么此时必须完成；
- 现有工作如何处理；
- 不完成会造成什么；
- 结果进入下一步什么工作。

这决定 MVP 是否真正连接任务。

## 8. 完整不等于覆盖全部

核心场景应完整处理：

- 理想输入；
- 最常见可修正错误；
- 权限拒绝；
- 请求失败或超时；
- 用户取消；
- 结果确认；
- 一条安全恢复路径。

可以暂不处理：

- 低频格式；
- 自定义扩展；
- 高级批量配置；
- 跨组织迁移；
- 自动化 API；
- 与核心结果无关的偏好。

不能删掉：

- 授权；
- 数据正确性；
- 错误反馈；
- 取消和未知结果；
- 基本无障碍；
- 支持与审计；
- 结果衡量。

## 9. 核心路径与 Happy Path 的区别

Happy Path 通常假设一切正常。核心路径是目标条件下可持续完成任务的最短路径，必须承受合理失败。

```text
Happy Path：
上传 → 成功

核心路径：
校验适用性
→ 上传
→ 解析
→ 映射
→ dry run
→ 处理可修正错误
→ 确认范围
→ 提交
→ 权威结果
→ 失败恢复/撤销
```

后者才提供可用价值。

## 10. 场景状态表

| 状态 | 进入条件 | 界面责任 | 服务端责任 | 下一步 |
|---|---|---|---|---|
| 不适用 | 格式/规模超范围 | 尽早说明替代 | 不创建批次 | 使用服务或退出 |
| 准备 | 条件符合 | 说明格式和数据边界 | 返回上传策略 | 选择文件 |
| 解析 | 文件已接收 | 显示真实进度和取消 | 隔离、扫描、解析 | 预览或失败 |
| 可修正 | 有字段错误 | 定位错误、保留工作 | 返回稳定错误码 | 修正 |
| 可提交 | dry run 通过 | 显示影响范围 | 再次授权和校验 | 确认 |
| 提交中 | 写入任务存在 | 防重复并提供 task ID | 幂等执行 | 结果 |
| 部分成功 | 逐项结果不同 | 分组并导出错误 | 返回逐项事实 | 重试失败项 |
| 未知 | 客户端超时 | 不猜成功/失败 | 可按 ID 查询 | 对账 |
| 成功 | 权威事实完成 | 结果和下一步 | 审计与版本 | 使用/撤销 |

## 11. 核心用户选择矩阵

| 候选组 | 问题强度 | 方案匹配 | 安全可控 | 学习价值 | 结论 |
|---|---|---|---|---|---|
| 每周导入的普通管理员 | 高 | 高 | 高 | 高 | 核心 |
| 一次性小文件用户 | 低 | 高 | 高 | 低 | 后续 |
| 超大自定义迁移 | 高 | 低 | 低 | 中 | 人工服务 |
| 无创建权限成员 | 中 | 不适用 | 高 | 低 | 不提供入口 |
| API 技术用户 | 高 | 中 | 高 | 高 | 独立方案 |

矩阵不使用精确总分。写出证据和反例。

## 12. 场景范围与产品范围

一个核心场景常需要跨多个组件：

```mermaid
flowchart LR
    H["帮助与适用性"] --> U["上传"]
    U --> P["解析服务"]
    P --> V["预览与修正"]
    V --> W["批次写入"]
    W --> R["结果与撤销"]
    R --> S["支持与审计"]
```

MVP 不能因为团队只拥有上传页，就把解析失败和撤销排除。组织边界不是用户结果边界。

若跨团队依赖无法获得：

- 调整方案；
- 缩小但仍闭合的结果；
- 提供人工桥接；
- 停止当前 MVP；
- 不用静态假数据冒充已完成。

## 13. 案例一：知识库问答

### 13.1 初始“所有员工”范围

“所有员工可以问任何公司问题”包含：

- 多语言；
- 不同权限；
- 时效规则；
- 人事、财务、法律高风险；
- 无答案；
- 冲突来源；
- 复杂表格；
- 写操作。

首版无法建立可解释边界。

### 13.2 核心用户

```text
已登录的内部支持人员，
在处理产品使用问题时，
需要从已批准、版本化的公开支持文档中
找到当前适用步骤；
不包含客户隐私、合同解释和写操作。
```

选择理由：

- 高频、可观察；
- 文档范围明确；
- 权限单一；
- 结果可由引用验证；
- 无答案可安全升级；
- 能形成检索、引用和反馈的学习闭环。

### 13.3 核心场景

触发：人员收到产品配置问题。

输入：

- 问题文本；
- 产品版本；
- 语言；
- 当前时间；
- 已认证主体。

路径：

1. 输入问题；
2. 系统检索批准文档；
3. 显示有来源的答案或无答案；
4. 人员打开来源核对；
5. 复制或修改后用于回复；
6. 标记结果是否有用。

失败：

- 没有相关证据；
- 来源冲突；
- 文档过期；
- 权限变化；
- 请求超时；
- 引用无法打开。

### 13.4 非目标

- 不自动发送；
- 不执行账户操作；
- 不回答合同和退款资格；
- 不检索个人工单；
- 不把自由反馈直接用于训练。

### 13.5 完成

核心结果不是“模型返回文字”，而是：

```text
支持人员在固定时间内取得由当前批准文档支持的可用步骤，
或正确进入无答案/升级状态。
```

## 14. 案例二：团队首次配置

### 14.1 候选用户

- 1–5 人个人团队；
- 20–200 人成长团队；
- 1,000 人以上企业；
- 合作伙伴实施人员；
- 内部试用人员。

成长团队的管理员被选为首版：

- 成员和权限配置已经产生明显成本；
- 决策链比大型企业短；
- 无需定制身份系统；
- 数据量能暴露真实状态；
- 价值和支持成本可衡量。

### 14.2 核心场景

```text
拥有创建工作区权限的管理员，
在新工作区建立后 7 天内，
添加 5–50 名成员，
选择一个受支持的角色模板，
连接一个数据源，
并让至少一名成员完成首次任务。
```

起点：

- 工作区已创建；
- 管理员已认证；
- 套餐支持；
- 成员名单可获得。

完成事实：

- 成员加入正确工作区；
- 权限与预览一致；
- 数据源连接通过；
- 至少一项真实任务完成；
- 错误和支持记录可追踪。

### 14.3 必要异常

- 成员已属于工作区；
- 邮件投递未知；
- 角色在接受前改变；
- 管理员权限被撤销；
- 数据源认证过期；
- 成员使用不同账户；
- 会话中断后恢复。

### 14.4 排除

- 自定义 SSO/SCIM；
- 500 人批量同步；
- 跨组织域验证；
- 自定义角色表达式；
- 多地区数据驻留。

为大型企业提供明确联系和迁移路径，而不是让其进入后失败。

## 15. 核心指标

### 15.1 端到端完成

```text
核心场景完成率
= 在时间窗内满足全部完成事实的唯一主体数
÷ 进入核心场景且符合范围的唯一主体数
```

### 15.2 阶段指标

- 符合范围；
- 开始；
- 输入准备；
- 解析/连接；
- 确认；
- 权威结果；
- 后续首次任务；
- 恢复；
- 支持升级。

阶段指标用于定位，不替代核心结果。

### 15.3 护栏

- 错误权限；
- 无权数据暴露；
- 不可恢复副作用；
- 支持量；
- P95 等待；
- 无障碍阻断；
- 错误后工作丢失；
- 不符合范围用户误入。

## 16. 入口资格判断

入口可以问最少问题：

```yaml
eligibility:
  file_format: "csv"
  row_count: 1840
  source: "supported-template-v2"
  capability: "assets.create"
```

服务端重新判断权限和套餐。不能只依赖前端隐藏。

不符合时显示：

- 哪条条件不满足；
- 数据是否已上传；
- 是否会保留；
- 有什么替代；
- 如何取得支持；
- 何时扩大范围。

## 17. 迭代扩展

扩大 MVP 的条件来自证据：

```text
核心场景达到质量和护栏门槛
→ 选择相邻用户或场景
→ 识别新增假设
→ 补充机制与恢复
→ 灰度
```

相邻扩展可以是：

- 相同用户 + 更大数据；
- 相同用户 + 新格式；
- 新用户 + 相同任务；
- 相同任务 + 新平台；
- 相同机制 + 更高自动化；
- 低风险 → 高风险。

一次只扩大可归因范围。

## 18. 调试范围

### 完成率低

按资格、入口、角色、数据、平台和失败状态切片。检查分母是否混入不符合范围用户。

### 完成率高但价值不明显

检查完成事实是否过浅。上传成功不等于资产可用，答案生成不等于问题解决。

### 支持量高

区分：

- 资格边界不清；
- 输入准备困难；
- 错误不可恢复；
- 产品术语错误；
- 运营或依赖故障；
- 高风险用户误入。

### 核心用户太少

确认问题规模和频率。若必须不断扩大范围才能证明价值，可能核心问题不值得产品化。

### 核心用户定义变化

版本化定义。历史指标继续引用旧版本，不能用新定义重算后直接比较。

## 19. 常见失败模式

### 19.1 用职位作为唯一用户定义

同为“管理员”，任务频率、权限和数据规模差异很大。

### 19.2 只选最容易接触的人

内部员工和专家常有额外知识。说明偏差并补目标条件。

### 19.3 只做主页面

没有入口、错误、结果和支持，价值链不完整。

### 19.4 把异常全部移到后续

权限、超时、无效输入和未知结果属于核心可靠性。

### 19.5 用功能数量定义 MVP

“只有三个功能”不说明能否完成任务。

### 19.6 不说明不适用

用户会误入并承担无意义成本。入口需早判断。

### 19.7 核心用户永久化

首版范围是当前证据下的选择，应有扩展和停止条件。

## 20. MVP 范围卡

```yaml
mvp:
  version: 1
  core_user:
    task: "批量导入资产"
    frequency: ">= 1/week"
    capability: "assets.create"
    data: "supported CSV, 500–5,000 rows"
    platform: "desktop web"
  core_scenario:
    trigger: "从旧系统迁移一批资产"
    result: "正确、可审计、可撤销的批次"
    window: "一次会话或可恢复任务"
  must_handle:
    - "格式错误"
    - "权限撤销"
    - "部分失败"
    - "未知结果"
    - "撤销"
  excluded:
    - "自定义脚本列"
    - "> 5,000 rows"
  alternative: "受控迁移服务"
  success:
    - "核心完成率 >= 80%"
    - "严重数据错误 = 0"
    - "不可撤销批次 = 0"
  review:
    date: "2026-08-15"
    trigger: "连续两周护栏通过"
```

## 21. 练习

选择一个 MVP：

1. 列出操作者、受益者、管理员、付款者、审核者和被影响者；
2. 用任务频率、权限、数据和约束定义 3 个候选用户组；
3. 选一个核心组并说明排除；
4. 写触发、起点、路径、结果和后续；
5. 列最常见错误、权限、超时、取消和未知结果；
6. 明确完成事实；
7. 写端到端指标和护栏；
8. 为不符合范围者提供安全替代；
9. 写扩大、停止和重新评审条件；
10. 让第三方从范围卡复现一次完整任务。
