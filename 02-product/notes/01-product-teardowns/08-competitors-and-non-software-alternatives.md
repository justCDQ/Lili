# 竞品与非软件替代方案比较

竞品分析不是把多个产品的功能数量放进表格。用户面对的是一个待完成任务，会在直接竞品、间接工具、人工流程、组合工具、延后处理和不做之间选择。有效比较必须让不同方案在相同条件下完成同一个任务，再解释各自的结果、成本和边界。

## 一、先定义比较任务

比较对象不是“日历产品”，而可以是：

> 一名组织者要在三天内，为分处三个时区的五人安排一次 45 分钟会议；所有人确认后建立日历事件，避免与已有事件冲突，并允许一人取消后重新安排。

这个定义包含角色、人数、时间限制、完成标准和异常分支。缺少这些条件，“更方便”“协作更强”都无法验证。

任务卡至少记录：

| 字段 | 含义 |
| --- | --- |
| 目标角色 | 谁承担协调与结果责任 |
| 参与者 | 人数、账号条件、语言、时区 |
| 输入 | 候选日期、已有忙闲、联系方式 |
| 完成标准 | 什么状态才算安排成功 |
| 失败分支 | 冲突、未回复、取消、权限不足 |
| 时间边界 | 何时开始，最晚何时完成 |
| 风险边界 | 隐私、费用、不可逆操作 |

## 二、四类替代方案

### 1. 直接竞品

面向相近用户，以相近机制完成相同任务。例如两个预约或会议协调工具。直接竞品适合比较任务流程、默认值、恢复路径和商业边界。

### 2. 间接竞品

机制不同，但结果相同。例如用项目管理工具收集所有人的时间，再人工创建日历事件。它可能没有“预约”定位，却参与同一选择。

### 3. 非软件或组合方案

包括电话、线下协调、邮件、纸笔、共享表格、群消息，以及“表格收集 + 人工检查 + 日历创建”。现实中的替代通常是多个低成本工具的组合，而不是一个完整产品。

### 4. 延后或不做

如果任务价值低于协调成本，用户可能取消会议或改用异步文档。把“不做”排除，会高估需求强度和付费意愿。

分类依据是同一任务下的替代关系，不是行业标签。一个工具在个人一对一预约中可能是直接竞品，在五人协商中却只能覆盖部分步骤。

## 三、冻结条件，保证可比

产品、套餐和页面会持续变化。每轮比较必须固定：

- 观察日期与复测日期；
- 国家或地区、语言、币种和税费条件；
- 账号类型、套餐、试用状态与管理员策略；
- 桌面端或移动端、浏览器和辅助技术；
- 相同参与者、忙闲数据、时间窗口和异常；
- 是否允许安装扩展、授权日历或导入联系人；
- 从哪个入口开始，到哪个结果结束。

不同套餐不能直接用功能数量排名。免费方案和企业方案可以比较，但结论必须说明成本与权限条件，不可写成产品的普遍能力。

## 四、证据分层

### 官方声明

来自帮助文档、价格页、条款、更新日志和接口文档。它可以证明提供者公开承诺了什么，不能单独证明任务在你的条件下一定成功。

### 亲自观察

固定账号和输入后的页面、步骤、耗时、错误和输出。记录截图或操作日志，但结论只适用于测试条件。

### 外部行为数据

支持工单、搜索趋势、公开状态页、应用商店反馈和社区问题可用于发现模式。它们有选择偏差，不能直接当总体发生率。

### 分析推断与待验证假设

“多一步授权可能降低完成率”是推断；“参与者不愿授权日历”是待验证假设。应写可证伪条件，例如在受控任务中测量授权页退出和完成率，而不是伪装成事实。

```json
{
  "claim": "方案 A 会自动避开组织者已有事件",
  "evidence_type": "官方文档与控制测试",
  "observed_on": "2026-07-17",
  "conditions": {
    "region": "中国",
    "account": "测试账号",
    "calendar_permission": "已授权",
    "device": "桌面浏览器"
  },
  "result": "测试中的两个冲突时段未显示为可预约",
  "boundary": "未测试多个日历、企业管理员限制与移动端"
}
```

## 五、比较结果，不比较功能数量

同一任务可沿以下维度记录：

### 任务覆盖

方案覆盖了从候选时间到确认、通知、改期和取消的哪些环节？未覆盖部分由谁补齐？

### 总任务成本

包括订阅、配置、授权、学习、操作、等待、人工检查、错误恢复和迁移。一个月费为零的方案，仍可能有较高协调成本。

### 结果质量

检查是否建立正确事件、时区是否正确、参与者是否收到确认、冲突是否被处理，以及取消后状态是否一致。

### 可靠性与恢复

记录重复提交、参与者未回复、权限被撤销、外部服务失败和错误输入时的反馈、重试、回退与人工接管。

### 控制权与可逆性

谁能查看忙闲和身份信息？组织者能否修改结果、导出历史、撤回授权和删除数据？错误操作是否可恢复？

### 切换成本

包括历史数据导出、已有链接替换、流程培训、集成改造、审批和旧系统双轨期。锁定不是只有“不能导出”，也包括迁移成本高到用户不愿行动。

## 六、建立可执行的测试协议

1. 写出唯一任务、完成标准和失败条件；
2. 选择两个软件方案、一个非软件或组合方案，并保留“不做”；
3. 创建同一组测试账号、日历事件和三个时区；
4. 为每个方案从零开始，不沿用另一个方案的学习结果；
5. 记录每一步的操作者、系统状态、等待时间和结果；
6. 主动触发冲突、取消、未回复和权限撤销；
7. 保存官方依据、观察日志和推断，三者分列；
8. 输出条件化选择规则，不选脱离场景的“总冠军”；
9. 对价格、套餐和关键能力设置复测日期。

为减少顺序偏差，可改变方案测试顺序；为减少熟练度偏差，可让另一名未参与设计的人按同一协议执行。个人测试仍不能代表总体用户，只能证明在记录条件下可复现。

## 七、完整案例：五人跨时区排会

### 1. 方案与边界

本案例比较：

- 方案 A：Google Calendar 预约安排；
- 方案 B：一个具备候选时间投票的独立协调工具；
- 方案 C：邮件 + 共享表格 + 组织者人工创建事件；
- 方案 D：取消会议，改用异步文档评论。

Google 官方帮助说明，预约安排可创建预约页面、展示可用时间并在发生日历冲突时阻止相应时间；具体功能可能受账号和订阅影响。方案 B 不写品牌能力结论，必须在测试账号中观察。方案 C 和 D 是非单体软件替代。

### 2. 固定输入

```text
参与者：组织者 1 人，参与者 4 人
时区：Asia/Shanghai、Europe/London、America/Los_Angeles
会议时长：45 分钟
候选范围：未来 3 个工作日
冲突：组织者有 2 个忙碌时段
异常：1 人在首次确认后取消
完成：5 人收到时区正确的新事件，旧事件不残留
隐私边界：参与者不应看到他人的事件标题
```

这些是测试输入，不是对真实用户群体的统计描述。

### 3. 执行记录模板

| 步骤 | 方案 A | 方案 B | 方案 C | 方案 D |
| --- | --- | --- | --- | --- |
| 建立候选 | 记录步骤与权限 | 记录步骤与权限 | 表格录入 | 不需要 |
| 收集选择 | 预约页 | 候选投票 | 邮件回复 | 文档评论 |
| 检查冲突 | 日历机制 | 实测结果 | 人工检查 | 不适用 |
| 建立结果 | 自动或确认后创建 | 实测结果 | 人工创建 | 无会议 |
| 取消改期 | 实测恢复路径 | 实测恢复路径 | 邮件重发 | 不适用 |
| 数据与权限 | 官方声明 + 验证 | 官方声明 + 验证 | 各工具权限 | 文档权限 |

表格中不能预填“更快”“更安全”。先记录事实，再根据任务权重做判断。

### 4. 示例观察结果

以下为教学场景数据，不是对当前产品的真实测评：

| 方案 | 组织者主动操作 | 等待轮次 | 首次完成 | 取消后恢复 | 主要未覆盖工作 |
| --- | ---: | ---: | --- | --- | --- |
| A | 7 分钟 | 1 | 成功 | 重新选择时段 | 参与者共同协商偏好 |
| B | 9 分钟 | 2 | 成功 | 重开投票 | 最终事件权限复核 |
| C | 18 分钟 | 3 | 成功 | 人工汇总并重发 | 冲突检查、提醒、状态同步 |
| D | 5 分钟 | 0 | 达成异步目标 | 不适用 | 不提供实时讨论 |

主动操作时间不等于日历总时长。等待轮次、参与者负担和恢复动作必须一起解释。

### 5. 从事实到选择规则

**已观察事实**：在场景数据中，A 的组织者操作时间最短；C 需要更多人工汇总；D 没有会议，但异步文档完成了信息收集目标。

**分析推断**：当任务是低频、参与者不愿授权外部服务且组织者需要完全控制候选范围时，C 的额外步骤可能是可接受成本。当实时讨论不是必要条件时，D 可能比任何排会工具更接近完整目标。

**待验证假设**：参与者对预约页的接受程度高于表格。验证方式是对同类真实任务随机分配入口，比较完成率、完成时间和隐私投诉；在没有数据前不写成结论。

### 6. 失败分支

- 若 A 所需功能不在当前账号套餐，应记录“条件不满足”，不能用另一个套餐的视频代替；
- 若 B 在参与者取消后保留旧事件，应把清理成本计入恢复，不只记录首次成功；
- 若 C 的表格暴露所有人的忙闲细节，应缩减字段或改为组织者单独收集；
- 若 D 导致关键争议无法收敛，则回到会议方案，异步不是无条件替代；
- 若产品更新导致步骤或价格变化，应建立新观察版本，不静默修改旧记录。

## 八、如何处理评价、价格与营销材料

评价适合寻找待验证问题，不适合直接计算普遍发生率。价格页应记录币种、地区、套餐、计费周期、席位和税费。营销文案属于官方声明，必须通过帮助文档或实际任务确认边界。

对无法登录或无法购买的方案，可以完成文档层比较，但必须把结果标记为“未执行任务”。没有观察证据时，不应填入完成时间、可靠性或恢复能力。

## 九、输出不是排名，而是条件化决策

有效结论应采用以下形式：

```text
当 [任务条件]、[账号条件]、[风险边界] 成立时，
方案 X 更适合，因为 [可观察结果]；
代价是 [未覆盖工作与切换成本]；
如果 [关键假设失效]，改选方案 Y。
```

示例：当组织者已有统一日历、会议为一对多预约且参与者无需共同协商时，预约页能减少人工收集；若任务需要多人表达多个偏好并共同收敛，则应测试投票机制；若频率低且信息敏感，人工方案可能以更多步骤换取控制；若实时会议无必要，异步处理应进入候选集合。

## 十、完成检查与练习

一份合格比较应满足：

- 比较的是同一任务而非整个产品；
- 至少包含两个软件方案、一个非软件方案和“不做”；
- 日期、地区、套餐、设备、输入和完成标准一致；
- 官方声明、亲测事实、推断和假设分开；
- 记录首次成功、失败恢复、权限和迁移，不只记正常路径；
- 总任务成本包含学习、等待、人工和切换；
- 结论说明适合谁、不适合谁、何时失效；
- 可变信息有复测日期。

练习：选一个你每周实际完成的任务，按本章协议执行两个软件方案和一个手工方案。保留原始步骤和证据，主动触发一次失败，最后写三条条件化选择规则。完成标准是另一位读者能在相同账号条件下复现，并能区分观察结果与分析判断。
