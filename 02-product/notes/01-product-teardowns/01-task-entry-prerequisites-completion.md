# 具体任务、入口、前置条件与完成标准

产品拆解应从一个用户能够完成并验证结果的任务开始，而不是从页面、按钮或功能清单开始。任务边界确定后，才能比较入口、前置条件、步骤、失败和替代方案。

## 任务是什么

具体任务描述某个角色在特定场景中，对一个对象执行动作并得到可观察结果。推荐名称使用“动作 + 对象 + 结果”：

- 将文档共享给组织外协作者，并限制为只读；
- 导出本月账单，并确认文件包含全部交易；
- 取消重复订单，并确认没有再次扣款；
- 邀请五名成员加入项目，并为失败对象恢复。

“进入设置”“打开共享面板”“点击导出”只描述操作步骤。用户即使完成这些动作，也可能没有得到目标结果。

### 任务粒度

任务过大时无法复现，例如“管理团队”；过小时只剩控件，例如“点击加号”。判断粒度可使用三项标准：

1. 有明确触发和结束；
2. 结束后用户可以判断结果；
3. 可以在一次受控测试中重复执行。

“完成新成员入职”可能跨多个产品和多天，适合服务拆解；“邀请成员并确认其获得项目只读权限”更适合功能拆解。

## 入口

入口是用户开始任务的实际触点，不只包括主导航：

- 页面按钮、菜单、右键和快捷键；
- 搜索结果、命令面板和最近项目；
- 通知、邮件、短信和浏览器深链；
- 空状态、错误提示和升级引导；
- 移动端分享菜单、桌面扩展和 API；
- 产品外搜索结果、客服或人工服务。

每个入口记录：适用角色、出现条件、可发现性、进入后的状态、是否保留上下文、能否到达同一完成结果。

入口数量多不一定更好。不同入口若进入不同权限规则、旧流程或不一致默认，会增加错误。拆解要验证入口等价性，而不是统计按钮数量。

## 前置条件

前置条件是任务开始或继续所需的状态。至少分为：

| 类型 | 示例 | 不满足时应发生什么 |
| --- | --- | --- |
| 身份 | 已登录、已验证邮箱 | 引导认证并返回原任务 |
| 权限 | 文档所有者、账单管理员 | 说明无权并提供申请路径 |
| 数据 | 文档已保存、存在可导出交易 | 显示空状态或补充要求 |
| 系统状态 | 订单未取消、链接未过期 | 阻止冲突动作并解释当前状态 |
| 设备与渠道 | 支持的浏览器、可用存储 | 说明限制并提供替代 |
| 外部依赖 | 邮件服务、支付或身份系统可用 | 保存进度并允许查询或恢复 |
| 规则 | 套餐、地区、组织政策允许 | 在操作前显示限制 |

必要前置和可选前置要分开。收件人已有账号可能让共享更顺畅，但若产品支持公开只读链接，就不是必要条件。

### 隐性前置

只测试成熟账号容易漏掉隐性条件：管理员已配置域名、历史同意仍有效、浏览器已有缓存、用户熟悉术语或测试数据已准备。使用新账号、无权限账号、空数据和第二设备能暴露这些依赖。

## 完成标准

同一任务有四层完成：

| 层级 | 含义 | 共享案例 |
| --- | --- | --- |
| 操作完成 | 用户发出动作 | 点击“创建链接” |
| 请求接受 | 产品接收并处理 | 显示链接生成中 |
| 系统完成 | 权威状态已经改变 | 只读策略持久化 |
| 用户结果 | 目标对象获得预期结果 | 外部对象可读且不可编辑 |

异步任务中四层可能相隔很久。页面显示“已提交”不等于邮件送达、文件生成或退款入账。

### 用户完成标准

用户完成标准描述目标结果：对方能访问、文件能核对、款项状态正确。它必须由目标角色或可观察结果验证。

### 系统完成标准

系统标准描述权威状态、权限和可恢复性：策略已保存、任务可查询、重复执行不会产生第二次副作用、结果可撤销或过期。

### 失败完成

任务无法完成时，也需要稳定结束状态：明确失败原因类别、哪些数据已保存、下一步是什么。无限加载、关闭页面即丢失和统一“请重试”不属于合格完成。

## 证据方法

拆解记录分为三类：

- 观察事实：在指定环境中亲自看到的行为，或官方材料明确说明的规则；
- 推断：基于事实形成的解释，例如菜单位置可能强调当前文档任务；
- 假设：改动后的预测，例如新增深链会提高完成率。

观察事实仍有限定条件。一次桌面端测试不能代表移动端或所有套餐。推断和假设必须给出反证条件。

### 个人可执行证据

1. 使用自己控制的账号和非敏感测试数据复现；
2. 对照帮助、权限、定价和限制文档；
3. 使用第二账号或无痕窗口验证结果；
4. 查看公开评论、Issue、FAQ 和已知问题寻找失败；
5. 在同一任务下拆解竞品与非软件替代；
6. 有授权时使用漏斗、错误和支持数据验证频率。

评论能证明某种问题被报告，不能直接证明总体发生率。帮助文档能证明公开规则，不能证明所有环境的实际行为。

## 完整案例：外部只读共享

### 输入与证据

使用一份自己创建、不含敏感内容的私有文档，以及所有者、普通成员、组织外测试账号和未登录浏览器。官方帮助说明共享方式，权限文档说明角色能力。测试环境记录日期、客户端、套餐和组织策略。

当前证据：所有者菜单存在共享入口；普通成员看得到文档但不能改变组织级共享策略；帮助文档声明链接可撤销。这些是观察事实。“入口适合高频协作”只是推断。

### 步骤一：确定任务边界

任务起点是所有者决定让组织外对象查看当前文档；终点是外部对象能够阅读但不能编辑，所有者能确认和撤销。复制链接只是中间步骤。

非目标包括共同编辑、公开搜索收录、批量共享和跨组织数据迁移。

### 步骤二：枚举入口

检查文档菜单、工具栏、全局搜索、命令面板、通知和已复制深链。记录入口是否出现、需要几步、是否保留当前文档和权限上下文。

若全局搜索只能找到文档、不能直接开始共享，它是到文档的入口，不是独立共享入口。

### 步骤三：验证前置条件

分别测试所有者、普通成员、未登录状态、组织策略禁止外部共享、文档未保存和离线。每种状态记录提示、已输入内容是否保留和恢复动作。

权限拒绝应引导申请或联系管理员；不能通过直接 URL 绕过。组织政策不允许时，不应让用户完成多个步骤后才得知。

### 步骤四：执行成功路径

所有者选择只读范围并创建访问。保存操作提示、当前策略和链接。用外部测试账号访问，尝试查看、编辑、评论、下载和转发。不要根据“只读”标签假定所有动作都被禁止。

### 步骤五：验证系统状态

刷新所有者页面，重新打开权限设置，确认策略仍存在；在另一个浏览器访问；撤销后重新测试旧链接。若权限有缓存，记录生效时间，不把一次旧窗口结果概括为永久行为。

### 输出

产出三张表：

1. 入口表：角色、出现条件、步骤、上下文和结果；
2. 前置表：条件、必要性、不满足表现和恢复；
3. 完成表：操作、请求、系统、用户结果和验证证据。

最终观察结论限定为测试环境：所有者可以通过文档菜单创建并撤销只读访问；外部账号可读且编辑动作被拒绝。对后端存储和权限实现不作推断。

### 验证

- 第二账号确认用户结果；
- 权限页面刷新确认系统状态；
- 撤销后旧链接失效；
- 普通成员和未登录对象不能升级权限；
- 离线或请求失败时不显示最终成功；
- 重复创建不会生成无法管理的遗留访问。

### 失败分支

- 帮助与实测不一致：检查套餐、地区、组织策略和版本，结论降为未知；
- 链接可读也可编辑：完成标准失败，保存证据并停止；
- 显示撤销但旧链接仍长期有效：系统完成未验证；
- 无第二账号：只能验证操作与可见策略，不能声称用户结果完成；
- 外部邮件未送达但深链可访问：分开邀请通知与访问结果；
- 组织禁止外部共享：记录前置失败和转介，不重复尝试。

## 非软件替代

相同目标可通过发送 PDF、截图、纸质副本或人工转述完成。统一比较：更新是否同步、能否撤销、访问是否受控、接收方是否可搜索、泄露和制作成本。

一次性低风险查看中，PDF 可能足够；内容持续变化且需要撤销时，受控链接更合适。非软件方案不是落后基线，而是真实选择。

## 拆解检查表

- 任务名是否描述结果；
- 起点和终点是否可复现；
- 是否覆盖导航外入口；
- 必要和可选前置是否分开；
- 新账号、空数据和无权限是否测试；
- 四层完成是否分别验证；
- 是否由第二角色或权威状态确认；
- 失败时是否说明保存状态与恢复；
- 事实、推断和假设是否分栏；
- 是否与非软件替代按同一标准比较。

## 任务对比指标

入口或流程变化需要用任务指标验证，而不是只看入口点击：

| 指标 | 计算 | 需要同时检查 |
| --- | --- | --- |
| 入口到达率 | 进入任务的符合条件用户 / 产生任务意图的用户 | 意图识别是否可靠 |
| 任务完成率 | 达到用户完成标准的任务 / 已开始任务 | 失败与放弃是否保留在分母 |
| 首次成功时间 | 触发到首次用户结果的时长 | 异步等待与产品外步骤 |
| 前置阻断率 | 因缺少前置而停止的任务 / 已开始任务 | 权限、数据和系统状态切片 |
| 假成功率 | 显示成功但用户结果未完成 / 显示成功任务 | 第二账号或权威状态 |
| 恢复成功率 | 失败后最终完成的任务 / 可恢复失败 | 重试是否产生重复副作用 |

个人拆解没有总体数据时，可以在固定测试矩阵中报告原始计数，例如 12 条路径中 10 条达到系统完成、9 条达到用户结果。不能把受控测试比例当成真实用户发生率，但它能发现路径缺口并支持版本比较。

## 练习

选择“导出账单”“取消订单”或“邀请成员”之一，建立完整任务卡并实际复现。

验收：至少三个入口或说明为何只有一个；六类前置中覆盖四类；四层完成都有证据；包含无权限、离线和过期状态；至少一个第二账号验证；比较一个软件和一个非软件替代；所有结论标注事实、推断或假设。
