# 用户目标、任务、场景、入口、路径与完成标准

## 是什么

- **用户目标**：用户希望改变的现实状态，如“确认本月支出是否超预算”。目标不是界面动作。
- **任务**：为达到目标需要完成的一组动作，如筛选本月账单、查看分类汇总。
- **场景**：任务发生时的设备、时间、位置、权限、网络、已有知识和约束。
- **入口**：用户开始任务的位置，包括站内导航、搜索结果、通知、深链和外部分享。
- **路径**：从入口到结果的页面、操作、判断与系统状态序列。
- **完成标准**：可观察、可判断的成功条件，同时包括业务结果和用户确认结果。

## 为什么需要

只描述页面会遗漏用户为何进入、从哪里进入、何时算结束。目标和完成标准决定设计范围；场景决定响应式、权限、离线等约束；入口和路径决定信息是否可发现、流程是否连续。统一这些概念还能避免把业务提交成功、界面操作结束和用户目标达成混为一谈。

## 关键特性或原则

- 目标应描述现实结果，脱离当前产品界面仍然成立。
- 任务必须有明确主体、对象、触发、前置条件和终止条件。
- 场景只记录会改变设计决策的条件，不写与任务无关的人物背景。
- 入口可能绕过首页，深链、通知和恢复入口都要覆盖。
- 路径同时包含用户动作、系统响应和用户判断，不只是页面序列。
- 完成标准必须能通过数据状态、界面反馈或后续动作验证。

## 实际怎么使用

1. 用“用户需要……以便……”写目标，删除按钮名和页面名。
2. 写明主体、对象、触发条件、前置条件和限制。
3. 枚举直接入口、导航入口、通知入口、恢复入口和无权限入口。
4. 将路径写成“用户动作 → 系统响应 → 用户判断”，标出分支、失败和恢复。
5. 用可验证结果定义完成：数据已保存、系统状态正确、用户看到确认且下一步明确。

示例：目标是“在截止前补交发票使报销可继续”，入口可能是待办通知或报销详情；完成标准是发票上传成功、系统校验通过、报销状态变为待审批且用户得到确认。

## 常见错误与边界

- 把“打开页面”“点击按钮”当目标。
- 默认用户总从首页开始。
- 只画主路径，不写前置条件和中断恢复。
- 用业务提交成功代替用户确认成功。
- 将虚构人物背景当作场景证据。

## 补充知识

任务分析可继续拆为层级任务分析；用户流程通常覆盖多页面和分支，任务流程聚焦单一目标下的步骤。两者都应以完成标准结束。落地时可为每条路径增加证据编号、状态清单和复测结果，使后续改版能够追踪目标是否真正达成。
