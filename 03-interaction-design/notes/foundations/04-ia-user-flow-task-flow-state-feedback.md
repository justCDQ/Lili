# 信息架构、用户流程、任务流程、状态与反馈

## 是什么

- **信息架构（IA）**：内容与功能的分类、命名、层级、导航和检索结构。
- **用户流程**：用户为达到目标穿越入口、页面、决策、分支和出口的整体路径。
- **任务流程**：在约定场景下完成一个任务的具体动作序列。
- **状态**：系统、对象、页面或控件在某时刻的数据与可操作条件。
- **反馈**：系统对用户动作或状态变化给出的可感知响应。

## 为什么需要

IA 决定功能在哪里；用户流程决定如何到达；任务流程说明如何操作；状态定义每一步可能发生什么；反馈让用户理解刚才发生了什么。任何流程节点都应至少写出触发条件、用户动作、系统响应、结果状态和下一步。

统一这一概念可以避免团队使用同一词语指代不同对象，也能防止只讨论页面外观而遗漏任务、状态、反馈和完成条件。它为流程图、原型、规范和工程验收提供共同检查单位。

## 关键特性或原则

判断这一知识点时，应明确对象、触发条件、可见状态、用户可执行动作和完成标准。不同平台或任务风险会改变具体方案，不能只根据单张截图或单一控件名称下结论。

## 实际怎么使用

### 实际方法

1. 建对象清单和内容分类，统一术语。
2. 写目标、入口、前置条件与完成标准。
3. 画主流程，再补选择分支、异常、中断和恢复。
4. 给每个节点标状态：初始、加载、成功、失败、空、权限、离线、冲突。
5. 给关键状态定义反馈渠道、出现时机、持续时间和用户可执行下一步。

示例：上传文件不是“选择文件 → 完成”，而是选择、格式预检、上传进度、服务端处理、成功；还包括超限、断网、取消、重试、重复文件和部分成功。

## 常见错误与边界

### 检查与边界

- IA 不等于站点地图；站点地图只是其一种表达。
- 用户流程不是页面截图串联，应包含判断和状态。
- 状态名称必须能映射到数据或条件。
- 反馈应靠近动作和结果；Toast 不适合承载必须处理的错误。

## 补充知识

该概念应落实为可检查产物，而不只保留术语定义。至少记录任务与场景、对象或组件、状态和反馈、异常与恢复、键盘与响应式要求；涉及实现时，还要标出数据来源、权限和无法由静态稿证明的行为。
