# 导航、表单、列表与详情

导航、表单、列表和详情是任务型产品最常见的四类结构。它们分别解决“去哪里”“输入或修改什么”“有哪些对象”“这个对象是什么及能做什么”。一个界面可以同时包含四者，但每个区域应有明确职责。

## 概念与边界

| 结构 | 主要对象 | 用户问题 | 典型结果 |
| --- | --- | --- | --- |
| 导航 | 目的地与位置关系 | 我在哪里，还能去哪里？ | 进入另一个内容上下文 |
| 表单 | 字段、约束与提交 | 需要提供什么，输入是否有效？ | 创建或修改数据 |
| 列表 | 同类对象集合 | 有哪些，怎样找到和比较？ | 定位、选择或批量处理对象 |
| 详情 | 单个对象及其关系 | 它当前是什么状态，能做什么？ | 理解或操作一个对象 |

筛选控件改变当前集合，不等于导航；Tab 在同一上下文中切换互斥面板，不应用来表示任意站点层级；表格强调行列关系，卡片和普通列表强调逐项浏览，三者不能只按视觉偏好互换。

## 组合机制

```mermaid
flowchart LR
    N["导航：选择内容范围"] --> L["列表：定位对象"]
    L --> D["详情：理解对象状态"]
    D --> F["表单：创建或修改"]
    F --> V["校验与提交"]
    V -->|成功| D
    V -->|失败| F
    D --> L
```

这不是固定页面顺序。移动端可能以多个页面呈现，宽屏管理工具可能同时显示列表与详情。无论布局如何变化，对象标识、筛选条件、选中项和编辑状态都应保持一致。

## 导航设计

### 导航层级

- **全局导航**：产品主要区域，如项目、报表、设置。
- **局部导航**：当前区域内的同级目的地。
- **上下文导航**：与当前对象直接相关的目的地，如父项目、关联工单。
- **面包屑**：表达当前页面在层级结构中的祖先路径，不代替全局导航。
- **页内导航**：跳转当前长页面的标题或区域，不改变内容对象。

导航项应使用目的地名称；按钮应使用动作名称。点击“项目”进入项目集合属于导航，点击“新建项目”触发创建动作属于按钮。

### 必须定义的状态

- 当前项与非当前项在视觉和程序语义上可区分；仅靠颜色不够。
- 目的地不可用时，说明是隐藏、禁用还是允许进入后解释权限。
- 窄屏折叠后，菜单的打开状态、焦点与关闭方式明确。
- 深链打开时，页面标题、导航当前项和面包屑保持一致。
- 键盘焦点按文档顺序移动，提供跳过重复导航进入主内容的机制。

## 表单设计

### 字段结构

每个字段至少包含可见标签、输入控件、必要说明、必填状态和错误位置。相关选择可用 `fieldset` 与 `legend` 分组。占位符在输入后会消失，不能代替标签。

字段顺序应符合用户准备信息和完成任务的顺序。只收集完成当前事务必要的数据；额外字段会增加理解、输入、校验和隐私成本。

### 校验时机

| 时机 | 适合 | 风险 |
| --- | --- | --- |
| 输入过程中 | 字符计数、明确格式约束 | 过早报错会打断尚未完成的输入 |
| 离开字段后 | 可独立判断的字段 | 用户可能还会返回修改关联字段 |
| 提交后 | 跨字段规则、服务端权威校验 | 若只在顶部报错，用户难以定位 |

提交失败时保留合法输入，并同时提供错误摘要和字段附近的具体错误。错误消息要说明问题及修正方法，如“截止日期必须晚于开始日期”，而不是“输入无效”。客户端校验改善反馈速度，服务端仍需执行权威校验。

### 键盘与状态

- 使用原生控件和语义标签，保持自然 Tab 顺序，避免正 `tabindex`。
- Enter 是否提交取决于控件和表单语义；多行文本中的 Enter 应换行。
- 提交中阻止重复写入，并显示可感知的等待状态。
- 超时、断网或服务端失败后保留输入，说明是否已保存。
- 成功结果必须明确，必要时通过状态消息让辅助技术获知动态变化。

## 列表设计

### 先定义比较任务

列表项展示什么，取决于用户据以识别、比较和决策的属性。先写出：

- 主标识，如项目名或订单号；
- 关键状态，如逾期、草稿、已退款；
- 用于比较的数值或日期；
- 常用行级操作与批量操作；
- 默认排序及其业务含义。

关系明确、需要跨行比较时使用语义表格；每项内容结构较松、需要逐项浏览时使用列表；卡片适合多种内容类型与较强视觉预览，但会降低密集比较效率。

### 集合状态

- 初始加载与后续分页加载应可区分。
- “集合本来为空”和“筛选后无结果”提供不同下一步。
- 筛选、排序、搜索、分页和选中项的组合规则明确。
- 批量选择跨页是否保留，选择范围必须可见。
- 数据更新、删除或权限变化后，列表位置和焦点有合理落点。
- 虚拟滚动不能破坏键盘访问、屏幕阅读器集合信息或浏览器查找等关键能力。

## 详情设计

详情页应先呈现对象身份、当前状态、主要操作和完成当前任务所需的关键属性。次要元数据、历史记录和关联对象按用户任务分组，而不是按数据库字段顺序罗列。

主要操作应随对象状态变化。例如“已归档项目”不再显示“发布”为主要操作；没有编辑权限时，界面应避免展示最终必定失败的操作，或在操作前明确权限申请路径。

需要覆盖：加载、对象不存在、已删除、无权限、内容过期、并发更新、局部模块失败和操作成功。删除当前对象后，应进入仍然有效的父级或列表，并说明结果。

## 完整案例：项目管理后台

### 任务

项目管理员要找到逾期项目、确认负责人并调整截止日期。

### 结构设计

1. 全局导航当前项为“项目”，页面主标题为“项目”。
2. 列表默认按最近更新排序，提供“逾期”筛选；筛选结果数通过可感知状态消息更新。
3. 每行展示项目名、负责人、截止日期和状态。截止日期使用可比较的统一格式。
4. 选择项目进入详情，URL 包含项目 ID；浏览器返回后恢复筛选和滚动位置。
5. 详情首部显示项目名、逾期状态、负责人和“修改截止日期”。
6. 编辑表单解释允许的日期范围；提交时执行客户端格式校验和服务端业务校验。
7. 成功后详情显示新日期并宣布更新结果；失败时保留输入，错误关联到字段。

### 边界状态

| 条件 | 界面行为 |
| --- | --- |
| 没有任何项目 | 解释集合为空，并在有权限时提供新建入口 |
| “逾期”筛选无结果 | 保留筛选条件，提供清除筛选 |
| 项目被他人删除 | 详情显示对象不存在，返回项目列表 |
| 截止日期被他人更新 | 提示版本冲突，展示最新值，让用户比较后重试 |
| 用户失去编辑权限 | 保持详情可读，移除或禁用编辑并说明申请路径 |

## 可执行设计步骤

1. 建立对象、对象状态、用户任务和权限清单。
2. 划分全局、局部和上下文导航，验证每个目的地名称。
3. 为集合确定识别与比较属性，再选择列表、表格或卡片。
4. 定义筛选、排序、分页、选择和 URL 状态的组合规则。
5. 为详情排序身份、状态、主要操作、关键属性和历史信息。
6. 按用户准备信息的顺序组织表单，写出每个字段和跨字段规则。
7. 补齐四类结构的加载、空、失败、权限、过期和并发状态。
8. 在窄屏、键盘、屏幕阅读器和真实长度数据下验证完整任务。

## 常见错误

- 导航层级、URL、标题和当前项不一致。
- 用 Tab 模拟跨层级导航，导致返回与深链行为混乱。
- 只用 placeholder 作为表单标签，输入后失去字段含义。
- 客户端校验通过就假设提交一定成功。
- 列表展示很多字段，却没有支持真实比较任务。
- 筛选无结果与集合为空使用相同文案和操作。
- 详情把所有操作都设为主要按钮，忽略对象状态与风险。
- 删除对象后留在失效详情，没有下一步与焦点管理。

## 验证步骤

1. 直接打开三个层级不同的深链，检查标题、当前导航和面包屑。
2. 只用键盘完成筛选、打开详情、编辑、处理错误和返回列表。
3. 用屏幕阅读器核对地标、标题、表单标签、错误、列表或表格语义以及动态结果数。
4. 准备零条、一条、数百条、超长名称和缺失字段数据，检查布局与信息优先级。
5. 制造加载失败、提交失败、对象删除、权限变化和并发更新。
6. 在窄屏完成同一任务，确认信息没有因布局变化而丢失，操作顺序仍合理。

## 练习与完成标准

设计一个“工单列表 → 工单详情 → 修改优先级”的完整流程。

完成时应满足：

- 标出全局、局部和上下文导航；
- 说明列表项的主标识、比较属性、默认排序和筛选规则；
- 详情按对象身份、状态、操作和属性组织；
- 表单具有标签、说明、校验时机和错误恢复；
- 覆盖空、筛选无结果、加载失败、无权限和并发更新；
- 写出键盘顺序、焦点落点和动态状态的辅助技术通知；
- 测试者能够依据完成标准判定任务是否成功。
