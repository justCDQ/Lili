# 评估设计收益、成本、风险与工程约束

设计方案评估需要同时说明它改善什么任务、需要哪些一次性与持续成本、引入什么用户和系统风险，以及哪些工程条件决定方案能否可靠实现。评估不是用一个加权总分自动选方案，而是把决策前提、证据、不可接受风险和验证方式透明化。

## 四个维度

| 维度 | 定义 | 典型内容 |
| --- | --- | --- |
| 收益 | 方案在特定条件下带来的可观察改善 | 成功率、错误恢复、等待理解、可访问性、业务完成 |
| 成本 | 建设、迁移、运行、支持和维护所需资源 | 设计、开发、测试、基础设施、内容、培训、运维 |
| 风险 | 不确定事件发生后造成的损害 | 误操作、数据丢失、隐私、安全、法律、锁定与回滚失败 |
| 工程约束 | 当前系统、平台和组织限制的可行范围 | 数据模型、接口、异步、性能、权限、兼容与发布方式 |

工程约束不是“做不了”的同义词。记录约束的来源、强度、持续时间和改变成本，区分确定不变量、当前实现限制和团队暂时缺少能力。

## 评估流程

```mermaid
flowchart LR
    P["问题事实与目标"] --> A["形成多个候选方案"]
    A --> B["定义收益指标和基线"]
    B --> C["拆分全生命周期成本"]
    C --> R["识别风险与控制措施"]
    R --> E["验证工程可行性"]
    E --> D["选择、试点或停止"]
    D --> V["上线后验证与回滚"]
```

至少保留“保持现状”作为基线。没有候选方案比较时，很难判断复杂设计是否必要。

## 收益怎样定义

### 用户任务收益

- 更多用户正确完成任务；
- 减少错误、重复输入与恢复时间；
- 状态、权限和结果更易理解；
- 键盘、辅助技术、窄屏和低带宽可完成；
- 用户能撤销、恢复或重新确认权威结果。

### 产品与业务收益

- 合格任务完成、采用或留存变化；
- 支持请求、人工处理和重复记录减少；
- 合规、安全和审计能力提高；
- 数据质量和运营可观测性改善。

### 工程收益

- 状态模型统一、重复实现减少；
- 请求幂等、故障隔离和恢复能力提高；
- 测试可重复、日志可追踪；
- 组件和模式可复用。

收益应写基线、目标、采样点和时间范围。“体验更好”不可验收。相关指标变化不自动证明设计是原因，需要实验、分阶段发布或其他证据。

## 成本怎样拆分

### 一次性建设成本

- 问题验证、设计、内容和无障碍评审；
- 前端、服务端、数据迁移和第三方集成；
- 自动化、手工、兼容、安全和性能测试；
- 发布、监控、文档和支持准备。

### 持续成本

- 基础设施、存储、队列、通知和第三方费用；
- 日志、监控、告警和人工处理；
- 多平台、多语言与辅助技术维护；
- 数据保留、删除、审计与合规响应；
- 组件升级、浏览器变化和技术债。

### 迁移与机会成本

- 旧数据转换和双写；
- 老用户学习与支持；
- 版本兼容、灰度和回滚；
- 延迟其他工作带来的机会成本。

只估前端页面工作会系统性低估异步任务、权限、数据一致性和运营成本。

## 风险怎样记录

### 风险条目

```text
触发条件
受影响对象与人群
可能后果
发生可能性及证据
影响严重度
预防、检测、响应与恢复控制
残余风险
责任人与复核时间
```

### 风险分类

- **任务风险**：错误完成、无法恢复、误导状态。
- **数据风险**：丢失、重复、不一致、错误覆盖。
- **权限与安全风险**：越权、泄露、重放、注入。
- **隐私风险**：过度收集、错误共享、保留过长。
- **可访问性风险**：键盘陷阱、状态不可感知、缩放阻塞。
- **运营风险**：队列堆积、第三方故障、支持无证据可查。
- **迁移风险**：新旧状态不一致、无法回滚。

高风险项应设置硬性通过条件，不应靠总体加权分被低风险收益抵消。

## 工程约束怎样验证

### 数据与状态

- 权威数据在哪里，版本怎样控制？
- 操作是原子、部分成功还是最终一致？
- 幂等键、唯一约束和事务边界是什么？
- 刷新、离线、跨设备怎样恢复？

### 性能与规模

- 数据量、并发、响应时间分布与峰值是什么？
- 是否需要分页、流式、后台任务或增量更新？
- 大列表、长内容和弱网行为怎样？

### 平台与兼容

- 浏览器、原生系统、输入方式和辅助技术支持范围；
- HTML 原生能力与自定义组件成本；
- 深链、历史、通知、文件和后台任务能力。

### 发布与运维

- 能否按角色、组织或比例灰度？
- 新旧数据能否共存，如何回滚？
- 有哪些日志、指标、追踪和告警？
- 支持人员怎样定位单个任务结果？

## 完整案例：把同步 CSV 导入改为异步任务

### 现状与具体证据

```text
任务：管理员导入 10,000 名成员
当前：浏览器同步等待，网关超时 60 秒
数据：最近 30 天 1,200 次导入；P95 处理 94 秒
失败：18% 页面超时，其中部分后台实际写入
支持：73 个“是否导入成功/重复导入”工单
约束：写入允许逐行部分成功；邮件通知按成员异步发送
```

### 候选方案

- **基线**：保持同步等待，优化单次处理性能。
- **方案 A**：上传后创建持久异步任务，任务详情显示处理与逐行结果。
- **方案 B**：在浏览器分批上传与处理，每批显示结果。

### 收益比较

| 收益 | 基线 | A | B |
| --- | --- | --- | --- |
| 超过网关时限 | 仍有风险 | 页面不需持续连接 | 单批可控制，但总流程依赖浏览器 |
| 离开与恢复 | 弱 | 任务 ID 可恢复 | 浏览器关闭后恢复复杂 |
| 逐行结果 | 可增加 | 持久结果与报告 | 可逐批显示 |
| 重复提交 | 需另做幂等 | 任务和行级幂等 | 批次与行级幂等更复杂 |
| 运维定位 | 现有日志分散 | 任务 ID 贯穿追踪 | 多批次需要聚合 |

### 成本与工程约束

方案 A 需要任务表、对象存储、队列、工作进程、进度聚合、幂等约束、结果保留、通知、任务详情和告警。持续成本包括队列运行、文件存储、失败重试和支持工具。

方案 B 前端状态更复杂，浏览器关闭、网络切换和多标签页恢复困难；仍需要服务端批次幂等。它不天然比 A 成本低。

### 风险与控制

| 风险 | 控制 | 验证 |
| --- | --- | --- |
| 同一文件重复创建成员 | 导入意图 ID + 成员唯一约束 | 重放相同请求 |
| 任务卡死 | 心跳、超时、死信和告警 | 停止工作进程 |
| 结果与成员数不一致 | 事务、逐行状态和对账任务 | 固定 CSV 对照数据库 |
| 错误报告泄露邮箱 | 权限、短期签名下载、保留期 | 角色与过期测试 |
| 状态不可感知 | 可见阶段、状态消息和持久详情 | 键盘与读屏测试 |
| 无法回滚 | 新旧入口灰度；任务数据独立；开关回退 | 演练回滚 |

### 决策输出

选择方案 A 进行 5% 管理员灰度，原因是它能处理页面超时、恢复与运维追踪，且数据模型可支持部分成功。否决条件：幂等与权限测试未通过、任务状态无法对账、回滚演练失败。

目标指标：页面超时不再导致未知结果；重复成员创建为 0；支持人员可用任务 ID 定位；键盘和屏幕阅读器能获取处理与结果。性能目标由当前基线和容量测试确定，不在设计文档中宣称通用阈值。

### 失败分支与回滚

队列故障时，新任务进入“等待处理”，不显示失败成功；超过内部阈值触发告警并允许安全暂停新上传。若灰度发现任务状态不一致，关闭新入口，已有任务继续按版本处理；不能删除任务记录或让用户结果消失。

## 可执行评估步骤

1. 写清问题事实、目标、基线与不可接受结果。
2. 提出基线加至少两个可行方案。
3. 为收益定义任务、业务和工程指标及采样点。
4. 拆分建设、运行、迁移、支持和机会成本。
5. 建立风险条目，定义预防、检测、响应和残余风险。
6. 与工程共同验证数据、状态、性能、兼容、发布和回滚。
7. 通过原型、技术试验和固定数据验证关键未知。
8. 记录选择、否决条件、灰度与上线后复核。

## 常见错误与修正

- 用一个主观分数合并全部维度：保留证据、范围和否决条件。
- 只写用户收益，不估数据、运维和支持成本。
- 把当前技术实现称为永久约束：说明来源与改变成本。
- 只评估成功路径，不估故障、迁移和回滚。
- 用前端控制承担权限、幂等和业务不变量。
- 指标没有基线、采样点和时间范围。
- 方案选定后停止验证，不检查残余风险和生产表现。

## 练习与完成标准

评估“为文档编辑增加自动保存”的基线、方案与风险。

完成时应满足：

- 定义现有保存失败、数据丢失和用户任务基线；
- 比较显式保存、定时自动保存和操作级增量保存；
- 评估前端、接口、版本、冲突、存储、日志与迁移成本；
- 覆盖离线、跨标签页、权限变化、撤销和服务端失败；
- 明确本地已保存与云端已同步的状态和辅助技术反馈；
- 设定不可接受的数据丢失与覆盖条件；
- 有灰度、观测、回滚和固定文档对账测试。
