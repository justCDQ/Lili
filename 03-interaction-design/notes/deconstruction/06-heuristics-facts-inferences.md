# 用启发式原则分析事实与推断

启发式分析使用一组经验性原则检查界面中可能影响任务完成的问题。它适合发现问题线索和组织检查，但不能替代真实行为证据、可访问性标准、业务规则或工程测试。可靠拆解必须把观察事实、原则判断、原因推断和修复建议分开。

## 四类记录

| 类型 | 定义 | 示例 |
| --- | --- | --- |
| 事实 | 可重复观察或由权威规则确认 | 保存请求期间按钮仍可点击，产生两个 POST |
| 启发式判断 | 事实与某条经验原则的关系 | 系统状态反馈不足；错误预防不足 |
| 推断 | 对原因或影响机制的可证伪解释 | 用户可能因无反馈而重复提交 |
| 建议 | 基于证据的候选修复 | 立即显示忙碌状态；服务端增加幂等保证 |

同一事实可以违反多个启发式；同一启发式也能解释多个问题。不要为了填满十条原则重复记录同一个现象。

## 分析链

```mermaid
flowchart LR
    C["任务、角色与状态"] --> F["可复现事实"]
    F --> H["启发式分类"]
    H --> I["机制与影响推断"]
    I --> E["补充证据或反证"]
    E --> R["修复候选"]
    R --> V["完整任务复测"]
```

启发式不是自然定律。它们帮助提出检查问题，不能单独证明严重度、原因或最佳方案。

## 十类常用启发式怎样操作化

### 系统状态可见

检查动作是否被接收、当前阶段、进度、结果和下一步。异步任务要区分已提交、处理中和最终完成；状态变化要视觉可见并可被辅助技术获知。

可观察问题：无加载状态、旧数据看似最新、上传完成与处理完成混淆、短暂结果不可再次查看。

### 系统与现实匹配

检查对象、动作、顺序和单位是否符合用户任务及业务事实，而不是内部实现。使用“归档”却永久删除，或用内部状态码作为主要文案，都会破坏匹配。

### 用户控制与自由

检查返回、取消、退出、撤销和恢复。取消要说明是否停止真实任务；撤销要处理并发和副作用。

### 一致性与标准

检查相同功能名称、结果、位置和键盘模式，以及平台原生语义。不同状态需要不同行为时，差异应明确表达。

### 错误预防

检查规则是否在操作前可见、危险动作是否可复核、写操作是否幂等、服务端是否保证不变量。前端禁用按钮不是完整预防。

### 识别优于回忆

检查是否要求记住前页信息、内部 ID、隐藏规则或不常用格式。提供历史、建议、对象上下文和可见选择，但不要暴露敏感信息或展示无关选项。

### 灵活与高效

检查初次用户路径与熟练用户加速器：搜索、最近使用、批量、快捷键和自动化。加速器必须有可见等价路径，快捷键需要避免冲突。

### 简洁与相关

检查当前决策所需信息是否突出，无关内容是否干扰。简洁不等于删除风险、规则、状态和恢复信息。

### 识别、诊断与恢复错误

错误应指出对象和字段、发生了什么、哪些结果已生效、怎样修正。文本描述不能只靠颜色；输入应尽量保留。

### 帮助与文档

检查界面是否能独立完成常见任务，以及复杂规则是否有可搜索、上下文相关的帮助。帮助不能代替修复错误流程，也不能成为唯一的权限与风险说明。

## 启发式与规范的边界

| 内容 | 性质 | 处理方式 |
| --- | --- | --- |
| Nielsen 启发式 | 广泛适用的经验原则 | 用于发现与分类潜在问题 |
| WCAG 成功标准 | Web 可访问性规范要求 | 按版本、级别与适用条件评估合规 |
| WAI-ARIA APG | 信息性组件指导 | 用于语义与键盘实现参考，需实际测试 |
| 设计系统 | 特定组织的组件与模式约定 | 按适用服务和版本使用 |
| 业务规则 | 产品与组织定义的确定性约束 | 由受控系统执行并验证 |

“违反启发式”不等于“违反 WCAG”；通过启发式检查也不等于可访问或可用。

## 严重度评估

可使用以下维度分别记录，不急于合成一个分数：

- **任务影响**：轻微延迟、可恢复错误、无法完成或错误完成。
- **频率**：在明确样本、版本和时间范围内出现多少。
- **范围**：影响哪些角色、设备、权限和对象状态。
- **可恢复性**：用户能否自行发现并恢复，是否丢数据。
- **风险**：资金、隐私、安全、法律和不可逆结果。
- **证据强度**：复现、日志、公开反馈或仅为推断。

严重度不是优先级。优先级还受修复成本、依赖、法规和发布时间影响。

## 完整案例：报销提交出现重复记录

### 上下文与具体事实

```text
角色：报销申请人
对象：报销草稿 R-DRAFT-88
设备：移动 Web，弱网 600 ms RTT
输入：金额 ¥128，1 张发票
完成标准：只创建一个报销记录，并显示编号与待审批状态
```

复现步骤：

1. 填写草稿并点击“提交”。
2. 8 秒内按钮无文字、状态或禁用变化。
3. 第 3 秒再次点击，第 6 秒第三次点击。
4. 网络面板出现三个 POST，请求体相同且没有幂等键。
5. 最终出现三个报销编号，页面只显示最后一个“提交成功”Toast。

### 事实、原则与推断分离

| 编号 | 类型 | 内容 |
| --- | --- | --- |
| F-01 | 事实 | 首次点击后 8 秒无可见或可感知状态变化 |
| F-02 | 事实 | 三次点击产生三个 POST 和三个对象 |
| F-03 | 事实 | 结果只显示最后一个编号，其他记录需从列表发现 |
| H-01 | 启发式 | 系统状态可见、错误预防和结果恢复存在问题 |
| I-01 | 推断 | 无反馈提高重复点击可能性；需要行为数据或观察补证 |
| I-02 | 推断 | 服务端没有幂等约束；由请求结果支持，仍需检查实现规则 |

### 修复方案

1. 点击后立即把按钮进入明确忙碌状态，并阻止界面重复发起同一意图。
2. 客户端为一次提交意图生成幂等键，服务端以用户、草稿和键保证重复请求返回同一结果。
3. 结果使用持久页面，显示报销编号、金额、状态和下一步。
4. 超时结果未知时，按幂等键查询，而不是生成新意图。
5. 已生成的重复记录需要安全的清理或撤销路径，并保留审计。

### 可观察输出

```text
提交中：报销草稿 R-DRAFT-88
提交结果：报销 R-2041 已创建
状态：待审批
重复请求结果：仍返回 R-2041，不创建新记录
```

### 失败分支

- 服务端创建成功但响应丢失：客户端用相同键重试并取得 R-2041。
- 校验失败：保留草稿，定位金额或发票错误，不创建对象。
- 权限被撤销：服务端拒绝，页面说明未提交且保留本地输入。
- 状态消息只视觉显示：屏幕阅读器无法获知，需使用适当程序语义。

### 验证

1. 连续点击、网络重放和多标签页发送相同幂等键，只产生一个记录。
2. 在 200 ms、8 秒和超时三种延迟下检查忙碌与未知结果。
3. 仅用键盘提交，焦点不丢失且结果可感知。
4. 刷新与重新登录后，结果页和列表显示同一记录。
5. 对照 WCAG 状态消息与错误要求，不把启发式结论当作合规证明。

```markdown
- 预期结果：
- 实际事实：
- 证据编号：
- 相关启发式：
- 相关规范或业务规则：
- 原因推断：
- 反证条件/未知项：
- 用户与业务影响：
- 修复候选：
- 验证步骤与通过标准：
```

## 多人检查与去重

不同检查者会发现不同问题。合并结果时按“同一触发条件、同一机制、同一影响”去重；一个问题可标多个原则，但保留一个事实记录。分歧应回到可复现证据，不用原则名称投票。

个人学习没有多人条件时，可在不同时间、角色和输入方式下重复检查，并把首次结论隐藏后重新执行，降低记忆影响，但仍不能声称覆盖了真实用户行为。

## 常见错误与修正

- 为每条启发式强行找问题：只记录有证据的现象。
- 把原则名称当事实：“不一致”要说明哪里、怎样不同。
- 从一个截图推断用户一定失败：补任务观察和交互证据。
- 给出修复方案却不写机制和验证标准。
- 用启发式代替 WCAG、权限、安全或性能测试。
- 将多个相同根因拆成十条重复问题。
- 只报告数量和严重度，不保存复现与证据。
- 认为没有发现问题等于界面可用。

## 可执行分析步骤

1. 固定角色、任务、版本、设备、数据与完成标准。
2. 先完整执行任务，记录动作、状态和证据。
3. 第二遍按启发式逐项检查，但不强行填充。
4. 把记录拆为事实、原则判断、推断和建议。
5. 对涉及键盘、状态、错误等问题对照 WCAG 与平台规范。
6. 用日志、请求、公开反馈或故障注入验证原因推断。
7. 评估任务影响、范围、恢复、风险和证据强度。
8. 去重后写修复候选，并在完整任务与边界状态复测。

## 练习与完成标准

对一个“账户安全设置”流程进行启发式分析。

完成时应满足：

- 固定普通用户和管理员两个角色及明确任务；
- 至少记录 8 条可复现事实，不要求覆盖每条启发式；
- 每条事实与启发式判断、原因推断分开；
- 涉及认证、错误、焦点和状态时引用适用标准或规则；
- 严重度按任务影响、范围、恢复和风险说明；
- 修复方案覆盖主路径、失败、键盘和权限；
- 复测报告能说明问题是否真实消失，而不是只检查画面变化。
