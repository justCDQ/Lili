# 记录产品拆解上下文

产品拆解的第一步不是评价界面，而是记录能够复现观察的上下文：产品与版本、用户角色、任务目标、入口、设备、数据状态、权限、时间和证据。缺少上下文，同一截图可能对应完全不同的规则，后续推断无法验证。

## 什么是拆解上下文

拆解上下文是一组限定“在什么条件下看见了什么”的事实。它与用户画像、结论和改版方案不同。

| 内容 | 回答的问题 | 示例 |
| --- | --- | --- |
| 产品与版本 | 观察的是哪个实现？ | Web 版，构建号 2026.07.15 |
| 账户与角色 | 谁在操作，拥有什么权限？ | 项目编辑者，不是管理员 |
| 任务与目标 | 为什么进入，何时算完成？ | 修正逾期任务的截止日期 |
| 入口与前置状态 | 从哪里开始，已有数据怎样？ | 邮件深链；任务状态为“进行中” |
| 环境 | 什么设备、视口、输入和网络？ | Chrome、1440×900、键盘、弱网模拟 |
| 时间 | 何时观察，是否有时效条件？ | 2026-07-17 10:30，截止前一天 |
| 证据 | 什么可以被其他人重新检查？ | URL、截图、录像、DOM、网络记录、帮助文档 |

上下文记录不要求包含个人身份信息。用于分析的账户、订单、邮件和日志应去标识化，凭据、令牌、内部地址与敏感业务数据不得进入公开仓库。

## 事实、推断与问题

记录时将三类内容分开：

- **事实**：可以直接观察或由权威资料确认，例如“按钮在请求期间仍可点击”。
- **推断**：对机制或原因的解释，例如“可能缺少重复提交保护”。
- **待验证问题**：需要进一步证据的问题，例如“服务端是否使用幂等键”。

```mermaid
flowchart LR
    C["上下文条件"] --> O["观察事实"]
    O --> H["形成可证伪推断"]
    H --> Q["列出待验证问题"]
    Q --> E["补充证据"]
    E --> H
```

“页面不直观”既不是精确事实，也不是可证伪机制。应改写为：“编辑者从通知深链进入任务详情后，在 90 秒内三次打开‘更多’菜单，未找到‘修改截止日期’；操作最终未完成。”

## 必须记录的字段

### 产品标识

- 产品名称、平台和部署环境；
- 页面 URL 或稳定对象标识，公开记录时去除敏感参数；
- 版本号、构建号、提交或观察日期；
- 实验分组、功能开关和地区差异；
- 语言、时区和主题。

只写“最新版”不可复现。无法取得版本号时，至少记录精确时间、平台和可见版本特征。

### 用户与权限

- 当前角色、组织与对象关系；
- 查看、创建、编辑、删除、审批等能力；
- 登录、未登录、会话过期与多因素认证状态；
- 是否使用辅助技术、键盘、语音或放大。

权限是动态状态。记录“按钮隐藏”时要同时说明账户角色和对象状态，不能据此推断所有用户都没有该功能。

### 任务

- 现实目标与具体任务；
- 触发事件和截止条件；
- 输入材料、已有知识和外部渠道；
- 可观察完成标准；
- 错误成本和是否可恢复。

拆解不要默认从产品首页开始。通知、搜索结果、分享链接、浏览器历史和失败恢复都可能是主要入口。

### 设备与环境

- 操作系统、浏览器或原生应用版本；
- 视口 CSS 像素、缩放和屏幕方向；
- 鼠标、触控、键盘或其他输入；
- 网络、离线、延迟和带宽条件；
- 屏幕阅读器、放大、高对比或减少动态设置。

物理屏幕分辨率不能代替视口尺寸。响应式行为通常由 CSS 视口与内容决定。

### 数据与系统状态

- 对象 ID、生命周期状态和所有者；
- 集合数量、数据长度、空值和特殊字符；
- 当前筛选、排序、分页和选中项；
- 请求、缓存、离线队列和同步状态；
- 与任务相关的并发修改或过期数据。

使用合成或脱敏测试数据时，保留影响设计的分布：同名对象、长名称、零条、数百条、不同权限和错误项。

## 证据类型及能力边界

| 证据 | 能支持 | 不能单独支持 |
| --- | --- | --- |
| 截图 | 某一时刻可见内容、布局与视觉状态 | 键盘、动效、请求、前后状态和原因 |
| 录屏 | 操作顺序、等待、可见反馈 | DOM 语义、服务端权威状态 |
| DOM/无障碍树 | 结构、名称、角色、状态 | 业务规则是否正确 |
| 网络记录 | 请求、响应、耗时和错误 | 用户是否理解结果 |
| 日志与追踪 | 系统处理与失败路径 | 用户端是否看见反馈 |
| 帮助与规则文档 | 预期规则和术语 | 当前实现一定遵守规则 |
| 公开反馈 | 真实问题线索 | 发生率、总体代表性和根因 |

截图必须记录触发步骤和状态，不能脱离流程作为唯一证据。动态任务至少保存开始、进行、结果和失败的可检查记录。

## 完整案例：从待办通知修改截止日期

### 具体上下文

```text
产品：项目管理 Web 应用
版本：build 2026.07.15
角色：项目编辑者，可编辑任务，不可管理项目
入口：邮件中的任务深链
设备：macOS、Chrome、CSS 视口 1280×720、缩放 200%
输入：任务 T-204，当前截止 2026-07-18，目标改为 2026-07-22
网络：DevTools 模拟 400 ms RTT
完成标准：详情显示新日期；刷新后仍一致；操作记录出现变更
```

### 逐步观察

| 步骤 | 动作 | 可见结果 | 额外证据 |
| --- | --- | --- | --- |
| 1 | 打开邮件深链 | 先进入登录页 | URL 保留任务重定向参数 |
| 2 | 登录 | 进入 T-204 详情，焦点在页面顶部 | 任务标题在当前视口下被固定导航遮挡一部分 |
| 3 | 打开“更多” | 菜单含“修改日期” | 触发按钮可访问名称为“更多操作” |
| 4 | 选择修改 | 对话框打开，当前日期已填入 | 焦点进入日期字段，背景不可操作 |
| 5 | 输入 2026-07-22 并保存 | 按钮显示处理中 1.7 秒 | 一次 PATCH 请求，响应 200 |
| 6 | 对话框关闭 | 详情日期更新，出现短暂成功消息 | 焦点返回“更多操作” |
| 7 | 刷新 | 日期仍为 2026-07-22 | 操作记录出现变更 |

### 输出

事实记录：

- F-01：登录后成功返回原任务，而不是项目首页。
- F-02：200% 缩放下任务标题部分被固定导航遮挡。
- F-03：保存期间有可见忙碌状态，且只产生一次写请求。
- F-04：成功后焦点返回触发菜单按钮。
- F-05：刷新与操作记录确认修改已持久化。

推断与待验证：

- H-01：固定导航可能违反内容在放大条件下可见的目标；需要在其他视口和文本长度复测。
- H-02：成功消息是否被屏幕阅读器宣布未知；需要检查 live region 或实际读屏输出。

### 失败分支

把任务状态改为“已关闭”后重复步骤。保存请求返回 409，界面保留用户输入并显示“任务已关闭，截止日期未修改”，提供刷新详情。此分支需要单独记录对象状态，不能与正常路径截图混在一起。

### 验证

1. 另一位分析者使用记录中的角色、对象和环境重复步骤。
2. 对照请求、刷新结果和操作记录确认权威状态。
3. 在 100% 与 200% 缩放复查标题遮挡。
4. 使用屏幕阅读器确认成功与冲突错误是否被感知。

## 证据命名与仓库维护

建议使用稳定编号而不是在正文写临时文件名：

```text
ctx-task-deadline-20260717.md
e01-entry.png
e02-dialog-open.png
e03-conflict.png
network-redacted.har
```

HAR、日志和截图在提交前必须检查凭据、Cookie、用户数据和内部地址。若无法安全脱敏，只记录必要字段的文本摘要，不上传原文件。

## 常见错误与修正

- 只保存最终截图：补记录入口、动作、等待、结果和失败状态。
- 写“普通用户”：改为具体角色、对象关系和权限。
- 写“桌面端”：补操作系统、浏览器、视口和缩放。
- 把帮助文档当实现事实：分别记录预期规则与实际观察。
- 把单个公开评论当总体结论：作为问题线索，补充范围与其他证据。
- 记录真实邮箱、令牌和订单：使用合成数据并脱敏证据。
- 一边观察一边改版，导致原始状态丢失：先冻结事实记录，再建立方案。

## 验证步骤

1. 检查记录是否能唯一限定产品、版本、角色、入口、设备和对象状态。
2. 让未参与观察的人按步骤复现，标出缺失前置条件。
3. 对每条事实找到对应证据，对每条推断写出反证条件。
4. 检查截图、HAR 和日志是否包含敏感信息。
5. 用主路径、失败路径、键盘和响应式条件各复现一次。
6. 核对完成标准同时包含界面结果与权威数据状态。

## 练习与完成标准

选择一个公开 Web 产品，记录“创建并删除一个测试对象”的拆解上下文。

完成时应满足：

- 有产品、精确时间或版本、角色、权限、入口、设备和视口；
- 使用合成数据，不包含凭据与个人信息；
- 主路径按动作、响应、状态和证据逐步记录；
- 包含加载、成功、删除恢复或失败中的至少两个状态；
- 事实、推断和待验证问题完全分栏；
- 另一位读者能复现并判断完成标准；
- 所有证据已完成隐私和敏感信息检查。
