# 实现关键交互并验证键盘、响应式与状态切换

## 是什么

实现关键交互并验证键盘、响应式与状态切换是一种以真实任务和可复现证据为单位的产品拆解工作。它要求记录观察条件、界面行为和任务结果，并把事实与解释分开。

## 为什么需要

可交互实现用于验证静态稿无法证明的焦点、键盘、滚动、异步、响应式和状态变化。实现范围应覆盖关键风险，不要求复制完整产品。

拆解若缺少这一项，观察就无法稳定对应到具体任务、版本和状态，后续重新设计也无法说明改动解决了什么。把它单独记录，才能让问题、证据、设计决策和复测结果形成可追踪链路。

## 关键特性或原则

### 验收清单

- 焦点始终可见且顺序符合阅读与任务顺序。
- 模态焦点不泄漏，关闭后回到合理对象。
- 颜色不是唯一状态线索；状态消息可被辅助技术感知。
- 横向滚动和固定元素不遮挡内容。
- 状态切换不会导致布局跳动或操作对象丢失。

## 实际怎么使用

### 实现步骤

1. 选择一个主任务和高风险状态，定义完成标准。
2. 优先使用语义 HTML 和平台标准控件；自定义组件对照 WAI-ARIA APG。
3. 为初始、加载、成功、失败、空和权限建立可切换数据夹具。
4. 定义每次打开、关闭、删除、插入和路由后的焦点位置。
5. 用 `Tab`、`Shift+Tab`、方向键、`Enter`、`Space`、`Escape` 完成任务。
6. 在窄屏、宽屏、200% 缩放、长文案和大数据下验证重排、滚动和命中区。
7. 用慢网、断网和错误响应检查取消、重试、防重复和输入保留。

## 常见错误与边界

- 只验证鼠标点击和一种屏幕宽度。
- 给所有元素加 `tabindex` 破坏原生顺序。
- 视觉隐藏背景但读屏仍可访问。
- 原型假数据无法证明真实并发、性能和兼容性，需在实现阶段继续测试。

## 补充知识

建议把本知识点的产出连接到三类记录：原始证据编号、设计决策编号和复测结果。若产品版本、账户权限或平台变化，应新建一次记录，不覆盖旧结论；这样可以进行历史对比，也能识别问题来自设计变化还是运行条件变化。
