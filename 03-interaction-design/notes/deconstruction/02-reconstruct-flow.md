# 还原入口、前置条件、主流程、分支、异常与完成状态

## 是什么

流程还原是以实际操作和可见证据记录任务如何发生，而不是根据界面猜测产品逻辑。结果应包含入口、前置条件、动作、系统响应、判断分支、异常、恢复和完成状态。

## 为什么需要

拆解若缺少这一项，观察就无法稳定对应到具体任务、版本和状态，后续重新设计也无法说明改动解决了什么。把它单独记录，才能让问题、证据、设计决策和复测结果形成可追踪链路。

## 关键特性或原则

### 检查清单

- 浏览器返回、刷新和重新进入是否保持合理状态？
- 中断后从何处恢复？
- 异步处理期间能否离开，结果在哪里查看？
- 分支是否写出互斥条件？
- 部分成功是否被误记为完成？

## 实际怎么使用

### 实际步骤

1. 列出通知、深链、搜索、导航等入口。
2. 写明登录、权限、数据、依赖服务等前置条件。
3. 在真实产品中逐步执行主流程，每步记录“动作—反馈—状态”。
4. 主动改变输入、返回、取消、刷新、重复提交，发现分支。
5. 制造断网、超时、无权限和数据变化，记录异常与恢复。
6. 以可观察结果标记完成，不以最后一次点击结束。

流程节点格式：`编号 / 页面或状态 / 用户看到 / 用户动作 / 系统响应 / 进入条件 / 下一节点 / 证据编号`。

## 常见错误与边界

- 只连接截图，不记录条件和反馈。
- 把推断写成事实；无法观察的后台规则应标“待验证”。
- 只用已有测试数据，遗漏空数据和大数据。
- 对生产产品进行破坏性或越权测试；只能在授权范围内制造异常。

## 补充知识

建议把本知识点的产出连接到三类记录：原始证据编号、设计决策编号和复测结果。若产品版本、账户权限或平台变化，应新建一次记录，不覆盖旧结论；这样可以进行历史对比，也能识别问题来自设计变化还是运行条件变化。
