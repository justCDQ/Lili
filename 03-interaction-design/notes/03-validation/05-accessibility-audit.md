# Web 无障碍审计

Web 无障碍审计依据明确范围与 WCAG 版本，结合自动检查、代码检查、键盘、缩放、视觉条件和辅助技术，判断真实任务是否可感知、可操作、可理解且与用户代理兼容。任何单一工具或“无障碍分数”都不能判定完整合规。

## 标准、指导与测试工具

| 内容 | 性质 | 用法 |
| --- | --- | --- |
| WCAG 2.2 | W3C Recommendation，规范性成功标准 | 定义 Web 内容的可测试要求和级别 |
| WCAG Understanding | 信息性解释 | 理解意图、例子、技术和常见失败 |
| WAI-ARIA 1.2 | 角色、状态与属性规范 | 暴露复杂界面语义，不能替代行为 |
| ARIA APG | 信息性实现指导 | 参考复杂组件键盘与语义模式 |
| 自动检测工具 | 对部分机器可判定规则执行检查 | 快速发现问题，不能覆盖全部标准 |
| 屏幕阅读器等辅助技术 | 实际用户代理组合 | 验证名称、状态、顺序和任务输出 |

审计报告必须写目标 WCAG 版本、级别、技术、页面/流程范围和测试环境。WCAG conformance 适用于完整页面与完整过程等要求，不能只抽一个组件就宣称整个产品符合。

## 审计流程

```mermaid
flowchart LR
    S["确定范围和 WCAG 目标"] --> P["选择页面、状态与过程"]
    P --> A["自动与代码检查"]
    A --> K["键盘、缩放和视觉检查"]
    K --> T["辅助技术任务测试"]
    T --> R["按成功标准记录问题"]
    R --> F["修复与回归"]
```

W3C 建议在开发和重设计过程中持续评估。审计不是上线前一次性活动。

## 确定范围

### 页面与模板

选择首页、导航、搜索、列表、详情、表单、认证、帮助和高风险结果等代表页面。基于模板的页面应包含不同内容长度和数据状态。

### 完整过程

选择注册、购买、提交申请、上传、取消等端到端任务。过程中的第三方支付、身份验证和文档也属于用户完成任务的链路。

### 状态

初始、加载、空、成功、失败、无权限、离线、过期、对话框、菜单、校验错误和动态更新。只测默认状态会遗漏大量 name/role/value 与状态消息问题。

### 环境

- 操作系统、浏览器和版本；
- 视口、缩放、主题和系统设置；
- 键盘与指针；
- 屏幕阅读器与版本；
- 语言与内容方向。

## 快速自动与代码检查

自动工具适合发现：

- 部分缺少可访问名称、表单标签和图像替代文本；
- 一部分对比问题；
- 重复 ID、无效 ARIA 和结构错误；
- 某些标题、地标和语言问题。

自动工具通常不能可靠判断：

- 替代文本是否准确；
- 焦点顺序是否符合任务意义；
- 键盘操作与状态变化是否完整；
- 错误说明是否有帮助；
- 动态消息是否过度或不足；
- 整个任务是否能完成。

修复工具报告时先验证真实 DOM 和用户影响，不按数量机械清零。工具可能误报，也可能因组件尚未打开而漏报。

## 手工检查

### 页面结构

- 页面语言正确；
- 主标题和标题层级表达内容关系；
- header、nav、main、footer 和区域有适当语义；
- 提供跳过重复内容机制；
- DOM 顺序在无 CSS 和键盘导航时仍有意义。

### 键盘

- 所有功能可用键盘操作；
- 无键盘陷阱；
- 焦点顺序保留意义与可操作性；
- 焦点指示可见且不被遮挡；
- 自定义组件遵循其键盘模式；
- 对话框打开、关闭和焦点恢复正确。

### 名称、角色、值与状态

- 控件有可区分名称；
- 可见标签包含在可访问名称中；
- 展开、选中、无效、忙碌和当前状态可程序确定；
- 链接与按钮语义符合实际行为；
- 不使用 ARIA 隐藏仍可聚焦的内容。

### 表单与错误

- 每个控件有标签和必要说明；
- 相关控件用语义分组；
- 必填、格式和限制在输入前可获得；
- 错误用文本描述、关联字段并提供修正建议；
- 提交失败保留输入，错误摘要可定位字段；
- 高风险提交支持可逆、检查或复核确认中的适当机制。

### 缩放、重排和视觉

- 文本放大到 200% 后可读可操作；
- 在 WCAG Reflow 指定条件下通常无需二维滚动；
- 文本、非文本控件与焦点有足够对比；
- 信息不只靠颜色、位置、形状或声音；
- 文本间距调整不导致内容或功能丢失；
- 悬停/焦点附加内容可关闭、可悬停且保持。

### 动态内容

- 加载、结果数、保存和错误等状态消息可被辅助技术获知；
- 状态更新不无理由移动焦点；
- 进度频率合理，不持续打断；
- 内容插入后焦点与阅读位置可预测。

## 屏幕阅读器任务测试

不要把“听起来能读”作为全部标准。执行完整任务并记录：

- 地标、标题和控件列表能否帮助定位；
- 控件名称、角色、值和状态是否准确；
- 表单错误、加载和结果是否按时出现；
- 对话框与弹层边界是否明确；
- 动态更新是否遗漏或重复；
- 完成后能否确认权威结果。

至少选择产品支持的浏览器与屏幕阅读器组合。不同组合可能有实现差异，报告中要写环境。

## 完整案例：审计成员邀请表单

### 范围与具体输入

```text
页面：项目成员 > 邀请
任务：邀请 3 个邮箱并分配“编辑者”角色
输入：li@example.com、bad-address、member@example.com
状态：member@example.com 已是成员
目标：WCAG 2.2 AA
环境：macOS/Chrome/VoiceOver；键盘；320 CSS px；200% 放大
```

### 自动检查

工具报告：角色选择按钮缺少可访问名称；错误文本对比不足；两个元素使用重复 ID。人工检查确认三项均在打开邀请对话框后存在。

### 键盘与结构检查

1. Tab 可进入邮箱多值输入，但删除标签只能鼠标点击。
2. 角色选择视觉像下拉框，实际为 `div`，不能用 Enter 或方向键。
3. 提交后焦点仍停在按钮，但顶部出现错误摘要，键盘用户不知道。
4. 对话框可用 Escape 关闭，但关闭后焦点进入页面顶部而非触发按钮。

### 屏幕阅读器检查

- 三个邮箱标签都读为“删除”，没有对象名称。
- `bad-address` 红色边框出现，但错误文本未与输入关联。
- 部分成功结果视觉显示“1 已邀请、1 已存在、1 错误”，live region 只播报“成功”。

### 问题记录

| ID | WCAG/机制 | 事实 | 影响 |
| --- | --- | --- | --- |
| A11Y-01 | 2.1.1 Keyboard / 4.1.2 | 标签删除仅指针可用 | 键盘无法修正输入 |
| A11Y-02 | 3.3.1 / 3.3.3 | 错误未关联且只靠颜色突出 | 无法定位和修正 |
| A11Y-03 | 4.1.3 | 部分结果播报为“成功” | 错误理解实际结果 |
| A11Y-04 | 2.4.3 | 关闭后焦点失去上下文 | 重复导航成本 |

### 修复输出

1. 每个标签提供按钮“删除 li@example.com”，使用原生 button。
2. 角色使用原生 select 或完整实现 Combobox/Listbox 模式。
3. 提交失败时焦点进入错误摘要，摘要链接到具体项；输入保留。
4. 部分结果消息包含完整数量与下一步，逐项状态可重新查看。
5. 对话框关闭后焦点返回“邀请成员”；成功后若进入成员列表，则进入新增结果或列表标题。

### 失败分支与验证

- 服务失败：保留所有邮箱并说明未发送任何邀请。
- 部分成功：成功项不重复提交，错误项可修正。
- 无权限：对话框不打开或明确说明申请路径，背景焦点不丢失。
- 320 CSS 像素下控件堆叠，错误和按钮无需二维滚动。

复测自动工具、键盘、VoiceOver 和固定三邮箱结果；同时检查服务端实际邀请记录，确保无障碍反馈与权威状态一致。

## 报告格式

```markdown
- ID：
- 页面/过程/状态：
- WCAG 版本、级别与成功标准：
- 环境：浏览器、辅助技术、视口
- 复现步骤：
- 实际结果：
- 预期要求：
- 用户影响：
- 证据：
- 修复建议：
- 复测：通过/失败/未测
```

不要只粘贴工具规则。报告应让开发者复现，让测试者依据成功标准复测。

## 常见错误与修正

- 只跑自动工具并报告分数：加入人工与辅助技术任务测试。
- 只测首页：按模板、状态与完整过程取样。
- 使用 ARIA 修补错误 HTML：优先原生语义与行为。
- 屏幕阅读器能朗读就算通过：检查任务、状态和结果。
- 忽略 200% 放大、重排、悬停和动态内容。
- 报告没有 WCAG 版本、级别和测试环境。
- 修复后只重跑扫描：重复真实任务并检查回归。
- 宣称单个组件通过就代表产品符合 WCAG。

## 可执行步骤

1. 确定 WCAG 版本、级别、技术、页面与完整过程。
2. 选择代表页面、模板、组件、数据和状态。
3. 运行自动与代码检查，人工确认报告。
4. 检查结构、键盘、焦点、名称、表单、缩放和动态状态。
5. 使用支持的屏幕阅读器执行固定任务。
6. 按成功标准记录复现、影响、证据与环境。
7. 修复时保持业务状态、键盘和视觉结果一致。
8. 用自动、人工、辅助技术和端到端任务回归。

## 练习与完成标准

审计一个“搜索 → 筛选 → 打开详情 → 编辑保存”过程。

完成时应满足：

- 声明 WCAG 2.2 目标级别与测试环境；
- 覆盖加载、无结果、错误、详情对话框和保存结果；
- 自动工具发现均经过人工确认；
- 只用键盘和屏幕阅读器完成；
- 验证 200% 放大、320 CSS px、对比和非颜色表达；
- 每项问题关联具体成功标准、步骤和用户影响；
- 修复后重复完整过程，不能只看扫描分数。
