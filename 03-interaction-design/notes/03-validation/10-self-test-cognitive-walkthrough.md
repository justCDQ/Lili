# 个人自测中的认知走查

个人自测认知走查是在没有独立参与者时，由设计者或学习者严格按照目标用户知识假设和任务脚本，逐步检查正确子目标、入口发现、动作理解与进展反馈。它能快速暴露明显的可学习性缺口，但检查者知道产品结构，必须控制熟悉度偏差并降低结论强度。

## 与普通自测的区别

| 普通自测 | 个人认知走查 |
| --- | --- |
| 按自己熟悉路径完成 | 先写目标用户知道与不知道什么 |
| 看到问题立即修改 | 先记录每一步事实、四问和失败过程 |
| 关注是否能完成 | 关注首次用户为什么会或不会选择正确动作 |
| 依赖自己的记忆 | 使用冻结任务、证据和判断脚本 |
| 结论容易写成偏好 | 结论限定为潜在问题与待验证假设 |

个人走查不能代表真实用户。它适合初筛、学习和迭代早期；高风险、复杂或广泛用户产品仍需其他证据。

## 四问框架

对每个必要动作回答：

1. 用户会形成正确子目标吗？
2. 用户会发现正确操作吗？
3. 用户会把操作与预期结果联系起来吗？
4. 操作后，反馈会让用户知道取得正确进展吗？

```mermaid
flowchart LR
    K["用户已有知识"] --> G["形成当前子目标"]
    G --> N["发现可用动作"]
    N --> A["理解并执行动作"]
    A --> F["从反馈判断进展"]
    F --> G
```

回答“是”也要写支持证据。没有证据时写未知，不用自己会操作作为理由。

## 控制熟悉度偏差

### 延迟执行

完成设计后间隔一段时间再走查，避免短期记忆直接提供路径。延迟不能消除长期产品知识，只是降低对刚完成细节的依赖。

### 隐藏设计稿说明

只保留用户能看到的界面，不查看交互标注、组件名、代码和流程图。走查过程中不打开后台或日志找路径。

### 使用角色卡

明确：

```text
知道：浏览器基本导航、任务对象名称、已持有输入
不知道：内部团队名、隐藏快捷键、状态码、设置位置
权限：具体角色
环境：设备、输入、语言、网络
```

### 从多个入口开始

使用通知、深链、搜索、失败恢复和首页导航，避免因为设计者记得唯一入口而漏掉真实路径。

### 反向寻找失败证据

每个“是”都问：什么条件会变成“否”？例如长文案、无权限、触屏、缩放、加载中或对象已变化。

### 交换检查

若有同伴，可交换不同模块进行走查，但不要求用户访谈。检查者仍可能具备技术知识，报告要说明。

## 准备脚本

### 冻结任务

```text
场景：收到项目即将逾期通知
目标：把任务 T-204 的截止日期改为 2026-07-22
角色：项目编辑者
设备：移动浏览器
完成：详情和刷新后均显示新日期
```

不能写“打开更多菜单并选择修改日期”。

### 冻结正确动作序列

记录当前实现的一个或多个有效路径，但走查时折叠答案。每个动作包括前置状态和可见结果。

### 准备状态

主路径之外准备无权限、无效输入、慢请求、冲突和成功。首次用户也会遇到失败，不能只检查理想状态。

## 逐步记录模板

```markdown
### 步骤 N
- 当前状态与可见信息：
- 正确动作：
- Q1 子目标：是/否/未知；证据
- Q2 发现：是/否/未知；证据
- Q3 关联：是/否/未知；证据
- Q4 反馈：是/否/未知；证据
- 可能错误动作与结果：
- 熟悉度污染：我是否因内部知识知道答案？
- 修复候选与验证：
```

## 完整案例：自测“从通知恢复失败的上传”

### 角色卡与任务

```text
知道：曾选择一个 18 MB PDF；收到“上传未完成”通知
不知道：上传任务中心位置、会话 ID、是否能续传
权限：项目编辑者
环境：Android 移动 Web，网络已恢复
目标：继续上传 roadmap-v3.pdf，并确认服务端处理完成
完成：项目中出现 V3，状态为可查看；没有重复版本
```

正确路径：通知深链 → 登录 → 上传任务详情 → 重试/续传 → 服务端处理 → 结果详情。

### 步骤 1：打开通知

- Q1：会形成“恢复上传”子目标，因为通知说明“上传未完成”。
- Q2：通知主体可点击，但操作按钮只写“查看”，未说明查看什么，发现尚可。
- Q3：通知含文件名和项目名，可以把入口与任务关联。
- Q4：打开后先进入登录，页面没有说明登录后返回上传，进展反馈未知。
- 熟悉度污染：设计者知道重定向会保留，目标用户无法从页面确认。

问题 ST-CW-01：认证中断缺少任务保留说明。候选是登录页显示“登录后继续上传 roadmap-v3.pdf”，并保留安全对象上下文。

### 步骤 2：登录后进入任务详情

- Q1：用户希望继续，子目标明确。
- Q2：页面同时有“重新选择文件”和“继续上传”；正确动作可见。
- Q3：“继续上传”与目标直接关联，但没有说明从 45% 继续还是重新开始。
- Q4：选择后进度从 45% 开始，支持正确反馈。

问题 ST-CW-02：操作后能理解，但执行前无法预测流量和时间。修复为“从 45% 继续上传”。

### 步骤 3：传输完成

- 当前界面显示“上传完成”和 100%，后台仍在解析。
- Q1：用户可能认为目标已经完成，不再形成“等待处理”的子目标。
- Q2：有一个弱视觉“查看处理状态”链接。
- Q3：术语“处理状态”没有说明文件尚不可用。
- Q4：100% 与“上传完成”反馈可能错误地暗示最终成功。

问题 ST-CW-03：阶段模型不清。应改为“文件已上传，正在处理”，保持任务 ID、进度阶段和可离开说明。

### 失败分支

处理失败时，页面显示错误码 `PARSER_42` 和“重试”。走查判断：首次用户无法把重试与“重新处理同一文件”关联，也不知道是否会重复上传。修复候选是“重新处理文件”，说明不重复创建版本，并给出可修正文件要求。

权限在登录期间撤销时，深链进入通用 404。为了防止资源泄露，系统可以不确认对象存在，但仍应说明“无法访问此项目或上传任务”及返回项目列表/申请权限路径。

### 输出

| ID | 问题 | 四问失败 | 证据 | 结论强度 |
| --- | --- | --- | --- | --- |
| ST-CW-01 | 登录中断未说明保留任务 | Q4 | 当前页面 | 自测潜在问题 |
| ST-CW-02 | 续传范围不可预测 | Q3 | 当前文案 | 自测潜在问题 |
| ST-CW-03 | 上传与处理混合 | Q1/Q4 | 状态与服务端数据 | 当前事实较强 |
| ST-CW-04 | 错误码与重试含义不清 | Q3 | 故障状态 | 自测潜在问题 |

### 外部补证

- 检查公开帮助是否区分上传与处理；
- 查支持问题中“100% 后找不到文件”；
- 用任务日志观察重复上传；
- 查看服务端重复版本数据；
- 找同伴按冻结任务执行，不提供路径。

## 自测后的修复规则

每项修复必须对应失败问题：

- Q1 子目标失败：明确当前任务、阶段和下一步。
- Q2 发现失败：调整入口位置、可见标签与状态，不只增大颜色。
- Q3 关联失败：使用准确动作、对象和后果文案。
- Q4 反馈失败：显示动作被接收、处理中、结果和恢复，并提供程序语义。

修改后重复相同角色卡、任务和状态，避免换任务后宣称修复有效。

## 与键盘和响应式结合

认知走查关注理解，但正确动作若键盘不可达，也应记录实际失败并交给无障碍审计。窄屏可能隐藏入口、改变顺序或遮挡反馈，因此至少在支持的桌面/移动布局各走一遍。

## 常见错误与修正

- 设计者以“我能找到”回答 Q2：使用角色卡与界面线索。
- 为了模拟新用户故意装作看不见：依据可见证据与合理知识，不表演困惑。
- 四问回答没有反证和失败过程。
- 走查时打开标注或代码找正确路径。
- 只检查主路径，不检查等待、错误和恢复。
- 把自测预测写成真实用户结论。
- 修改后换输入和任务，无法比较。

## 可执行步骤

1. 冻结产品版本、角色卡、任务、输入和完成标准。
2. 折叠正确路径，准备主状态与失败条件。
3. 延迟执行并隐藏设计/实现说明。
4. 对每个动作回答四问、证据和反证。
5. 记录熟悉度污染、错误动作和恢复。
6. 按失败问题生成修复规则，不直接美化页面。
7. 用相同任务、键盘和窄屏复测。
8. 通过公开证据、日志、同伴任务或其他方法补证。

## 练习与完成标准

自测“首次创建 API 密钥并安全保存”。

完成时应满足：

- 角色卡明确知道与不知道什么；
- 任务不包含设置路径或按钮名；
- 每一步完整回答四问并记录熟悉度污染；
- 覆盖权限、一次性显示、复制失败、页面离开和密钥撤销；
- 说明密钥为何不能再次明文查看；
- 键盘与屏幕阅读器路径另行验证；
- 所有结论标为当前事实或自测假设，并有补证计划。
