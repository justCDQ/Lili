# 用产品评论、公开 Issue、帮助文档、更新日志和社区问题取证

## 是什么

这些公开资料是非介入式证据：评论反映自发反馈，Issue 和社区帖包含复现与讨论，帮助文档描述官方预期流程，更新日志提供变化时间线。它们适合发现问题线索、术语、边界和版本差异，不能代表全部用户或直接给出发生率。

## 为什么需要

验证的价值在于减少未经检查的设计假设。明确这一方法的适用问题、输入条件和输出，可以避免把单次观察、评估者判断或工具结果扩张为普遍结论，并为改版前后复测提供一致基线。

## 关键特性或原则

### 判断规则

- 多来源一致可提高假设可信度，但仍不等于总体比例。
- 官方文档说明预期行为，不证明实现始终如此。
- 未关闭 Issue 不等于问题仍存在；必须核对版本。
- 记录反例和无效检索，避免只保存支持结论的材料。

## 实际怎么使用

### 实际方法

1. 固定产品、平台、版本和时间范围。
2. 用任务词、错误文案、对象名和状态词检索。
3. 保存直接链接、发布日期、版本、复现条件和原文短摘录。
4. 去重同一事件，区分报告数量与受影响人数。
5. 将资料编码为任务步骤、问题类型、严重后果、环境和解决方式。
6. 与帮助文档声明、实际产品行为和更新日志交叉核对。

## 常见错误与边界

不收集无关用户名、联系方式和敏感内容；引用时最小化个人信息。不得绕过访问控制、批量抓取违反条款的数据或将社区发言当作已授权研究材料。

## 补充知识

单一验证方法通常只能覆盖部分问题。原则性检查适合发现风险，可运行测试适合确认行为，日志与埋点适合描述已发生事件；需要判断原因或推广范围时，应组合不同证据并记录反证、样本限制和版本时效。
