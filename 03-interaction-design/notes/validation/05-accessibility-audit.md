# Accessibility Audit：无障碍审计

## 是什么

无障碍审计通过人工检查与自动工具验证界面是否可感知、可操作、可理解且兼容辅助技术。自动扫描只能覆盖部分规则，键盘流程、焦点、读屏语义和错误理解必须人工验证。

## 为什么需要

视觉稿和鼠标操作不能证明界面可被键盘、读屏、放大工具或不同感知能力的用户使用。审计把可访问性要求落实到具体页面、组件、状态和成功准则，并为修复优先级与回归测试提供证据。它也能发现设计系统中的系统性缺陷，避免同一问题在多个页面重复出现。

## 关键特性或原则

- 以 WCAG 的可感知、可操作、可理解和兼容性为总体框架，问题必须映射到具体成功准则和用户影响。
- 自动工具用于扩大规则覆盖，人工检查用于验证任务、焦点、语义、读屏输出、缩放和动态状态。
- 优先使用语义 HTML；ARIA 只补充缺失语义与状态，不能改变原生键盘行为或修复错误交互模型。
- 审计范围必须写明页面、组件、浏览器、辅助技术、版本和未覆盖条件，不能用局部通过声明整体合规。
- 修复后以同一复现步骤回归，并检查设计系统中所有相同组件实例。

## 实际怎么使用

### 基础审计步骤

1. 仅用键盘完成主任务：可达、焦点可见、顺序合理、无陷阱。
2. 检查标题层级、地标、控件名称/角色/值和表单标签。
3. 验证文本对比度，确认状态不只依赖颜色。
4. 在 200% 文本缩放和 400% 页面缩放下检查内容与功能不丢失。
5. 用读屏走主流程，确认动态状态、错误和模态标题被合理播报。
6. 检查目标大小、悬停/焦点内容、动效控制和超时。
7. 自动扫描后逐项人工确认，记录 WCAG 条款、复现和影响。

### 输出

记录组件、页面、条件、WCAG 成功准则、等级、用户影响、证据、修复和复测结果。优先修复阻断任务与跨页面系统性问题。

## 常见错误与边界

- 只跑 Lighthouse 或 axe 即宣布合规。
- 用 ARIA 修补可用原生 HTML 的控件。
- 焦点环存在但被遮挡。
- 读屏可读不代表键盘或低视力用户可用。

## 补充知识

单一验证方法通常只能覆盖部分问题。原则性检查适合发现风险，可运行测试适合确认行为，日志与埋点适合描述已发生事件；需要判断原因或推广范围时，应组合不同证据并记录反证、样本限制和版本时效。
