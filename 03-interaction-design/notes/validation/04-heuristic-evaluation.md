# 启发式评估

启发式评估由一名或多名检查者依据明确的可用性原则，独立检查界面并记录潜在问题，再合并、去重和评定影响。它是一种检查方法，能在没有招募用户时发现问题线索，但不能替代真实任务观察、无障碍合规评估或工程验证。

## 方法的输入与输出

### 输入

- 明确产品范围、版本和平台；
- 目标角色、任务、数据和环境；
- 适用启发式集合及定义；
- 可运行产品或覆盖关键状态的原型；
- 业务、权限和安全规则；
- 问题记录与严重性规则。

### 输出

- 可复现问题清单；
- 触发条件、证据、任务影响与相关原则；
- 严重性和证据强度；
- 去重后的机制分组；
- 修复候选、验证步骤和未覆盖范围。

评估输出不是“合格/不合格”总分。单个检查者会漏掉大量问题，原始研究显示汇总多名检查者能提高覆盖；具体人数不是对所有项目的保证。

## 标准流程

```mermaid
flowchart LR
    S["确定范围、任务与原则"] --> B["建立基线和测试数据"]
    B --> I["检查者独立走查"]
    I --> R["记录事实、影响与原则"]
    R --> M["合并、去重与讨论"]
    M --> P["评定严重性和证据"]
    P --> V["修复并执行任务复测"]
```

独立检查可以减少互相暗示；合并阶段再讨论相同机制和分歧。个人学习只有一名检查者时，可按不同角色、输入方式和时间分轮检查，并明确覆盖有限。

## 选择启发式集合

Nielsen 的十条启发式适合作为通用起点：系统状态、现实匹配、用户控制、一致性、错误预防、识别、效率、简洁、错误恢复、帮助。它们是广泛经验原则，不是规范性标准。

高风险或特殊领域需要增加检查项：

- 金融：金额、收款人、授权、幂等、对账与退款；
- 医疗：身份、单位、版本、警告与审计；
- 协作：权限、并发、通知、归属和冲突；
- AI：输入边界、输出不确定性、引用、复核与失败；
- Web 可访问性：另按 WCAG 和实际辅助技术评估。

不要随意改写原则后仍声称使用原始量表。记录使用的版本和自定义项。

## 确定范围与样本

### 代表性页面与状态

覆盖高频、高风险、关键入口、模板页面、复杂组件和已知问题。每个页面至少检查相关的加载、空、成功、失败、权限和响应式状态。

### 任务优先

按完整任务走查比逐页浏览更容易发现跨页面与状态问题。任务应有固定输入和完成标准，不向检查者泄露按钮路径。

### 设备与输入

至少按产品支持范围选择桌面、窄屏、键盘和辅助技术。静态截图不足以评估动态状态、焦点、等待和恢复。

## 两轮检查

### 第一轮：理解整体

执行端到端任务，理解对象、流程、术语与主要状态，不急于记录每个微小问题。

### 第二轮：逐原则检查

对每一步询问：

- 系统状态是否准确可感知？
- 对象和动作是否与任务语言匹配？
- 能否取消、返回、撤销和恢复？
- 相同功能是否一致并符合平台约定？
- 错误是否被预防、识别和修正？
- 是否要求记住不必要信息？
- 是否同时支持初次与重复任务？
- 信息是否与当前决策相关？
- 帮助是否可找到且不替代界面修复？

## 问题记录格式

```markdown
- ID：HE-001
- 上下文：版本、角色、设备、对象状态
- 任务与步骤：
- 复现：
- 事实：
- 相关启发式：
- 用户/业务影响：
- 严重性依据：
- 证据：
- 原因推断与未知：
- 修复候选：
- 验证标准：
```

标题应描述机制和结果，如“保存期间无反馈导致重复创建”，不要只写“违反系统状态可见”。

## 严重性

分别评估：

- 任务是否还能完成；
- 发生条件和影响范围；
- 用户能否发现并自行恢复；
- 数据、资金、隐私、安全和法律后果；
- 问题是否重复出现；
- 证据是当前复现还是推断。

可以使用团队量表，但每一级必须有定义。严重性不是优先级，后者还涉及修复成本、依赖和时间。

## 合并与去重

当问题具有相同触发、机制和影响时合并，即使检查者引用不同启发式。保留所有证据与影响场景。若只是表面相似但根因不同，例如“按钮不可用”分别由权限与请求故障造成，应分开。

分歧处理：

1. 回到产品复现和任务结果；
2. 查业务、平台或可访问性规则；
3. 无法确认则记录为未知，不用多数投票替代证据；
4. 建立下一步测试。

## 完整案例：项目删除流程评估

### 范围与输入

```text
产品：项目管理 Web build 2026.07.15
角色：项目管理员
对象：项目 P-204，含 12 名成员、80 个任务、3 个外部集成
设备：Chrome 1280×720；键盘；另测 320 CSS px
目标：安全删除测试项目并确认数据处理
完成：项目不可用；可恢复性、集成和审计结果明确
```

### 第一轮流程

设置 → 危险区域 → 删除项目 → 输入项目名 → 确认 → 返回项目列表。成功后出现“项目已删除”Toast。

### 第二轮发现

**HE-001：影响范围不完整**

- 事实：确认页只显示项目名，没有成员、任务、集成和恢复期限。
- 原则：现实匹配、错误预防、帮助恢复。
- 影响：管理员无法判断外部集成和数据是否永久删除。
- 严重性：高风险不可逆任务；证据为当前界面事实。

**HE-002：按钮与结果术语冲突**

- 事实：按钮写“永久删除”，帮助文档写项目进入回收站 30 天。
- 原则：一致性、现实匹配。
- 影响：用户可能误判可恢复性；实现结果需验证。

**HE-003：焦点恢复不合理**

- 事实：删除后路由到列表，焦点仍位于已卸载对话框元素，键盘下一次 Tab 从文档起点开始。
- 原则：系统状态、用户控制；同时需对照 WCAG 焦点顺序。
- 影响：键盘用户丢失上下文。

**HE-004：结果过于短暂**

- 事实：Toast 5 秒消失，项目列表无删除记录或撤销入口。
- 原则：系统状态、恢复。
- 影响：无法重新确认删除对象和恢复方式。

### 机制分组

HE-001、HE-002、HE-004 共享“删除生命周期与结果表达不完整”，可以作为一个产品/交互主题处理；HE-003 是独立焦点实现问题。

### 修复输出

1. 用准确操作“将项目移到回收站”，显示 30 天恢复期限。
2. 确认页列出对象名、成员、任务和集成影响；服务端重新验证权限与版本。
3. 结果进入持久回收站详情，显示删除时间、恢复和永久删除规则。
4. 删除后焦点进入结果标题；返回列表时进入列表标题或合理位置。
5. 永久删除作为单独高风险任务，重新认证并显示不可恢复影响。

### 失败分支

- 权限提交前撤销：不删除，保留对话框上下文并说明。
- 某外部集成清理失败：项目状态与集成清理分别显示，支持追踪。
- 请求超时：先查询项目状态，避免重复删除或显示错误成功。
- 项目被他人更新：版本冲突时重新显示最新影响范围。

### 复测

使用同一 P-204 克隆数据执行主路径与失败；检查回收站、数据库状态和审计记录；仅用键盘和屏幕阅读器完成；确认 320 CSS 像素和 200% 放大下影响信息可读。

## 方法局限与补充

- 检查者不能代表真实用户，应结合任务观察、支持问题或行为数据。
- 原则较宽，容易产生解释差异，需要具体复现和证据。
- 自动扫描不能执行启发式判断；它可以补充规则检测。
- 领域规则必须由对应专家与受控系统验证。
- 通过评估不意味着所有问题被发现。

## 常见错误与修正

- 检查范围只写“整个产品”：选择代表任务、页面与状态。
- 检查者一边讨论一边走查：先独立，后合并。
- 为十条原则强行找十个问题：只记录可复现事实。
- 问题没有任务影响和修复验证。
- 把启发式严重性当发布优先级。
- 用评估代替 WCAG 合规、性能、安全和用户证据。
- 修复后只检查截图，没有执行完整任务与失败分支。

## 可执行步骤

1. 固定范围、版本、角色、任务、输入、设备与完成标准。
2. 选择并记录启发式集合与领域扩展。
3. 准备可触发主路径与关键状态的测试数据。
4. 检查者独立执行整体与逐原则两轮检查。
5. 记录事实、影响、证据、推断和验证，不只写原则名。
6. 合并、去重、评定严重性与证据强度。
7. 对规范、安全和业务问题使用对应方法补证。
8. 修复后用原任务、键盘、响应式和故障重新验证。

## 练习与完成标准

对“邀请成员并分配角色”执行一次启发式评估。

完成时应满足：

- 范围含版本、角色、5 个固定邮箱和权限条件；
- 覆盖输入、等待、部分成功、失败、重复和无权限；
- 至少两轮检查，问题由事实而非原则名开始；
- 每项有任务影响、严重性依据、证据和修复验证；
- 相同机制问题已去重，不强求十条原则全覆盖；
- 键盘、屏幕阅读器和窄屏单独验证；
- 报告明确未覆盖范围和不能推断的结论。
