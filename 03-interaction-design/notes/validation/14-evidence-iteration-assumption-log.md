# 根据证据迭代并维护假设清单

## 是什么

证据迭代把每项设计决策连接到问题、证据、改动和复测结果。假设清单明确哪些判断尚未被充分验证，防止把设计意图写成用户事实。

## 为什么需要

验证的价值在于减少未经检查的设计假设。明确这一方法的适用问题、输入条件和输出，可以避免把单次观察、评估者判断或工具结果扩张为普遍结论，并为改版前后复测提供一致基线。

## 关键特性或原则

### 证据等级

- 直接可复现行为、可访问性失败、技术日志：证明特定条件下发生。
- 多来源公开反馈、任务日志、行为数据：加强范围或频率判断，但受样本与采集限制。
- 启发式、认知走查、竞品对比：发现潜在问题和候选方案。
- 个人偏好或单一案例：仅作为待验证线索。

## 实际怎么使用

### 迭代步骤

1. 每项问题写事实、来源、版本和适用范围。
2. 写假设：“若修改 X，将通过机制 Y 改善指标 Z”。
3. 选择最小可验证改动和护栏指标。
4. 用同一任务复测，记录支持、反驳或无法判断。
5. 更新设计、证据强度和剩余风险；保留被否定方案及原因。

### 假设条目字段

`编号 / 判断 / 证据 / 反证 / 影响 / 不确定性 / 验证方法 / 状态 / 日期`。

## 常见错误与边界

- 只记录支持方案的证据。
- 改动多项变量后归因于单一设计。
- 用“未发现问题”写成“没有问题”。
- 旧结论未标版本，产品变化后仍被引用。

## 补充知识

单一验证方法通常只能覆盖部分问题。原则性检查适合发现风险，可运行测试适合确认行为，日志与埋点适合描述已发生事件；需要判断原因或推广范围时，应组合不同证据并记录反证、样本限制和版本时效。
