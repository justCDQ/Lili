# 证据、迭代与假设日志

证据日志记录事实来源和适用范围；假设日志记录尚未确认、可被证伪的解释与预测；迭代日志记录方案、版本、验证和决策。三者把“为什么这样设计、依据是什么、后来学到了什么”连成可追踪链路，避免把旧假设当事实或重复讨论已否定方案。

## 三类记录的边界

| 记录 | 核心内容 | 不能写成 |
| --- | --- | --- |
| 证据 | 可观察事实、来源、时间、范围、质量与限制 | 没有来源的结论 |
| 假设 | 对问题机制或方案效果的可证伪预测 | “用户会喜欢”“应该更好” |
| 迭代 | 具体改动、版本、验证结果、决策与残余风险 | 只列发布日期的更新日志 |

决定也需要记录。相同证据可以支持多个候选方案，最终选择还受风险、成本和工程约束影响。

## 追踪链

```mermaid
flowchart LR
    E["证据：观察到什么"] --> H["假设：为什么/改动会怎样"]
    H --> X["验证：怎样证伪"]
    X --> R["结果：支持、否定或不确定"]
    R --> D["决策：采用、修改、停止"]
    D --> I["迭代版本与监测"]
    I --> E
```

证据和假设都有生命周期。产品版本、用户群、规则或平台改变后，旧证据可能仍真实但不再适用于当前决策。

## 证据记录

### 必要字段

```markdown
- ID：E-YYYYMMDD-NN
- 事实：
- 来源类型：产品复现/日志/支持/公开资料/任务观察/规范
- 原始位置：链接、查询、Issue 或证据文件
- 产品版本与时间：
- 角色、任务、平台与数据：
- 样本/分母/采样：
- 支持的结论：
- 不支持的结论：
- 质量与限制：
- 敏感性与访问：
- 状态：有效/需复核/已过期
```

### 证据类型

- **规范与业务规则**：说明要求和适用条件；不证明实现。
- **直接复现**：说明当前条件下行为；不证明总体频率。
- **日志与分析**：说明范围与分布；不自动解释原因。
- **支持与公开反馈**：说明问题线索和用户语言；有自选择偏差。
- **任务观察**：说明具体参与者怎样行动；小样本不代表总体。
- **代码和测试**：说明实现机制与回归；不证明用户理解。

### 证据状态

证据不会因“旧”自动无效。应说明何时需要复核：目标代码变化、业务规则变化、数据口径变化、平台升级或人群变化。

## 假设记录

### 机制假设

解释问题原因：

```text
因为上传 100% 被标为“完成”，用户把传输结束理解为最终处理完成，离开后重复上传。
```

### 方案假设

预测改动结果：

```text
如果分开“上传完成”和“处理中”，并提供持久任务入口，则重复导入下降，结果查看提高。
```

### 假设字段

```markdown
- ID：H-001
- 关联证据：E-...
- 假设：
- 适用角色/任务/条件：
- 预测结果：
- 反证条件：
- 验证方法与数据：
- 护栏：
- 截止/复核时间：
- 状态：未测/部分支持/支持/否定/不确定
```

“更直观”“更现代”无法证伪。预测应落到成功、错误、恢复、状态理解或权威数据。

## 迭代记录

### 变更

写具体行为变化，不只写组件名称：

- 旧流程与新流程；
- 哪个入口、状态、反馈和恢复改变；
- 数据、权限和工程依赖；
- 版本、开关、发布范围和迁移；
- 相关证据与假设。

### 验证

记录任务、输入、环境、主要指标、护栏、故障和可访问性。失败测试与未执行项也要保留。

### 决策

- 采用：证据支持且风险可接受；
- 迭代：部分支持或出现新问题；
- 停止：假设被否定、成本或风险不可接受；
- 推迟：缺少关键证据或依赖；
- 回滚：生产护栏失败。

“没有时间”可以作为计划约束，但不能把未验证方案标为成功。

## 状态与版本管理

### 不覆盖历史

新结论追加记录并链接旧记录，不静默改写原假设。否则无法理解当时决策依据。

### 稳定 ID

使用 `E-`、`H-`、`I-`、`D-` 等稳定编号，让 Issue、提交、设计稿和指标引用同一记录。

### 适用范围

每条记录写产品版本、平台、角色、入口和数据条件。Android 旧版本证据不能直接支持 Web 当前版本。

### 复核触发

- 关键组件或状态模型更改；
- 事件口径或数据源改变；
- 业务、法律、权限或价格改变；
- 浏览器、平台或辅助技术升级；
- 新证据与旧结论冲突。

## 证据质量与冲突

### 质量维度

- 直接性：是否直接观察相关任务和结果；
- 可复现性：条件与步骤是否完整；
- 范围：人群、版本和分母是否明确；
- 时效：当前决策条件是否仍成立；
- 独立性：多条证据是否来自独立来源；
- 数据质量：事件、采样、缺失与去重是否验证。

### 冲突处理

不要按来源数量投票。检查：

1. 两条证据是否针对同一版本、角色和任务；
2. 成功定义和数据口径是否相同；
3. 是否一个描述行为、一个描述理解；
4. 是否存在平台、权限或实验差异；
5. 能否构造同时解释两者的新假设；
6. 需要什么判别测试。

冲突未解决时将结论标为不确定。

## 完整案例：迭代批量导入结果

### 初始证据

**E-20260701-01**

```text
事实：2,000 个服务端导入任务中 1,740 个完成/部分完成，只有 1,080 个结果详情访问。
范围：Web v5.3，14 天，排除内部账户。
支持：结果查看率低于最终任务产生率。
不支持：不能证明未查看者找不到入口。
```

**E-20260701-02**

```text
事实：38 个支持工单询问“上传后去哪看”，其中 31 个来自已成功任务。
范围：同一 14 天；提交支持者有自选择偏差。
支持：至少部分用户不知道结果入口。
```

**E-20260702-03**

```text
事实：固定任务认知走查中，上传到 100% 后“完成”使检查者不再寻找处理状态。
范围：单一自测，熟悉度偏差高。
支持：阶段文案可能误导；需其他证据。
```

### 假设

**H-001：** 如果把传输、处理和最终结果分开，并在成员页显示持久任务入口，则结果查看提高，24 小时重复导入下降。

反证：改版后同口径结果查看和重复导入无变化，或护栏如任务成功、错误修正、键盘完成下降。

### 迭代 I-001

改动：

1. 100% 后显示“文件已上传，正在处理”。
2. 生成任务 ID，并在成员页展示当前导入。
3. 最终状态通知深链到任务。
4. 同内容重复上传提示现有任务。

发布：v5.4，10% 组织灰度；服务端幂等修复全量发布，不作为实验变量。

### 验证结果

```text
结果查看：62.1% → 81.4%
24h 重复导入：9.0% → 2.8%
正确完成：98.7% → 98.8%
键盘固定任务：通过 → 通过
新问题：通知数量增加，5 个支持工单要求关闭
```

这些是案例数据，真实日志应关联查询版本和样本。结果支持 H-001 的预测，但通知副作用需要新假设 H-002：“按任务合并通知且提供偏好可降低通知负担，不降低结果查看。”

### 失败分支

Android v5.3 仍显示旧状态，导致跨端证据冲突。日志发现移动端不在 Web 灰度范围，因此不是 H-001 反证；创建 I-002 跟踪移动端一致性。另一次数据管道把通知点击重复计数，修正后重算，不保留错误数字作为证据。

### 决策

**D-001：** Web 方案继续扩大灰度；前提是通知偏好和移动端状态纳入下一迭代。保留旧流程记录与回滚开关，60 天后复核支持和重复导入。

## 仓库结构建议

```text
validation/
  evidence-log.md
  assumption-log.md
  iteration-log.md
  decisions/
    d-001-import-status.md
```

也可以用 Issue 或表格管理，但必须支持稳定链接、版本历史、权限与长期可维护。公开仓库不放敏感查询、日志和个人数据，只记录脱敏摘要与受控证据位置。

## 可复制最小格式

```markdown
## E-001 证据标题
- 事实/来源/范围/限制/状态

## H-001 假设标题
- 关联证据/预测/反证/验证/护栏/状态

## I-001 迭代标题
- 版本/改动/发布/验证/副作用

## D-001 决策标题
- 选择/理由/未解决/复核触发
```

## 常见错误与修正

- 把观点写成证据：要求可观察事实与来源。
- 假设无法证伪：增加具体预测和反证。
- 只保存成功结果：记录否定、不确定和失败测试。
- 新结论覆盖历史：追加并建立链接。
- 只写版本功能列表：记录问题、证据、验证和决策。
- 数据没有范围、分母和查询版本。
- 公开日志含个人信息、凭据和内部地址。
- 假设过期仍指导决策：设置复核触发和状态。

## 可执行步骤

1. 为当前问题创建证据、假设和决策稳定 ID。
2. 记录事实来源、版本、范围、限制和敏感性。
3. 把解释和方案写成可证伪预测与护栏。
4. 选择能区分支持与反证的验证方法。
5. 在迭代记录具体行为、数据、发布和回滚。
6. 保留支持、否定、不确定和副作用结果。
7. 处理冲突时检查口径与条件，不按数量投票。
8. 设置过期、复核和后续监测触发。

## 练习与完成标准

为“自动保存”建立一组完整日志。

完成时应满足：

- 至少 3 条不同来源证据，均有版本、范围和限制；
- 假设预测数据丢失、保存理解或恢复结果，并有反证；
- 迭代记录本地、云端、离线和冲突状态变化；
- 护栏包含覆盖、重复版本、性能和无障碍状态消息；
- 包含一次失败或不确定验证并保留历史；
- 决策说明发布、回滚、残余风险和复核时间；
- 所有公开记录脱敏且链接可维护。
