# Stepper 步骤器

步骤器表达有前后依赖的多阶段任务，显示当前步骤、完成状态和允许跳转范围。它不是普通 Tabs；已访问也不等于已完成。

## 能力边界与前置知识

本文假定已经能区分站点地图、用户流程、页面状态与服务端授权。完成后应能从真实任务选择Stepper 步骤器，实现它专属的路由、状态和键盘行为，并通过深链、窄屏和辅助技术验证。

不把视觉上相似的控件自动归为Stepper 步骤器。模式成立需要信息关系、操作模型和程序化语义同时匹配。

## 选择条件

| 方案 | 适用条件 | 收益 | 主要风险 |
| --- | --- | --- | --- |
| 每步独立页面 | 需保存、深链与恢复 | URL 和历史清晰 | 页面转换较多 |
| 同页分步 | 数据短、状态只在客户端 | 反馈即时 | 刷新恢复成本 |
| 单页分组表单 | 字段可同时理解和检查 | 易比较与修正 | 长页面 |
| 任务清单 | 步骤可任意顺序完成 | 不制造虚假线性 | 需要总体完成规则 |

## 信息与交互模型

Stepper 步骤器至少维护稳定项 ID、可见名称、目标或受控内容、当前状态、权限条件和来源范围。显示顺序可以变化，但稳定 ID 不随改名或本地化改变。

### 专属机制

- 步骤名称表达任务内容，不使用“第一步”作为唯一名称。
- 已完成表示数据满足当前规则，不表示页面曾经打开。
- 前置数据变化后依赖步骤进入失效状态并重新校验。
- 普通页面步骤使用有序列表和链接，不套用 tab 键盘模型。
- 每步保存需要服务端草稿 ID、版本和可恢复入口。
- 浏览器后退与步骤返回的结果应一致，不能重复提交。
- 条件分支使总步骤数变化时不伪造固定百分比。
- 最终提交前汇总法律、金融或数据删除结果并允许返回修正。

```json
{
  "applicationId": "application-42",
  "currentStepId": "beneficial-owners",
  "version": 17,
  "steps": [
    {
      "id": "entity",
      "state": "complete"
    },
    {
      "id": "beneficial-owners",
      "state": "current"
    },
    {
      "id": "settlement",
      "state": "not-started"
    }
  ]
}
```

步骤器通常使用有序列表、文本和普通链接，不套用 tabs 语义。current 表示当前页面；complete 只有在数据满足当前规则时成立。修改前置数据后，依赖步骤进入 invalidated 并要求重新校验。每步 URL、服务端草稿版本和浏览器历史共同支持刷新与返回。

### 状态

| 状态 | 可见结果 | 用户操作或系统转换 |
| --- | --- | --- |
| 未开始 | 步骤可见但无有效数据 | 进入或按规则跳过 |
| 当前 | 标题与 URL 同步 | 输入、保存、返回 |
| 已完成 | 数据满足当前规则 | 可返回修改 |
| 失效 | 前置数据改变 | 重新进入校验 |
| 错误 | 步骤有未解决问题 | 修正且保留输入 |
| 跳过 | 条件明确不适用 | 条件改变后重新出现 |

每个状态都要定义进入和退出条件。视觉高亮、DOM 焦点、当前页面、选中面板和已执行命令是不同概念，不能用一个 `active` 布尔值同时表示。

## 路由、历史与持久化

- Stepper 步骤器中的导航目的地使用真实链接并支持新标签、复制和后退。
- 同页选择是否写入 URL，依据分享和刷新恢复需要决定。
- 临时面板关闭后恢复触发点或进入新页面的合理起点。
- 旧 URL 先完成规范化重定向，再计算当前状态。
- 用户偏好与业务当前状态分开保存；权限和路由永远重新计算。
- 会话过期后只保存安全目标路径，重新认证后重新授权。

## 键盘与无障碍

先使用与模式匹配的原生 HTML。只有Stepper 步骤器确实属于 ARIA 复合控件时才接管方向键。

验证：

- Stepper 步骤器的可见标签进入可访问名称；
- 页面导航使用链接而不是模拟点击区域；
- DOM 顺序与视觉顺序一致；
- 焦点指示清晰且不被固定层遮挡；
- 当前页、选中项、展开状态和禁用状态分别表达；
- 异步结果更新不无故抢焦点；
- 关闭临时层后焦点位置可预测；
- 320 CSS px 等效宽度和 200% 缩放下仍可操作；
- 不依赖 hover、颜色、图标或复杂手势完成任务。

### 名称、角色、状态和值

- Stepper 步骤器每个可操作项都有不依赖图标的名称。
- 同一页面存在多个 navigation 地标时按用途分别命名。
- 链接、按钮、tab 和 option 不因视觉相似而混用角色。
- aria-current 表达当前页面或步骤；aria-selected 只用于支持选择语义的控件。
- aria-expanded 只放在真实控制可折叠内容的元素上。
- aria-controls 引用的目标 ID 在当前 DOM 中存在且唯一。
- 禁用原因对键盘和读屏用户可获得，不只改变透明度。
- 动态数量或结果更新使用适合频率的状态消息，不反复打断当前阅读。

### 焦点验收

1. 从浏览器地址栏开始，只用 Tab 和Stepper 步骤器规定按键完成主任务。
2. 打开临时界面，确认焦点进入可预测起点。
3. 在异步请求返回前切换目标，确认焦点不被旧结果夺走。
4. 关闭、取消或按 Escape，确认焦点回到触发点或新页面逻辑起点。
5. 收起包含焦点的区域，确认焦点先移到仍可见的控制项。
6. 触发权限变化，确认焦点不会落在被删除入口之后的未知位置。
7. 放大并滚动，确认固定层不完全遮挡焦点。
8. 使用屏幕阅读器的地标、链接或控件列表确认名称与状态。

### 发布前检查

- Stepper 步骤器的可见名称、程序化名称和产品术语表一致。
- 默认项不依赖客户端旧缓存，能从当前 URL 或数据重建。
- 权限改变后入口、目标页面和搜索结果使用同一能力口径。
- 快速连续操作不会让旧响应覆盖新上下文。
- 浏览器后退和前进不会重复执行写命令。
- 本地化长文本、从右到左语言和系统大字体有明确布局结果。
- 关闭 JavaScript 后，属于页面导航的链接仍能到达目标。
- 埋点不保存原始搜索词、对象名称或其他敏感文本。

## 案例 1：企业开户分步收集主体、人员和结算信息

### 约束与输入

- 四步；每步服务端草稿；最终提交有法律与金融后果。
- 主体类型改变会让后续字段失效。

### 设计与实现

1. 为每步定义输入、服务端草稿和完成规则。
2. URL 包含申请 ID 和步骤 ID。
3. 返回修改主体类型后，重新计算人员与结算步骤。
4. 最终页汇总所有关键数据并允许返回修正。
5. 提交使用权威版本，未知结果按申请 ID 查询。

### 验证结果

- 刷新和跨设备恢复到首个未完成步骤。
- 已失效步骤不会继续显示完成。
- 最终提交前可检查并修正。

### 失败分支

用户把主体从企业改为个人，旧实现仍保留企业受益人并显示完成。修正为依赖图使相关步骤失效并要求复核。


## 案例 2：批量导入经过上传、映射、校验和提交

### 约束与输入

- 文件 100,000 行；上传和校验是后台任务；映射可保存。
- 校验可能部分失败。

### 设计与实现

1. 把上传、字段映射、校验预览和正式导入分成四个阶段。
2. 后台阶段使用任务 ID，不靠本地计时器。
3. 允许返回映射会创建新的校验版本。
4. 正式提交前冻结文件和映射版本。
5. 结果页按逐行结果对账。

### 验证结果

- 关闭页面后可恢复后台校验。
- 旧校验结果不能用于新映射提交。
- 部分错误有下载明细和修正路径。

### 失败分支

用户在校验后修改映射，旧实现仍允许提交旧结果。修正为映射版本变化立即使校验步骤失效。


## 工程实现

### 数据来源

导航结构来自信息架构与路由配置；权限能力来自服务端；徽标和动态状态可以独立请求。目标页面必须再次授权，不能信任模式是否显示。

### 更新顺序

1. 解析Stepper 步骤器最终规范化 URL 或当前控件上下文。
2. 获取当前主体、范围和能力。
3. 过滤不可发现项并删除无意义空组。
4. 计算当前、选中、展开和禁用状态。
5. 渲染稳定 DOM 顺序和可访问关系。
6. 异步内容返回时确认请求仍属于当前项。
7. 路由或数据变化后重新计算，不复用过期布尔状态。

### 不变量

- Stepper 步骤器的稳定项 ID 唯一；
- 当前页或选中项数量符合模式定义；
- 程序化状态与可见状态一致；
- 隐藏内容中没有可到达焦点；
- 受控对象和控制项引用均存在；
- 失效异步响应不更新新上下文；
- 权限变化后旧入口不能继续读取数据；
- 浏览器后退恢复页面而不重复写操作。

## 窄屏与长内容

窄屏方案必须重新定义布局，不只是隐藏内容：

- Stepper 步骤器在窄屏使用触发器时，触发器有清晰名称和展开状态；
- 打开后阅读顺序从模式标题到当前项再到其他项；
- 关闭方式包括明确按钮，并在适用时支持 Escape；
- 发生页面跳转后不错误恢复到旧触发器；
- 长标签允许换行或进入更适合的布局；
- 滚动容器不会让当前项和焦点长期不可见；
- 横竖屏切换不把桌面偏好错误映射为移动端打开状态。

## 调试与观测

### 外向内调试

1. 固定Stepper 步骤器的角色、URL、视口、语言、数据和功能开关。
2. 检查页面标题、当前项、可访问名称和 DOM 焦点。
3. 检查路由匹配、重定向、请求取消和乱序响应。
4. 检查权限能力与目标页面授权结果。
5. 从深链、刷新、后退和新标签页重复任务。
6. 用键盘、读屏、窄屏和长文本回归。

### 指标

- Stepper 步骤器固定任务到达正确目标的成功率；
- 第一次选择正确率；
- 选择后短时间返回或改用搜索的比例；
- 切换响应延迟和失败率；
- 深链恢复失败率；
- 键盘阻断、焦点丢失与状态未播报问题；
- 按角色、入口、平台和新旧用户分群的任务时间。

点击数不能独立证明导航改善。点击增加可能代表更常用，也可能代表用户绕路。

## 失败注入

- Stepper 步骤器当前目标在操作期间被删除；
- 角色权限被撤销；
- 请求延迟并乱序返回；
- 当前项名称增长到 40 个汉字；
- 页面在 320 CSS px 与 200% 缩放下打开；
- 会话过期后从深链恢复；
- 旧 URL 发生重定向；
- 用户用浏览器后退恢复缓存页面；
- 当前项在不可见或收起区域；
- JavaScript 加载失败，检查基础链接是否仍可用。

## 综合练习

为一个真实管理工具实现Stepper 步骤器。交付信息模型、状态表、两个不同约束的可运行原型、路由与权限契约、键盘记录、窄屏结果和失败注入报告。

验收：

- 能解释为什么选Stepper 步骤器而不是表中其他方案；
- 深链、刷新、后退和新标签页行为符合任务；
- 当前、选中、焦点、展开和禁用没有混用；
- 异步失败保留可用上下文；
- 目标页面执行独立服务端授权；
- 键盘和屏幕阅读器完成主任务；
- 窄屏、长文本和放大不隐藏唯一操作；
- 两个案例都能从界面结果对账到路由或权威数据。
