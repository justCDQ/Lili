# Top Navigation 顶部导航

顶部导航连接少量、稳定、全局范围的目的地。它帮助用户确认产品身份和顶级区域，不负责列出站点全部页面。

## 能力边界与前置知识

本文假定已经能区分站点地图、用户流程、页面状态与服务端授权。完成后应能从真实任务选择Top Navigation 顶部导航，实现它专属的路由、状态和键盘行为，并通过深链、窄屏和辅助技术验证。

不把视觉上相似的控件自动归为Top Navigation 顶部导航。模式成立需要信息关系、操作模型和程序化语义同时匹配。

## 选择条件

| 方案 | 适用条件 | 收益 | 主要风险 |
| --- | --- | --- | --- |
| 顶部链接 | 3–7 个稳定顶级区域 | 直接、可深链 | 空间受语言和账户工具限制 |
| 顶部链接 + 服务菜单 | 服务级工具与业务目的地并存 | 边界清楚 | 两个层级需要独立命名 |
| 侧边导航 | 目的地多、桌面操作频繁 | 纵向扩展更强 | 占用内容宽度 |
| 窄屏导航页 | 移动端层级深 | 空间与焦点简单 | 多一次页面转换 |

## 信息与交互模型

Top Navigation 顶部导航至少维护稳定项 ID、可见名称、目标或受控内容、当前状态、权限条件和来源范围。显示顺序可以变化，但稳定 ID 不随改名或本地化改变。

### 专属机制

- 品牌链接、业务目的地、账户工具和产品切换属于不同责任区。
- 顶级目的地数量由可扫描性与可用宽度共同决定，不存在固定通用上限。
- 当前区域由规范化路由匹配，详情子页仍映射到所属顶级区域。
- 外部链接应分组并说明新上下文，避免与站内目的地混排。
- 账户菜单不是业务导航当前项，不参与页面位置表达。
- 窄屏触发器控制完整链接集合，关闭后焦点回到触发器。
- 长标签优先换行或转换布局，不靠截断唯一名称。
- 服务名、页面标题和当前区域形成从产品到页面的定位链。

```json
{
  "productId": "workbench",
  "serviceName": "项目工作台",
  "items": [
    {
      "id": "projects",
      "label": "项目",
      "href": "/projects",
      "current": true
    },
    {
      "id": "reports",
      "label": "报表",
      "href": "/reports",
      "current": false
    }
  ],
  "accountAreaId": "account-tools"
}
```

顶部导航的业务目的地都是普通链接。产品名链接到稳定首页；账户、语言和产品切换器属于独立工具区。当前区域依据规范化路由计算，详情子路由继承所属顶级区域。窄屏菜单按钮只控制链接集合的可见性，不改变链接身份。

### 状态

| 状态 | 可见结果 | 用户操作或系统转换 |
| --- | --- | --- |
| 默认 | 品牌、顶级链接和账户区可见 | 普通链接导航 |
| 当前区域 | 一个链接使用 aria-current=page | 进入子页仍保持顶级当前项 |
| 窄屏关闭 | 显示菜单按钮与当前产品 | 打开导航 |
| 窄屏打开 | 完整链接可达 | 选择、Escape 或关闭 |
| 长文本 | 换行或切换窄屏布局 | 不覆盖账户与主内容 |
| 无权限 | 过滤受限入口且不留空位 | 目标页仍服务端授权 |

顶部导航分别计算当前产品区域、窄屏菜单展开状态、账户菜单状态和每个目的地的可发现性。当前区域来自规范化 URL，菜单展开只属于当前视口；两者不能共用 `active`。

## 路由、历史与持久化

- Top Navigation 顶部导航中的导航目的地使用真实链接并支持新标签、复制和后退。
- 同页选择是否写入 URL，依据分享和刷新恢复需要决定。
- 临时面板关闭后恢复触发点或进入新页面的合理起点。
- 旧 URL 先完成规范化重定向，再计算当前状态。
- 用户偏好与业务当前状态分开保存；权限和路由永远重新计算。
- 会话过期后只保存安全目标路径，重新认证后重新授权。

## 键盘与无障碍

先使用与模式匹配的原生 HTML。只有Top Navigation 顶部导航确实属于 ARIA 复合控件时才接管方向键。

验证：

- Top Navigation 顶部导航的可见标签进入可访问名称；
- 页面导航使用链接而不是模拟点击区域；
- DOM 顺序与视觉顺序一致；
- 焦点指示清晰且不被固定层遮挡；
- 当前页、选中项、展开状态和禁用状态分别表达；
- 异步结果更新不无故抢焦点；
- 关闭临时层后焦点位置可预测；
- 320 CSS px 等效宽度和 200% 缩放下仍可操作；
- 不依赖 hover、颜色、图标或复杂手势完成任务。

### 名称、角色、状态和值

- Top Navigation 顶部导航每个可操作项都有不依赖图标的名称。
- 同一页面存在多个 navigation 地标时按用途分别命名。
- 链接、按钮、tab 和 option 不因视觉相似而混用角色。
- 所属顶级区域的链接使用 `aria-current="page"`；窄屏菜单按钮用 `aria-expanded` 和 `aria-controls` 表达链接集合可见性。普通目的地始终是链接，不套用 tab 的 `aria-selected`。
- aria-expanded 只放在真实控制可折叠内容的元素上。
- aria-controls 引用的目标 ID 在当前 DOM 中存在且唯一。
- 禁用原因对键盘和读屏用户可获得，不只改变透明度。
- 动态数量或结果更新使用适合频率的状态消息，不反复打断当前阅读。

### 焦点验收

1. 从浏览器地址栏开始，只用 Tab 和Top Navigation 顶部导航规定按键完成主任务。
2. 打开临时界面，确认焦点进入可预测起点。
3. 在异步请求返回前切换目标，确认焦点不被旧结果夺走。
4. 关闭、取消或按 Escape，确认焦点回到触发点或新页面逻辑起点。
5. 收起包含焦点的区域，确认焦点先移到仍可见的控制项。
6. 触发权限变化，确认焦点不会落在被删除入口之后的未知位置。
7. 放大并滚动，确认固定层不完全遮挡焦点。
8. 使用屏幕阅读器的地标、链接或控件列表确认名称与状态。

### 发布前检查

- Top Navigation 顶部导航的可见名称、程序化名称和产品术语表一致。
- 默认项不依赖客户端旧缓存，能从当前 URL 或数据重建。
- 权限改变后入口、目标页面和搜索结果使用同一能力口径。
- 快速连续操作不会让旧响应覆盖新上下文。
- 浏览器后退和前进不会重复执行写命令。
- 本地化长文本、从右到左语言和系统大字体有明确布局结果。
- 关闭 JavaScript 后，属于页面导航的链接仍能到达目标。
- 埋点不保存原始搜索词、对象名称或其他敏感文本。

## 案例 1：SaaS 工作台在项目、报表、自动化和设置之间切换

### 约束与输入

- 4 个顶级区域；账户菜单独立；中英文标签；1440 与 320 CSS px。
- 路由有详情子页和查询筛选；旧 URL 先重定向。

### 设计与实现

1. 把服务名链接到工作台首页。
2. 顶级链接使用普通 anchor，并按规范化路由计算当前区域。
3. 账户与语言工具放在独立区域，不混入业务目的地。
4. 在长标签条件下先允许布局转换，再进入窄屏菜单。
5. 子路由进入时仍保持所属顶级区域当前。

### 验证结果

- 任一深链都有产品身份和一个当前顶级区域。
- Tab 顺序与视觉顺序一致；浏览器新标签打开可用。
- 旧 URL 重定向后只标记最终区域。

### 失败分支

德语标签使最后一个入口与账户按钮重叠。修正为基于可用宽度切换布局，不按固定设备宽度隐藏文字。


## 案例 2：多品牌平台在产品与账户层切换

### 约束与输入

- 三个产品共享认证；每个产品导航不同；切换产品可能丢失草稿。
- 外部帮助链接与业务链接同时存在。

### 设计与实现

1. 把产品切换器与当前产品内部导航分开。
2. 切换前检测未提交工作，提供保存、放弃或取消。
3. 外部链接分组并明确新上下文。
4. 重新认证后恢复目标产品和目标路径。
5. 按产品权限过滤切换候选。

### 验证结果

- 产品名、内部当前项和账户身份不会混淆。
- 切换产品后页面标题、导航和 URL 同步。
- 无权限产品不泄露名称或对象。

### 失败分支

用户有未保存配置时直接切换产品，旧实现静默丢失输入。修正为由编辑状态阻止导航并提供明确选择。


## 工程实现

### 数据来源

顶级目的地来自已发布产品导航配置，当前区域由规范路由映射，账户与产品切换能力由会话返回。入口过滤只处理可发现性；每个目标页面仍需服务端授权，徽标请求也不能泄露不可见产品数量。

### 更新顺序

1. 解析Top Navigation 顶部导航最终规范化 URL 或当前控件上下文。
2. 获取当前主体、范围和能力。
3. 过滤不可发现项并删除无意义空组。
4. 计算当前、选中、展开和禁用状态。
5. 渲染稳定 DOM 顺序和可访问关系。
6. 异步内容返回时确认请求仍属于当前项。
7. 路由或数据变化后重新计算，不复用过期布尔状态。

### 不变量

- Top Navigation 顶部导航的稳定项 ID 唯一；
- 当前页或选中项数量符合模式定义；
- 程序化状态与可见状态一致；
- 隐藏内容中没有可到达焦点；
- 受控对象和控制项引用均存在；
- 失效异步响应不更新新上下文；
- 权限变化后旧入口不能继续读取数据；
- 浏览器后退恢复页面而不重复写操作。

## 窄屏与长内容

窄屏方案必须重新定义布局，不只是隐藏内容：

- Top Navigation 顶部导航在窄屏使用触发器时，触发器有清晰名称和展开状态；
- 打开后阅读顺序从模式标题到当前项再到其他项；
- 关闭方式包括明确按钮，并在适用时支持 Escape；
- 发生页面跳转后不错误恢复到旧触发器；
- 长标签允许换行或进入更适合的布局；
- 滚动容器不会让当前项和焦点长期不可见；
- 横竖屏切换不把桌面偏好错误映射为移动端打开状态。

## 调试与观测

### 外向内调试

1. 固定Top Navigation 顶部导航的角色、URL、视口、语言、数据和功能开关。
2. 检查页面标题、当前项、可访问名称和 DOM 焦点。
3. 检查路由匹配、重定向、请求取消和乱序响应。
4. 检查权限能力与目标页面授权结果。
5. 从深链、刷新、后退和新标签页重复任务。
6. 用键盘、读屏、窄屏和长文本回归。

### 指标

- Top Navigation 顶部导航固定任务到达正确目标的成功率；
- 第一次选择正确率；
- 选择后短时间返回或改用搜索的比例；
- 切换响应延迟和失败率；
- 深链恢复失败率；
- 键盘阻断、焦点丢失与状态未播报问题；
- 按角色、入口、平台和新旧用户分群的任务时间。

点击数不能独立证明导航改善。点击增加可能代表更常用，也可能代表用户绕路。

## 失败注入

- Top Navigation 顶部导航当前目标在操作期间被删除；
- 角色权限被撤销；
- 请求延迟并乱序返回；
- 当前项名称增长到 40 个汉字；
- 页面在 320 CSS px 与 200% 缩放下打开；
- 会话过期后从深链恢复；
- 旧 URL 发生重定向；
- 用户用浏览器后退恢复缓存页面；
- 当前项在不可见或收起区域；
- JavaScript 加载失败，检查基础链接是否仍可用。

## 综合练习

为一个真实管理工具实现Top Navigation 顶部导航。交付信息模型、状态表、两个不同约束的可运行原型、路由与权限契约、键盘记录、窄屏结果和失败注入报告。

验收：

- 能解释为什么选Top Navigation 顶部导航而不是表中其他方案；
- 深链、刷新、后退和新标签页行为符合任务；
- 当前、选中、焦点、展开和禁用没有混用；
- 异步失败保留可用上下文；
- 目标页面执行独立服务端授权；
- 键盘和屏幕阅读器完成主任务；
- 窄屏、长文本和放大不隐藏唯一操作；
- 两个案例都能从界面结果对账到路由或权威数据。
