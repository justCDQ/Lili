# 视觉层级、间距、字体、色彩与图标系统

界面视觉系统用位置、尺寸、间距、字体、颜色、形状和图标把内容关系与操作优先级变成可感知结构。视觉层级必须服务于任务和语义；它不能修复错误的业务流程，也不能用外观替代 HTML 关系或授权状态。

## 能力边界与前置知识

本文覆盖主视觉焦点、对齐、亲密性、重复、对比、留白、字体层级、语义色彩和图标。前置知识：[信息层级与操作层级](../02-deconstruction/04-hierarchy-defaults-copy-disclosure.md)、[信息架构](../04-information-architecture/01-current-sitemap.md)和 WCAG 对颜色、对比、重排与文本缩放的要求。

设计令牌是可复用值的命名层，不是审美结论。`space-300`、`text-secondary` 和 `status-danger` 只有在组件、主题与测试中保持稳定语义才有价值。

## 从任务建立视觉顺序

先列出页面元素及其任务角色：

| 层级 | 读者要回答的问题 | 常见元素 |
| --- | --- | --- |
| 页面身份 | 我在哪里、正在处理什么 | 页面标题、对象名称 |
| 当前状态 | 对象目前怎样、是否有风险 | 状态、错误、权限、进度 |
| 主要任务 | 下一步最重要动作是什么 | 单一主操作或当前步骤 |
| 支撑信息 | 做决定需要哪些证据 | 摘要、字段、图表、附件 |
| 次要动作 | 还有哪些可选操作 | 更多菜单、辅助链接 |
| 元信息 | 谁、何时、来源是什么 | 创建者、更新时间、ID |

视觉焦点不是“页面只能有一个按钮”，而是任一任务状态下有一个最先被注意且与当前目标匹配的区域。危险错误可以暂时高于原主操作；对话框打开后，焦点层级转移到对话框。

## 视觉变量如何产生层级

### 位置

页面开头、阅读方向起点和固定区域更容易成为入口。DOM 顺序、视觉顺序和键盘顺序应保持有意义的一致；不要用 CSS 重新排序把关键内容移到视觉前方却留在读屏末尾。

### 尺寸与字重

尺寸差异帮助区分标题、正文和辅助信息。字重用于局部强调，但过多粗体会让所有内容争夺注意。标题层级应映射文档结构，不能为视觉尺寸跳过语义标题级别。

### 对比

对比可以来自明暗、尺寸、字重、形状和空间。高对比应保留给任务关键内容；“次要”不意味着低到难以阅读。文本和非文本控件仍需满足适用的对比要求。

### 空间

组内距离小于组间距离能表达关系；但真实关系要由标题、列表、fieldset 或其他语义结构支持。留白为分组和阅读提供边界，不是无条件扩大页面高度。

## 对齐、亲密性与重复

对齐建立可扫描轴线。标签、数值、小数点、表头和操作列按比较任务选择轴线；金额通常右对齐，长文本通常按阅读方向起点对齐。

亲密性规则要求相关元素更近，不相关元素更远。表单字段的标签、说明、输入和错误属于一组；若错误离下一个字段更近，用户会错误归属。

重复使同一语义保持同一表示：主操作、危险状态、链接、可编辑字段和只读值在不同页面保持一致。重复不要求所有页面相同布局；不同任务仍可采用不同结构。

## 间距系统

间距系统定义有限刻度及语义用法，例如：

| 令牌 | 示例值 | 用途 |
| --- | ---: | --- |
| `space-050` | 2px | 图标内部微调，不用于点击目标 |
| `space-100` | 4px | 紧密文字与图标 |
| `space-200` | 8px | 同一控件内部元素 |
| `space-300` | 12px | 紧密字段或列表内容 |
| `space-400` | 16px | 常规组件内边距 |
| `space-600` | 24px | 组件或字段组间距 |
| `space-800` | 32px | 页面区块间距 |
| `space-1200` | 48px | 大区域分隔，需验证窄屏 |

具体数值是项目约定，不是平台定律。选择基准后验证文本放大、翻译、密集表格、触控目标和窄屏重排。不要把所有值机械四舍五入到刻度，边框、光学对齐和可访问命中区域可能需要不同值。

## 字体系统

一个文本样式由字体族、字号、字重、行高、字间距和用途共同定义。只记录“16px”不足以复现层级。

| 样式 | 用途 | 设计约束 |
| --- | --- | --- |
| display | 少量关键展示 | 不用于长段落或表格 |
| heading-lg | 页面标题 | 与 h1 语义通常对应但不强绑定 |
| heading-md | 主要区块 | 缩放后仍与正文区分 |
| heading-sm | 组件标题 | 不能仅靠字重表达折叠状态 |
| body | 连续阅读 | 行长、行高和语言共同验证 |
| body-strong | 正文强调 | 不代替标题 |
| label | 表单与控件名称 | 保持可读，不用全大写制造密度 |
| caption | 元信息 | 仍满足对比与缩放 |
| code | 标识符与代码 | 字符区分清楚并允许换行 |

限制字体家族与样式数量，减少随机组合。中文、拉丁文、数字和符号可能由不同字体回退，必须检查基线、字重、全角标点和数字宽度。

文本放大到 200% 或系统大字体时，内容不能被裁切、重叠或丢失。固定高度按钮若装不下两行翻译，应允许增长或采用更短标签，而不是缩小文字。

## 色彩角色

色彩令牌按语义角色命名，不按具体颜色命名：

| 角色 | 用途 | 必须同时提供 |
| --- | --- | --- |
| text-primary | 主要文本 | 适用背景上的文本对比 |
| text-secondary | 辅助文本 | 仍可读，不代表禁用 |
| surface | 页面与组件背景 | 明确层级和主题变体 |
| border | 边界与分隔 | 非文本对比或其他边界线索 |
| interactive | 链接与可操作控件 | hover、focus、visited 等状态 |
| success | 成功结果 | 文本或图标名称 |
| warning | 可继续但有风险 | 后果与处理方式 |
| danger | 错误或破坏性动作 | 错误文本、图标和语义 |
| info | 中性提示 | 不与可操作蓝色混淆 |

颜色不能作为唯一的信息表达。错误字段同时需要错误文本及程序化关联；图表系列增加形状、线型、直接标签或可读取数据表；状态徽标包含文本。

WCAG 2.2 的普通文本最低对比通常为 4.5:1，大文本为 3:1；必要的用户界面组件视觉边界和图形对象适用 3:1。标志、不可用控件及附带内容存在规范例外，不能把例外扩展到普通辅助文本。

浅色、深色、高对比和系统强制颜色模式要分别验证。语义角色在主题间保持含义，具体色值可变化。

## 图标系统

图标承担识别、状态或紧凑操作提示。每个图标需要：稳定名称、尺寸、视觉边界、描边/填充规则、对齐基线、RTL 行为、可访问名称和使用限制。

图标分三类：

- 装饰图标：不增加信息，从可访问树隐藏；
- 带文字图标：文字提供名称，图标通常隐藏；
- 独立图标按钮：按钮必须有准确可访问名称，并在不熟悉或高风险时提供可见标签或持久说明。

箭头、前进、撤销等方向性图标可能在 RTL 中镜像；播放、时间、商标和某些设备形状通常不镜像。镜像规则按语义定义，不能批量翻转所有 SVG。

图标的视觉框不等于命中区域。小图标按钮应通过内边距扩大真实命中区域，并保留可见焦点。

## 案例一：项目部署详情页

### 约束与输入

- 页面包含项目名、当前环境、部署状态、错误、日志摘要和 8 个操作；
- 当前部署失败，“重新部署”是主要恢复动作；
- 删除项目是低频危险动作；
- 桌面和 320 CSS px 窄屏均使用；
- 支持文本放大与键盘。

### 处理过程

1. 页面标题表达项目与环境；失败状态紧随标题并包含文字、图标和失败时间。
2. 当前错误成为第一视觉焦点，原“部署”主操作降为恢复区中的“重新部署”。
3. 日志摘要与错误归为一组，环境元信息降低层级但保持可读。
4. 常用操作保留在页面，低频操作进入命名为“更多项目操作”的菜单。
5. 删除动作在菜单底部独立分组，不用红色填充与主要恢复动作争夺焦点。
6. 窄屏按任务顺序单列重排，DOM 不使用反向排序。

### 失败分支

团队把所有状态都放进高饱和卡片：失败、配额提示、试用推广和更新日志同时争夺注意。修正为只让阻断当前任务的失败占据高层级；其他消息按是否需要当前行动排序。

### 验证

- 首次观察者能指出页面对象、当前失败和下一恢复动作；
- 键盘顺序与视觉顺序一致，菜单有准确名称；
- 200% 文本缩放后标题、按钮和状态不裁切；
- 去除颜色后仍可区分失败、进行中和成功；
- 320px 宽度没有页面级横向滚动，日志代码块可局部滚动；
- 删除图标或颜色后，危险动作仍有明确文字。

## 案例二：高密度财务对账表

### 约束与输入

- 每页 50 行，金额、币种、差异、状态和操作列；
- 用户需快速比较数值；
- 可选舒适与紧凑密度；
- 负数、警告和选中不能只靠颜色；
- 列标题可能翻译为更长文本。

### 处理过程

1. 文本列按阅读方向起点对齐，金额右对齐，币种单独列或与金额保持固定格式。
2. 行内垂直间距由密度令牌切换，但命中区域和焦点不低于项目无障碍要求。
3. 差异值同时使用正负号、文字/图标和颜色；状态使用带文本徽标。
4. 表头与正文采用有限字体样式，数值使用支持等宽数字的字体特性时验证字体可用性。
5. 行选择使用复选框和背景变化；高对比模式仍保留边界。
6. 超长表头允许换行或调整列宽，不压缩到不可读字号。

### 失败分支

紧凑模式把行高缩到图标视觉尺寸，导致复选框和菜单难以命中。修正为将视觉密度与交互命中面积分开，必要时使用透明内边距并测试相邻目标不重叠。

### 验证

- 仅灰度或模拟色觉差异时仍能识别异常行；
- 200% 缩放和长翻译下表头可读；
- 舒适/紧凑切换不改变数据或选择范围；
- 键盘焦点在滚动容器中持续可见；
- 金额小数位、负号、空值和币种不会因字体截断；
- 固定列与原表共享语义，不生成重复朗读内容。

## 方案取舍

| 决策 | 方案 A | 方案 B | 选择依据 |
| --- | --- | --- | --- |
| 分区 | 卡片背景 | 标题与留白 | 内容密度、边界是否必要 |
| 主要动作 | 填充按钮 | 普通按钮加位置 | 页面竞争元素与品牌约定 |
| 密度 | 单一舒适模式 | 用户可切换 | 高频扫描需求和维护成本 |
| 状态 | 徽标 | 行内文本 | 状态数量、可扫描性、空间 |
| 图标操作 | 图标加文字 | 仅图标 | 熟悉度、风险、空间与可访问名称 |

视觉方案必须以任务验证，不以设计稿整齐程度决定。过度分卡会割裂阅读；过度留白会增加滚动；过度压缩会损害目标命中和理解。

## 失败与边界

- 多个填充主按钮：重新确定当前任务，或用步骤/分区隔开独立任务。
- 灰色文字代表所有次要内容：检查对比和实际重要性。
- 用 placeholder 代替标签：输入后名称消失，改用持久标签。
- 仅用红绿表达结果：增加文字、图标形状或线型。
- 所有区块都加卡片：先确认是否真的需要边界和独立容器。
- 固定高度文本容器：翻译与放大后裁切，改为内容驱动尺寸。
- 图标风格混合：统一视觉框、线宽、圆角和光学尺寸。
- 令牌按页面复制：建立语义令牌与组件令牌的覆盖关系。
- 深色主题直接反转：分别检查对比、阴影、图表和语义色。
- 视觉排序脱离 DOM：重构语义与布局，不用 tabindex 补洞。

## 验证方法

1. 模糊或缩小页面，检查第一焦点是否匹配当前任务。
2. 隐藏颜色，检查状态、图表和交互仍可理解。
3. 以 200% 文本缩放、400% 页面缩放和 320px 宽度检查重排。
4. 开启高对比/强制颜色模式，检查边界和焦点。
5. 替换长中文、英文和 RTL 文本，检查对齐与溢出。
6. 用浏览器计算样式或设计令牌审计随机值。
7. 用对比工具测量所有主题和状态组合，而非只测默认页面。
8. 对图标按钮读取可访问名称，检查重复名称是否带对象上下文。
9. 比较键盘焦点顺序、DOM 顺序和视觉顺序。
10. 用真实数据检查空值、极值、长 ID 和密集错误。

## 综合练习

重新设计“企业发票详情与对账”页面。页面包含供应商、金额、付款账号掩码、审批状态、差异表、附件、评论和 10 个操作；同时支持浅色、深色、窄屏、200% 文本和中文/英文。

交付任务层级表、间距与字体令牌、语义色矩阵、图标清单、桌面与窄屏线框、对比测试和失败状态。

验收标准：每个任务状态只有一个明确第一焦点；组内/组间关系可从语义与空间同时理解；颜色不是唯一线索；文本与控件满足适用对比；放大和翻译后无信息丢失；所有图标按钮有准确名称和命中区域。
