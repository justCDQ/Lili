# 交互设计 Roadmap：从零基础到复杂体验设计

这是一条可以独立执行的交互设计路线。学习不以用户访谈为前提，而是让“原则学习、产品拆解、重新设计、实现验证”同步进行。交互不是把页面画漂亮，而是帮助用户理解、行动、获得反馈并从错误中恢复。

## 能力阶梯

| 阶段 | 能力目标 | 代表产出 |
| --- | --- | --- |
| 入门 | 看懂界面元素、状态与基础原则 | 界面与状态分析 |
| 初级 | 能还原任务流程并发现交互问题 | 功能拆解、流程图、状态清单 |
| 中级 | 能设计完整流程与可用原型 | 中保真原型和验证报告 |
| 高级 | 能处理 B 端、协作与 AI 复杂交互 | 权限、审批或 Agent 工作区原型 |
| 专项 | 能沉淀模式并与工程共同落地 | 交互模式库和 Design System 规范 |

---

## 阶段零：交互设计基础

### 基本语言

- [ ] 用户目标、任务、场景、入口、路径和完成标准。
- [ ] 页面、弹窗、抽屉、浮层、导航、表单、列表和详情。
- [ ] 信息架构、用户流程、任务流程、状态和反馈。
- [ ] 线框图、低/中/高保真原型与设计规范的用途。
- [ ] 区分产品问题、交互问题、视觉问题和工程问题。

### 基础原则

- [ ] 心智模型、概念模型和现实世界映射。
- [ ] 可见性、可供性、反馈、一致性和约束。
- [ ] 识别优于回忆、认知负荷和信息分组。
- [ ] 用户控制、撤销、容错、错误预防和恢复。
- [ ] 渐进披露、默认值、熟悉性与学习成本。
- [ ] Fitts、Hick 等规律的适用边界，不机械套公式。

必做：为十个页面标注用户目标、第一视觉焦点、主要操作、系统反馈、错误恢复和可以删除的信息。

验收：能使用一致的设计语言解释界面为什么有效或为什么令人困惑，而不只说“好看”或“不好用”。

## 阶段一：思考、产品拆解与重新设计

每次只拆一个真实任务，把观察、推理和实践放在同一份记录中。

- [ ] 记录产品、版本、平台、日期和本次任务。
- [ ] 还原入口、前置条件、主流程、分支、异常与完成状态。
- [ ] 列出初始、空、加载、成功、失败、无权限、离线和冲突状态。
- [ ] 分析信息层级、操作层级、默认值、文案和渐进披露。
- [ ] 识别产品使用的交互模式及其适用条件。
- [ ] 用启发式原则解释优点和问题，区分事实与推断。
- [ ] 与至少一个同类产品或现实工作流比较。
- [ ] 画出重新设计的流程或线框图。
- [ ] 写出改动的收益、代价、风险和工程实现注意事项。
- [ ] 条件允许时实现关键交互，验证键盘、响应式和状态切换。

验收：连续完成 30 个任务级拆解，其中至少 10 个包含重新设计，至少 3 个实现为可交互原型或前端页面。

## 阶段二：个人可执行的研究与验证

### 无需访谈的证据来源

- [ ] 产品评论、公开 Issue、帮助文档、更新日志和社区问题。
- [ ] 竞品任务对比、历史版本对比和跨平台对比。
- [ ] Cognitive Walkthrough：模拟新用户逐步判断是否知道目标、操作和反馈。
- [ ] Heuristic Evaluation：按一致性、反馈、控制、容错等原则系统检查。
- [ ] Accessibility Audit：键盘、焦点、语义、对比度、缩放和读屏基础检查。
- [ ] 状态审计：检查正常路径之外的空、错、慢、断、冲突和权限场景。
- [ ] Dogfooding 与任务日志：真实使用并记录耗时、误操作、绕路和恢复。
- [ ] 已有埋点、热图、搜索词、错误日志和客服材料；使用时注意权限与隐私。

### 原型验证

- [ ] 为原型定义真实任务、成功条件和观察项。
- [ ] 自测时使用认知走查，不把熟悉设计当作用户一定能理解。
- [ ] 使用键盘、窄屏、慢网、空数据和错误响应主动制造边界。
- [ ] 比较改版前后步骤数、完成率、时间、错误与认知成本。
- [ ] 如果能邀请体验者，可做可用性测试；它是增强证据，不是个人学习的准入条件。
- [ ] 根据证据修改设计，并记录哪些问题仍是假设。

验收：完成一份包含原则检查、认知走查、状态审计、无障碍检查和改版对比的验证报告。

## 阶段三：信息架构

学习分类、导航、页面层级、搜索、标签、卡片分类、站点地图和渐进披露。

Todo：

- [ ] 绘制一个复杂产品的现有站点地图。
- [ ] 标记重复、混乱和层级过深的入口。
- [ ] 按任务、角色或业务对象重新分类。
- [ ] 设计顶部、侧边、Tab、面包屑和搜索的边界。
- [ ] 通过树状结构走查、查找任务或已有行为数据验证目标功能是否容易找到。

验收：能对几十个功能建立清晰分类，并解释为什么使用页面、弹窗、抽屉或行内编辑。

---

## 阶段四：用户流程与状态

学习 User Flow、Task Flow、Journey Map、主流程、分支、异常、中断和恢复。

所有功能至少检查：

- [ ] 初始。
- [ ] 空。
- [ ] 加载。
- [ ] 成功。
- [ ] 失败。
- [ ] 部分成功。
- [ ] 无权限。
- [ ] 离线。
- [ ] 数据过期。
- [ ] 取消。
- [ ] 重试。
- [ ] 并发冲突。

验收：交付不再只有正常页面，而是流程、状态、异常、恢复、反馈和下一步。

## 阶段五：交互模式库

### 导航

Sidebar、Top Navigation、Tabs、Breadcrumb、Command Palette、Stepper。

### 输入

Form、Inline Edit、Auto Complete、Tag Input、Date Picker、Upload、Rich Text Editor。

### 数据

Table、List、Card、Tree、Timeline、Dashboard、Chart。

### 反馈

Toast、Alert、Inline Error、Progress、Skeleton、Empty State、Result Page。

### 操作

Create、Edit、Delete、Batch、Drag、Undo、Retry、Save Draft。

### 协作

Comment、Mention、Share、Notification、Presence、Version History、Conflict Resolution。

每个模式记录：用户目标、适用/不适用场景、流程、状态、原则、错误案例、优秀产品、工程实现、可访问性和指标。

验收：面对需求能说明它属于哪种模式、有哪些方案、为什么选当前方案、主要风险是什么。

## 阶段六：复杂 B 端交互

### 表格

- [ ] 信息层级、默认列和列配置。
- [ ] 固定列、横向滚动、排序、筛选和分页。
- [ ] 批量操作、跨页选择和不可选行。
- [ ] 行内编辑、树形数据和保存视图。
- [ ] 导出、权限字段和大数据性能。

### 表单

- [ ] 字段分组、默认值、必填与选填。
- [ ] 校验时机、异步校验和字段依赖。
- [ ] 草稿、自动保存、离开提醒。
- [ ] 提交失败保留输入。
- [ ] 服务端错误和多人冲突。

### 权限

- [ ] 无权限、只读、字段权限、数据范围。
- [ ] 临时权限、申请权限和权限过期。
- [ ] 不只隐藏功能，还说明原因和获取方式。

### 审批

- [ ] 当前节点、下一节点、审批人和预计时间。
- [ ] 驳回、撤回、转交、加签、抄送。
- [ ] 历史、超时、异常中断和并行审批。

---

## 阶段七：视觉基础、动效与无障碍

学习视觉层级、对齐、亲密性、重复、对比、留白、字体、色彩角色、图标和动效目的。

Todo：

- [ ] 页面只有一个主要视觉焦点。
- [ ] 使用系统化间距和有限字体层级。
- [ ] 颜色不是唯一信息表达方式。
- [ ] 动效说明元素从哪来、去哪和状态如何变化。
- [ ] 高频操作避免冗长动画。
- [ ] 支持键盘、焦点和减少动画设置。

验收：能独立完成结构清晰、层级合理、可访问的中保真方案。

---

## 阶段八：AI 产品交互

### Chat 与 Streaming

- [ ] 澄清需求、多轮、停止、重试、继续。
- [ ] 首 Token、部分结果、断线、中断和恢复。

### Artifact

- [ ] 创建、局部修改、版本、Diff、恢复、导出。
- [ ] 区分用户修改和 AI 修改。

### Tool Call 与 Approval

- [ ] 展示工具、参数、影响、风险和结果。
- [ ] 高风险写操作必须确认。

### Agent Task

- [ ] 目标、计划、步骤、当前进度。
- [ ] 暂停、取消、恢复、失败步骤和人工接管。

### Citation 与 Memory

- [ ] 引用定位原文、来源日期和推断说明。
- [ ] 记忆可查看、修改、删除、关闭并说明作用范围。

验收：用户始终知道 AI 正在做什么，可以停止、修改、确认、恢复，并能判断结果可信度。
