## 1. 正文原则

文章必须让没有外部上下文的读者完成一次独立学习。正文直接讲知识，不讲作者如何搜索、参考了谁的写法、为什么修改文章或资料何时更新。

每篇围绕一个明确知识点回答：

- 是什么：严格定义、输入输出、组成和相邻概念。
- 为什么：它解决的真实问题、缺失时会发生什么。
- 怎样工作：机制、流程、状态、因果和约束。
- 怎样使用：步骤、代码、命令、模板、计算或设计操作。
- 有什么特殊行为：默认值、继承、优先级、兼容性、生命周期和例外。
- 怎样失败：常见误用、边界、反例、安全与生产风险。
- 怎样验证：观察点、测试、调试工具、指标和完成标准。

## 2. 入门与初级模板

按主题调整标题，但必须覆盖以下内容：

1. 标题和一句准确定位。
2. 概念定义及最小前置知识。
3. 工作机制或结构图。
4. 组成部分逐项解释。
5. 最小完整示例。
6. 属性、参数、方法、状态或类型明细。
7. 至少一个可运行、可计算或可执行的应用示例。
8. 示例结果、渲染结果或可观察状态。
9. 常见错误、边界和修正方法。
10. 调试或验证步骤。
11. 练习与明确完成标准。
12. `## 来源`，列出 2–5 个直接可靠来源及访问日期。

不要为了模板强行重复。如果“组成部分”和“属性明细”是同一内容，应合并为一张有解释的表格。

扩写旧文章时必须重构成一条连续的教学路径。旧版“是什么/为什么/怎么用/反例/练习”等摘要内容应合并到深化后的对应章节，不能保留整套短模板后，再在文末追加第二套概念解释、完整案例和检查表。读者不应先读一遍摘要，再读一遍同义扩展。

## 3. 中级模板

中级文章假定读者已掌握前置概念，必须增加应用深度：

1. 能力边界和前置知识链接。
2. 底层机制、关键数据流或决策过程。
3. 至少两个不同约束下的完整应用案例。
4. 方案 A/B/C 的适用条件、成本和取舍。
5. 可运行代码、完整分析、交互状态表、SQL、命令或配置。
6. 调试路径、观测信号和验证方法。
7. 性能、安全、无障碍、数据、成本或运维边界中与主题有关的部分。
8. 失败注入、异常场景或反例。
9. 与其他模块的集成方式。
10. 综合练习或小项目及验收标准。
11. 2–5 个一手来源及访问日期。

## 4. 概念展开规则

出现以下内容时逐项解释，不只罗列名称：

- 属性：作用、值域、缺失值、无效值、适用元素、继承/反射、特殊行为和示例。
- 函数或方法：签名、参数、返回值、异常、副作用、时序和示例。
- 配置项：默认值、合法值、覆盖顺序、环境差异和安全影响。
- 状态：进入条件、退出条件、允许转移、持久化和恢复。
- 指标：定义、单位、采样点、聚合方法、误读方式和决策用途。
- 算法：输入、输出、不变量、复杂度、适用数据和失败边界。
- 产品或设计框架：它是组织假设的方法，不是自然定律；说明输入证据、判断步骤和不可用场景。
- 标准或规范：区分规范要求、浏览器/平台实现、兼容性和项目约定。

篇幅由知识点决定，不用重复文字凑长度。对本路线的入门/初级文章，低于约 120 行或 8 KB 通常意味着只有摘要，必须重新检查是否遗漏机制、逐项说明、完整案例、可观察结果、错误修正和练习；中级文章低于约 200 行或 12 KB 通常不足以承载两个实质案例、方案取舍、调试和生产边界。达到长度不是完成条件，一个名称不能代替解释，一张大而空的清单不能代替教程。

完整案例不得只有一两句话。至少包含具体输入或证据、逐步处理过程、产生的输出、验证方法和一个失败分支。产品与交互主题也必须展示实际推理和决策过程，不能在固定的“证据输入/反例/验证步骤”标题下各写一行后宣告完成。

## 5. 示例规则

- 示例先小后完整，变量和业务语义保持一致。
- 代码不得使用不存在的 API、无效占位语法或危险生产命令。
- 省略代码时明确标注省略范围，不把伪代码放进具体语言围栏。
- 命令说明运行环境、前置条件和副作用。
- SQL 说明事务、锁、索引、数据量或回滚风险。
- 产品示例给出输入证据、决策过程和输出，而不是只填一张空模板。
- 交互示例覆盖加载、空、错误、成功、权限、键盘和恢复等相关状态。
- AI 示例记录模型/API类别、结构校验、失败状态、费用/Token或评估边界；不承诺模型确定正确。
- 渲染图片必须来自仓库中的可运行示例。图片格式与扩展名一致，正文使用相对路径。
- 仓库内部链接统一使用相对 Markdown 链接，不使用 GitHub 无法解析的 Obsidian Wiki Link。
- 完整 HTML 示例必须通过 Nu HTML Checker 或等价 HTML5 校验；JavaScript、TypeScript、Go、SQL、Shell 与配置示例使用对应解析器、编译器、测试或安全的 dry-run 验证。

## 6. 事实与时效审计

对每个知识点从第一性问题开始：

1. 该词准确指什么对象？
2. 规则由哪个规范、实现、产品或团队定义？
3. 结论成立需要哪些前提？
4. 是否存在默认值、例外、兼容性或版本差异？
5. 示例能否真实复现结论？
6. 来源是否直接支撑正文，而非只与主题相关？

优先级：Living Standard/正式规范/官方文档 → 原始论文/源码/官方数据 → 高质量二手资料。对快速变化的 API、浏览器能力、法律、产品接口、模型和框架必须联网复核。

正文只保留读者需要的实际状态。例如写“`hidden` 是具有 Hidden 与 Hidden Until Found 状态的枚举属性”，不要写“核对资料后发现旧教程把它写错了”。

## 7. 正文禁区

删除以下类型文字：

- “本文参考了……的写作方式”。
- “经过联网搜索/核对后发现……”。
- “这个列表以后还会变化，请以官网为准”。
- “为了完整性，我们在这里加入……”。
- “前面已经说过”“不难理解”“显而易见”“只要记住”。
- 对读者没有操作价值的更新时间、生成过程、任务分工和维护说明。
- 比喻、鸡汤、重复结论和凑篇幅的小结。

版本、日期或兼容性若会改变读者决策，应直接写入具体条目，例如“该属性目前不是 Baseline，需要特性检测”，而不是写编辑说明。

## 8. 完成检查

- [ ] 标题与 roadmap 知识点一致。
- [ ] 所有关键名词均有定义和实际作用。
- [ ] 每个属性/参数/状态有行为与边界，不是名称清单。
- [ ] 至少一个完整示例可执行或可验证。
- [ ] 中级文章至少两个实质应用案例。
- [ ] 图表确实降低理解成本。
- [ ] UI/网页示例已真实渲染并检查交互和错误日志。
- [ ] 事实、兼容性和弃用状态已联网核对。
- [ ] 没有编辑过程或写作参考说明。
- [ ] 没有“旧摘要 + 后置扩展”的双重教学路径或重复案例。
- [ ] 来源直接、可访问且数量适当。
- [ ] Markdown 链接、图片、代码围栏和相对路径有效。
- [ ] 不存在 `[[Wiki Link]]`，GitHub、Obsidian、VS Code 均可导航。
- [ ] 完整可运行示例已通过适用的语法或行为验证。
- [ ] `git diff --check` 通过。
