# 封装第三方基础设施：隔离 SDK、平台差异与供应商语义

第三方封装把请求库、监控、存储、地图和支付 SDK 放在适配器后。封装目标是隔离本项目不愿承诺的语义、错误和生命周期，而不是给每个函数机械套一层同名包装。

- React State、Context、Effect 与 TypeScript 判别联合

本文处理浏览器侧第三方依赖适配；供应商采购与服务端 SDK 不展开。

## 1. 定义、所有权与数据流

第三方封装把请求库、监控、存储、地图和支付 SDK 放在适配器后。封装目标是隔离本项目不愿承诺的语义、错误和生命周期，而不是给每个函数机械套一层同名包装。

```mermaid
flowchart LR
    A0["业务语义"] --> A1
    A1["本地端口"] --> A2
    A2["适配转换"] --> A3
    A3["供应商 SDK"] --> A4
    A4["外部副作用"]
```

第三方封装隔离供应商语义、类型、错误、初始化和销毁。有效适配器以本地能力端口为契约；机械复制 SDK 方法的薄包装仍会把升级影响传播到业务。

## 2. 关键机制

### 2.1 语义差异

端口按业务需求命名，不照抄厂商 API。

若边界缺失，薄包装仍泄漏类型。

验证：业务测试不导入 SDK。

### 2.2 初始化

SDK 加载、配置和同意状态集中管理。

若边界缺失，组件重复初始化。

验证：实例数断言。

### 2.3 错误映射

供应商错误码转本地联合。

若边界缺失，页面识别字符串。

验证：fixture 覆盖错误。

### 2.4 取消

端口提供 signal 或 dispose。

若边界缺失，卸载后回调。

验证：延迟与卸载测试。

### 2.5 隐私

适配器过滤 PII、同意和采样。

若边界缺失，任意对象发送分析。

验证：schema 白名单。

### 2.6 SSR

浏览器全局只在客户端边界访问。

若边界缺失，导入时读取 window。

验证：SSR import 测试。

### 2.7 降级

SDK 不可用时返回明确能力状态。

若边界缺失，脚本阻塞整页。

验证：拦截域名测试。

### 2.8 版本

锁定版本并运行契约测试。

若边界缺失，自动升级改变默认。

验证：升级矩阵。

### 2.9 性能

延迟加载非关键 SDK，测量主线程和网络。

若边界缺失，首屏加载全部供应商。

验证：bundle 分析。

### 2.10 可替换性

第二实现通过同一端口与契约。

若边界缺失，只存在理论接口。

验证：内存/备选适配器测试。

## 3. 从业务能力反推适配器

分析模块需要 `track(ProductViewed|CheckoutCompleted)`，不需要供应商任意字符串事件；上传模块需要 createTask、progress、cancel 和 result，不需要厂商分片对象。先用内存实现证明端口足以完成用例，再写供应商映射。适配器可保留原错误用于诊断，但向外只返回稳定错误联合。

## 4. 运行顺序与边界

1. 组合根根据运行环境、同意状态和 feature flag 初始化 SDK。

2. 初始化是幂等的；并发调用共享同一 Promise，失败可降级。

3. 每个订阅或上传任务返回 dispose/cancel，组件卸载后不回调。

4. 适配器对输入做运行时 schema 和 PII 白名单检查。

5. 版本升级运行契约 fixture、包体分析、CSP 与脚本拦截测试。

## 5. 应用案例一：分析平台

1. 业务只发送 ProductViewed 和 CheckoutCompleted 联合。

2. 映射表定义供应商名和允许字段，email 等字段运行时拒绝。

3. 未同意追踪时注入 no-op，并记录被策略丢弃的计数。

4. 脚本被拦截时结算流程继续，不 await 分析完成。

5. 切换供应商只修改适配器和契约 fixture。

结果：无同意时丢弃且有计数，不向页面暴露 vendor。

失败分支：SDK 被拦截时结算仍成功。

## 6. 应用案例二：对象存储上传

1. UploadPort 用稳定 taskId、进度事件和 AbortSignal。

2. 适配器内部管理分片、签名续期和供应商句柄。

3. 签名过期重新获取，文件类型拒绝不重试。

4. 组件卸载取消订阅，后台策略是否继续上传由端口明确。

5. 替换为内存适配器验证暂停、乱序进度和失败恢复。

结果：替换供应商不改表单。

失败分支：签名过期映射为可重新获取，而不是无限重试。

## 7. TypeScript 核心实现

下面代码只实现本主题的核心契约；网络、DOM 或存储副作用留在调用边界。

```tsx
export interface AnalyticsPort {
  track(event: { type: "ProductViewed"; productId: string } | { type: "CheckoutCompleted"; orderId: string }): void;
}
export function createAnalyticsAdapter(vendor: { send(name: string, fields: object): void }): AnalyticsPort {
  return {
    track(event) {
      if (event.type === "ProductViewed") vendor.send("product_viewed", { product_id: event.productId });
      else vendor.send("checkout_completed", { order_id: event.orderId });
    },
  };
}
```

类型检查用于排除结构错误，运行时仍需校验外部输入、测试时序并执行安全约束。

## 8. 方案选择

| 方案 | 适用条件 | 成本与限制 |
|---|---|---|
| 直接调用 | 可丢弃原型 | 耦合迅速扩散 |
| 局部适配器 | 单域单供应商 | 边界清晰 |
| 平台服务 | 多个域共享能力 | 需严格防万能接口 |

选择应以所有权、生命周期、订阅范围和失败成本为依据。引入库不能替代这些判断；库只提供实现机制。

## 9. 调试与失败注入

| 现象 | 检查 | 修正 |
|---|---|---|
| 换 SDK 仍改页面 | 类型泄漏 | 重定义端口 |
| SSR 崩溃 | 导入时访问 window | 延迟初始化 |
| 用户数据外泄 | 任意 fields | 白名单 schema |
| 脚本失败阻塞 | 关键路径 await SDK | 降级异步 |
| 重复事件 | 多次初始化 | 单实例与幂等 |
| 卸载回调 | 无 dispose | 生命周期接口 |
| 包体增长 | 静态导入 | 延迟加载 |
| 错误无限重试 | 未分类 | 本地错误联合 |

调试顺序是：确认输入事实，再检查所有者和转换，随后检查订阅与渲染，最后检查异步资源。跳过前序证据直接增加 Effect，通常会制造第二个状态源。

## 10. 性能、安全与运维边界

- 第三方脚本受 CSP 和 consent 控制。
- 密钥不能打进客户端。
- SDK 失败不得阻断核心交易。
- 事件字段白名单并脱敏。
- 升级先跑契约 fixture。
- 记录加载耗时与失败率。
- SSR 安全导入。
- 准备禁用开关和回滚版本。

生产验证至少记录一次正常路径和一次故障路径；对“封装第三方基础设施”的结论必须能关联到日志、Profile、网络记录或自动化测试。

## 11. 与其他架构模块集成

- 依赖倒置定义端口。
- 统一错误模型承接供应商错误。
- ADR 记录选择与退出成本。
- 应用组合根注入实现。

集成时先画出事实所有者，跨边界只传递稳定契约。不要为了减少一层调用而复制同一事实。

## 12. 综合练习

封装一个分析 SDK 与上传 SDK，验证拦截脚本、撤回同意、取消上传和替换实现。

### 验收标准

- [ ] 业务代码不导入供应商包或错误类。
- [ ] 脚本拦截和初始化失败不阻断核心流程。
- [ ] 同意撤回后不再发送分析事件。
- [ ] 上传覆盖取消、签名过期和乱序回调。
- [ ] 两种适配器通过相同契约测试。

## 12. SDK 生命周期故障矩阵

| 故障 | 适配器行为 | 核心业务 |
|---|---|---|
| 脚本下载失败 | 返回 unavailable，退避后有限重试 | 继续运行 |
| 初始化并发 | 共享 single-flight Promise | 不重复注册 |
| consent 撤回 | 停止发送并清理允许清理的标识 | 不受阻塞 |
| 回调晚于卸载 | dispose 后丢弃 | 不写旧组件 |
| SDK 抛未知错误 | 归一为 providerFailure 并保留脱敏 cause | 显示稳定恢复路径 |

动态加载要测量首屏是否减少字节和主线程工作，也要记录首次使用延迟。对关键能力可以预加载，但不能因此把供应商对象暴露到业务层。

## 13. 初始化、并发与销毁协议

一个地图 SDK 可能通过 script 标签异步加载，随后创建实例并注册监听器。适配器维护 `idle | loading | ready | unavailable`，所有并发 `load()` 调用共享同一个在途 Promise。加载失败后清除在途引用，是否允许再次尝试由退避策略决定。

组件取得地图会话时同时取得 `dispose()`。dispose 需要注销监听器、停止定位订阅、销毁厂商实例，并把迟到回调标记为不可发布。React Strict Mode 开发检查可能执行额外 setup/cleanup，适配器必须使创建与销毁成对且幂等。

如果 SDK 只提供全局单例，适配器仍要按消费者维护引用计数或订阅集合。最后一个消费者离开后清理本地监听；是否卸载 script 需依据 SDK 支持，不能自行删除后留下悬空全局对象。

## 14. 供应商返回值的运行时校验

第三方 TypeScript 类型描述编译期预期，不保证远端脚本、版本错配或 JSON 响应真实符合。适配器在边界校验：

- 上传结果必须有本系统允许的对象 key，不接受完整任意 URL 作为永久身份；
- 地理坐标必须是有限数字并处于合法范围；
- 分析事件只接受已注册 event type 和字段；
- 支付结果只保留 provider reference，成功与否仍由服务端确认；
- 错误码白名单外的值映射为 unknownProviderError，并保留脱敏诊断。

错误映射不能把厂商“请求已接收”误写为业务成功。上传完成还需对象可读取或服务端确认；支付前端回调更不能替代服务端 webhook 与订单状态。

## 15. CSP、隐私与降级验证

在测试环境阻止供应商域名、让 script 返回 404、让初始化超时、撤回 consent，并观察核心页面。分析和客服 SDK 应降级为 unavailable；支付等关键能力应给出明确不可用状态和安全恢复路径，不能静默假成功。

网络面板检查脚本、beacon 和请求字段；Application 面板检查 SDK 写入的 cookie/storage；Performance 检查解析执行耗时。封装层要能回答供应商在何时加载、发送什么、如何停止和失败时影响什么。

## 16. 升级与回滚步骤

供应商主版本升级先在独立分支安装并生成 lockfile、bundle 和许可证差异。运行本地端口的契约 suite，再运行真实 sandbox 集成测试。对事件字段、错误码、默认重试、cookie 和网络域名逐项比较。

发布时用适配器级 feature flag 选择旧实现或新实现，不能让业务组件分支。两种实现并行存在期间，分析类能力可做脱敏 shadow 比较；支付和写操作不能双发。指标异常时切回旧适配器，并保留新版本诊断。

回滚条件包括初始化失败率、关键交互延迟、未知错误码比例、事件丢失和 CSP 违规。达到条件后自动或人工关闭 flag，随后用 ADR/变更记录分析是否修正再上线。

Feature flag 配置本身由受控平台提供并有安全默认；读取失败时使用经过评审的旧实现或明确不可用状态，不能随机选择供应商。
