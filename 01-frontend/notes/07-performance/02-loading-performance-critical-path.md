# 加载性能：关键请求链、缓存、渲染策略与 Hydration

加载性能研究的是：用户导航到一个页面后，决定首个有意义内容、最大内容和可交互状态何时出现的依赖链。优化对象不是抽象的“请求数量”，而是该用户路径上资源发现、连接、服务器响应、下载、解析、执行、渲染和缓存命中的先后关系。

本文讨论页面导航和应用路由加载。数据库慢查询、接口业务计算和图片生产链也会影响结果，但需要与后端、CDN 和图片服务共同处理。不要把客户端骨架屏当作服务端性能问题已经解决的证据。

## 1. 从一次导航画出关键请求链

用户输入 URL 后，浏览器需要取得 HTML，解析文档并发现样式、脚本、图片、字体和模块依赖。某个资源是否影响 LCP，不只取决于它有多大，也取决于它何时被发现、是否能复用连接、是否被更高优先级资源阻塞、响应是否可缓存，以及下载完成后是否还要等待 JavaScript 执行。

```mermaid
flowchart LR
    A["导航请求 HTML"] --> B["解析 HTML"]
    B --> C["发现 CSS、LCP 图和入口脚本"]
    C --> D["连接、请求、响应与缓存"]
    D --> E["解析 CSS / 执行 JavaScript"]
    E --> F["布局与绘制 LCP 元素"]
    F --> G["Hydration 绑定交互"]
```

关键请求链（critical request chain）是完成一个用户可见目标所必须等待的最长依赖路径。它并非固定名单：首页首屏图可能是关键资源，用户未进入的编辑器页面就不是；登录后由权限决定的导航数据可能关键，也可能只影响次要区域。先在性能 trace 与网络瀑布图中标出 LCP 元素，再倒推它依赖的 HTML、CSS、图片、字体、接口和脚本，才能避免优化无关资源。

### 1.1 资源发现时间

HTML 中直接出现的 `<img src>`、样式表和普通脚本可在解析时被发现。若 LCP 图片 URL 藏在 JavaScript 请求的 JSON 中，浏览器必须先下载、解析和执行脚本，再发出图片请求；这会把图片放到链路后段。将确定的首屏内容直接放入初始 HTML，通常比“客户端拿到数据后再插入图片”更早。

`preload` 只用于浏览器难以自行足够早发现、但当前导航确定需要的资源，例如由 CSS 背景承载的首屏图或关键字体。它不是通用加速按钮：预加载错误资源会占用并发连接与带宽，使真正关键资源更晚到达。

### 1.2 优先级与竞争

浏览器会根据资源类型、发现时间、视口和实现策略安排优先级。开发者可用 `fetchpriority` 提供提示，但不能保证覆盖浏览器调度。对首屏 LCP 图提高优先级可能有益；对页面每张图片都提高优先级，会让它们彼此竞争。优先级判断必须回到 waterfall：看请求是否更早开始、是否挤占 CSS 或关键脚本、LCP 是否实际下降。

### 1.3 下载后仍可能等待

图片完成下载不等于 LCP 已发生。它还可能等待 CSS 确定尺寸和可见性、字体完成、主线程释放以进行解码和绘制，或等待客户端渲染替换服务端占位内容。优化时把“资源加载时间”和“元素渲染延迟”分开记录；只压缩图片文件而忽略渲染延迟，常常没有改善。

## 2. 关键资源逐项处理

| 资源或环节 | 应处理的问题 | 典型错误 |
| --- | --- | --- |
| HTML | 首屏内容能否随初始响应到达；TTFB 是否稳定 | 把所有实际内容留给客户端请求 |
| CSS | 关键样式是否阻塞首次正确绘制；是否包含大量未使用规则 | 将所有页面样式放进一个巨型入口文件 |
| JavaScript | 初始执行是否阻塞解析、绘制与输入 | 为了“统一”而让入口导入整个应用 |
| 图片 | 候选尺寸、格式、请求时机和布局空间是否匹配 | 用单一超大源图服务所有设备 |
| 字体 | 文本可见、字体交换和版式稳定如何平衡 | 没有回退字体与字体加载策略 |
| 第三方脚本 | 业务价值、阻塞风险、失败降级和隐私边界 | 将分析、客服、广告全部置于首屏同步执行 |
| 缓存 | 缓存键、失效、私有内容与更新策略 | 给带用户数据的 HTML 配置共享长期缓存 |

### 2.1 HTML 与服务端响应

TTFB 包含重定向、连接、服务端处理和响应开始前的等待。CDN 命中能降低静态内容的响应时间，但不能把带用户身份、购物车或权限结果的 HTML 错误地共享给其他用户。对公共文档页，HTML 可以短缓存或使用验证缓存；带内容哈希的 JavaScript、CSS、图片等静态资产可使用较长缓存期，因为内容变化会产生新 URL。

服务端渲染不是天然更快。若服务端为等待多个慢接口而延迟整个 HTML，用户仍可能看到空白；若服务端输出了主要内容却让大体积客户端脚本立即阻塞主线程，LCP 后的交互仍会变差。服务端与客户端各自承担哪些数据与状态，必须按路径测量。

### 2.2 CSS、字体与布局稳定性

外部样式表会影响页面何时能以正确样式绘制。删除未使用样式、按路由拆分样式和减少阻塞链常有价值，但“内联全部 CSS”会增大 HTML、降低缓存复用并复杂化 CSP。适合内联的是很小的、首屏不可缺少的样式；其余样式仍应以可缓存文件交付。

字体会影响文本的字形、宽度和换行。加载策略要同时观察 FCP、LCP 与 CLS：过度等待自定义字体会延迟可读文本，过早替换且度量差异大则会造成布局移动。为图片和媒体声明尺寸，为异步插入区域预留空间，才能让加载策略不以 CLS 为代价。

### 2.3 图片的候选选择

响应式图片由浏览器基于 `srcset`、`sizes`、视口、设备像素比与实现选择候选。`sizes` 写错会使浏览器选择过大的源图；只提供一个原始大图会浪费移动带宽。首屏图应有明确的 HTML `width` 和 `height` 或 CSS `aspect-ratio`，并在真实移动网络上检查实际选中的 URL、传输字节和绘制时间。

非首屏图片可以使用 `loading="lazy"`，但不要把可能立即进入视口的 LCP 图懒加载。懒加载的边界取决于布局、滚动速度和浏览器策略，应以首屏与临界滚动区域的 trace 验证，而不是把属性批量加到所有图片。

## 3. 拆包：把可延迟的代码变成真正可延迟的代码

`import()` 创建异步模块边界。路由拆包让用户首次访问订单页时不下载报表编辑器；组件拆包让低频的地图、富文本编辑器或图表在需要时再下载。拆分后仍需检查：入口是否静态导入了同一依赖，公共 chunk 是否过大，新增请求是否在关键时刻竞争，以及加载失败时 UI 是否可恢复。

```ts
// 仅在用户进入“导出报表”流程后才请求该模块。
const loadExporter = () => import('./report-exporter');

export async function exportCurrentReport(data: ReportData) {
  const { exportReport } = await loadExporter();
  return exportReport(data);
}
```

上例的边界适合低频导出功能。若导出按钮就在首屏且用户通常立即点击，过度拆分可能引入可见等待；可在用户满足明确意图后预取，或将极小的关键逻辑保留入口。不要用“chunk 更小”作为成功指标，应测量该用户路径的初始传输、LCP、点击后的等待与失败率。

## 4. 四种渲染策略的实际差异

| 策略 | 内容何时生成 | 适合的条件 | 需要承担的成本 |
| --- | --- | --- | --- |
| CSR | 客户端下载并执行 JavaScript 后 | 强私有交互、初始内容不重要或已有应用 shell | 首屏内容与 SEO 依赖脚本；慢设备执行成本高 |
| SSG | 构建期 | 公共、稳定、可预先得到的数据 | 内容更新需重新构建或增量刷新 |
| SSR | 每次请求或受控缓存时 | 请求相关内容、需要较早内容输出 | 服务端负载、缓存分层、数据等待与一致性 |
| Streaming SSR | 独立区域准备好后逐段输出 | 页面存在可并行等待区域 | 边界、错误、顺序和 hydration 更复杂 |

选择渲染策略时先列出内容的身份相关性、更新频率、首屏必要性、服务端数据可用性和缓存安全性。公共商品介绍可能适合 SSG 或缓存 SSR；用户订单列表需要 SSR 或 CSR 的私有数据路径；同一页面可以混合使用静态 shell、服务端首屏数据和客户端低频功能。

### 4.1 Streaming SSR 的边界

流式输出能让已准备好的导航和主体先到达，让慢推荐区稍后补齐。它不能消除慢接口，只能避免无关区域互相等待。每个流式区域需要 loading、error 和 retry 状态；如果推荐服务失败，主内容仍可正常使用。服务端输出与客户端首次渲染必须一致，否则 hydration 会报警、重建 DOM 或产生闪烁。

### 4.2 Hydration 是一段独立成本

Hydration 把事件处理和状态关联到已存在的 HTML。它需要下载与执行对应组件代码，也可能重做计算。页面“已经显示”不表示按钮可及时响应。对可交互首屏，分别记录 LCP 与首次点击的 INP；对纯阅读区，可将交互代码推迟到用户真正进入功能时加载。

常见 hydration 不匹配来源包括：服务端和客户端分别调用随机数或当前时间；读取只存在于浏览器的 localStorage；按窗口宽度输出不同结构；请求数据在两端不一致。正确做法是把初始数据显式传给客户端、把浏览器专属逻辑放到客户端阶段，并在开发环境将 mismatch 警告视为需要修复的错误。

## 5. 完整案例一：修复首页 LCP 图片的晚发现

问题：首页移动端 p75 LCP 为 3.6 秒，trace 中 LCP 元素是顶部活动图。网络瀑布图显示 HTML 先下载入口脚本；脚本执行后请求活动接口；收到 JSON 后组件才插入图片。图片请求在导航后一秒多才开始，压缩图片本身无法消除这段等待。

处理步骤：

1. 确认活动图在绝大多数首页访问中都存在，并确定其 URL 可由服务端首屏数据得到。
2. 将图像输出为初始 HTML 的 `<img>`，提供 `srcset`、正确 `sizes`、`width` 与 `height`。
3. 只为该图评估 `fetchpriority="high"`；不对卡片列表图片设置高优先级。
4. 仍保留活动接口给后续刷新，但初次渲染使用服务端传入的同一数据快照，避免 hydration 改写图像。
5. 在冷缓存移动网络、暖缓存二次访问和无活动图三种条件下分别采集 trace。

```html
<img
  src="/campaign/hero-1280.avif"
  srcset="/campaign/hero-640.avif 640w,
          /campaign/hero-1280.avif 1280w"
  sizes="(max-width: 720px) 100vw, 720px"
  width="1280"
  height="720"
  fetchpriority="high"
  alt="夏季活动：订单满额享配送优惠"
>
```

验证输出：Network 面板中图片 initiator 是 HTML parser，请求开始时间前移；Performance trace 中 LCP 元素仍是该图片且渲染时间下降；布局移动分数不因图片尺寸缺失增加。失败分支：如果首页有多张轮播图，不能全部 high priority；只让首张可成为 LCP 候选，其余按用户动作或可见性加载。

## 6. 完整案例二：管理台把首屏与低频编辑器分离

问题：管理员打开订单页时，富文本备注编辑器、图表库和导出代码都在入口 bundle 中。即使用户只查看订单列表，移动 CPU 上的解析和执行仍使点击筛选后的 INP 变差。

处理步骤：

1. 用 bundle analyzer 和覆盖率确认三个依赖都被入口静态引用。
2. 列表、筛选、权限提示和错误边界保留在首屏；编辑器仅在打开“编辑备注”面板后动态导入。
3. 图表在用户展开分析区域后加载；导出模块在点击导出且服务器确认可导出时加载。
4. 为每个异步边界提供 loading、加载失败、重试和取消关闭面板的状态。
5. 检查无权限用户不会请求编辑器专属数据，关闭面板后未完成 import 的结果不会更新已卸载组件。

结果的验证不只看入口包字节数：在固定移动 CPU 下比较导航的脚本执行时间、LCP、首次筛选的 INP 和打开编辑器后的等待；在弱网下确认编辑器加载失败不会阻止查看订单。若绝大多数用户一进入页面就编辑备注，拆包只是把等待从导航移到点击，可能不值得。

## 7. 缓存、CDN 与第三方脚本的生产规则

静态资源文件名包含内容哈希时，内容改变会得到新 URL，因此可使用长期 `Cache-Control: public, max-age=..., immutable`。HTML 和 API 响应是否可共享，取决于身份、cookie、地理位置、实验分组和语言等缓存键；只要响应包含用户数据，就必须避免被其他用户复用。缓存命中率提高也不一定改善首次访问，必须把冷缓存与暖缓存分开报告。

第三方脚本常引入 DNS、TLS、下载和主线程执行成本，也带来隐私与供应链风险。对每个脚本记录所有者、业务目的、加载时机、失败行为、数据字段和移除条件。分析 SDK 不可用时不应阻断下单；支付或登录 SDK 的失败则需要面向用户的恢复路径和服务端状态核对。

## 8. 调试顺序与性能预算

1. 在真实或受控代表性设备上录制导航 trace，定位 LCP 元素。
2. 在 waterfall 中标出 HTML、CSS、LCP 资源、入口脚本和阻塞接口的开始与结束。
3. 区分 TTFB、资源加载时间和元素渲染延迟，先改最长且可控的一段。
4. 修改后同时检查 LCP、CLS、INP、传输量、缓存命中和错误率。
5. 按路由、设备、网络、release 和首次/重复访问分组观察真实用户数据。

性能预算是发布约束，不是一次性目标。例如可以规定“移动端首页冷缓存 p75 LCP 不高于 2.5 秒、初始 JavaScript gzip 不高于 170KB、关键图片必须有尺寸、第三方脚本必须有所有者”。预算超标后的动作同样要明确：阻断合并、批准例外、灰度发布或建立带期限的修复任务。

## 9. 常见错误与修正

- 将所有图片 `preload`：只预加载本次导航确定关键且浏览器发现较晚的资源，其余让调度器保留带宽。
- 将所有页面改 SSR：先确认内容是否需要请求时生成，以及服务端等待是否反而扩大 TTFB。
- 只看开发机 Lighthouse：用它定位问题，再用代表性设备、冷暖缓存和真实用户分位数验证。
- 以为 `loading="lazy"` 总是更快：首屏候选使用懒加载会延迟 LCP。
- 将私有 HTML 放进共享 CDN：按身份与缓存键设计，无法安全共享时使用 private/no-store 或受控边缘逻辑。
- 忽略 hydration：首屏可见后仍需检查点击、输入和错误恢复是否及时响应。

## 10. 综合练习与验收

选择一个现有路由，完成一份加载性能报告。报告必须包含：LCP 元素、关键请求链图、冷暖缓存 waterfall、一个服务端/客户端渲染选择、一个拆包或资源优先级改动、失败分支和发布后真实用户分位数。

验收标准：

- 能说明每个首屏资源为什么需要、何时被发现、何时完成绘制。
- 能区分“文件更小”“请求更早”“元素更早绘制”和“更早可交互”。
- 缓存策略不会共享私有内容，资源更新不会被旧缓存长期遮蔽。
- 异步模块在加载、失败、取消和无权限时有可观察状态。
- 结论同时提供实验室 trace 与按版本分组的真实用户数据或明确说明尚未有足够样本。
