# 运行时性能：调度、Worker、虚拟化与内存边界

运行时性能处理页面打开后的输入延迟、计算、渲染和资源释放。优化目标是缩短用户交互到下一次绘制之间的等待，而不是孤立地降低某一个函数的耗时。

## 前置知识与范围

- [Long Task 与 Layout Thrashing](../05-browser-runtime/09-long-task-layout-thrashing.md)
- [任务切片](../05-browser-runtime/10-raf-task-yielding.md)
- [内存分析](../05-browser-runtime/12-heap-allocation-profiling.md)

本文覆盖长任务、无效渲染、高频输入、Worker、虚拟列表、Canvas 更新与内存边界。服务端数据量、权限与持久化仍由后端控制。

## 1. 交互路径

```mermaid
flowchart LR
    A["输入事件"] --> B["等待主线程"]
    B --> C["事件处理与计算"]
    C --> D["样式、布局、绘制"]
    D --> E["下一帧显示"]
```

一次慢交互可能等待前一个脚本、在回调中同步计算，或在状态更新后触发布局与绘制。Performance trace 应先定位事件、Long Task、布局和帧，再决定是否切片、缓存、虚拟化或迁移计算。仅缩短事件回调不一定缩短 INP，因为后续渲染仍可能占用主线程。

## 2. 调度与取消

长循环按时间或小批次处理，并在批次之间让出主线程。批次必须可取消：筛选条件、路由或权限改变后，旧结果不再拥有写入界面的资格。高频输入保留最后一次状态；网络请求与 Worker 消息附带递增版本号，页面只接受当前版本。

对滚动和拖拽的视觉更新，每帧最多提交一次；对搜索，取消旧请求并显示当前查询的 loading 状态。Debounce 可降低请求数量，但不能替代取消和结果版本控制；过长 debounce 会让用户看不到及时反馈。

## 3. Worker 的适用条件

Worker 适合解析、大数组筛选、排序、图形布局和编码。Worker 不能直接访问 DOM，页面与它通过消息传递。大对象的 structured clone 有复制成本；ArrayBuffer transfer 会转移所有权，转移后原发送方不能继续读取。

| 选择 | 适合 | 代价 |
| --- | --- | --- |
| 主线程 | 短小且需要立刻更新的计算 | 可能阻塞输入与绘制 |
| Worker | CPU 密集、可序列化、与 DOM 无关的计算 | 初始化、消息复制、取消与错误协议 |
| 服务端 | 数据量大、需要权限或共享结果 | 网络等待与服务端容量 |

Worker 协议应有 requestId、输入 schema、成功、失败和取消消息。组件卸载时终止 Worker 或取消任务；长期 Worker 的缓存要有所有者、大小上限和淘汰规则。

## 4. 虚拟化与无效渲染

虚拟列表只渲染视口附近行，减少 DOM、布局和绘制。它需要处理滚动偏移、overscan、动态行高、稳定 key、键盘焦点和读屏语义。overscan 太小会空白，太大又失去收益。焦点目标尚未挂载时，应先滚动并挂载，而不是让键盘导航丢失。

无效渲染常来自过宽 context、不稳定对象 props、在 render 中重复派生大集合或状态归属错误。先用 Profiler 找到“为何渲染”，再使用 memo、selector 或拆分 context。对全部组件盲目 memo 会增加比较成本，并掩盖数据流问题。

## 5. Canvas、图表与资源上限

实时图表不应每收到一条消息就完整重绘。消息写入有上限的环形缓冲区，在下一帧合并绘制；隐藏页面不维持高帧率。绘制层区分数据更新、坐标计算、脏区域与提交帧。离开路由时取消 animation frame、订阅和图表实例。

缓存、计时器、事件监听、WebSocket、Blob URL、Worker 和图表实例都需清理路径。缓存明确最大条目、字节或 TTL；Heap snapshot 的 retainer path 用来定位对象由谁持有。多次进入和离开相同页面后 retained size 持续增长，才是需要修复的泄漏证据。

## 6. 案例一：十万条日志筛选

原实现每次输入都同步过滤、排序并渲染所有行，trace 出现数百毫秒 Long Task。改造后，Worker 保存搜索索引；输入携带 requestId；页面只渲染可视窗口；清空输入和离开页面发送取消。快速从 error 改为 warn 时，旧 error 结果被丢弃。

验证：低端 CPU trace 中 Long Task 与 INP 降低；Tab 能移动到下一可见行；卸载后 Worker 被终止；消息乱序、Worker 报错和空结果都有可见状态。若复制全量日志到 Worker 的成本更高，应让 Worker 只初始化一次索引或由服务端分页筛选。

## 7. 案例二：实时监控图

原图表每条 WebSocket 消息都更新组件状态并重绘。改造后，只保留最近十分钟的环形缓冲区，每帧合并一次更新；断线重连携带游标，避免重复点；数据超过上限时按产品规则抽样。服务端保留审计原始数据，前端窗口只服务可视化。

验证：后台五分钟返回后没有积压帧；网络抖动后的重复消息不使点数翻倍；窄屏、触摸缩放、关闭路由和权限撤销均能恢复；Heap、节点数和订阅数在重复进出后稳定。

## 8. 验收

## 8. 生产检查表

输入处理只读取当前所需数据，并且昂贵工作可以取消。

状态更新不重复计算大集合，列表 key 不使用可变索引。

布局读写不在循环中交替发生，避免强制同步布局。

Worker 消息包含版本、错误类别和明确的完成状态。

页面退出后 Worker、订阅、计时器和动画帧不再写入页面。

图表绘制有每帧上限，数据窗口有最大长度。

缓存、Blob URL 与图表实例有所有者和释放时机。

## 9. 失败注入

让 Worker 延迟返回旧结果，确认旧结果不能覆盖当前筛选。

中断网络订阅后重连，确认不会重复消费消息。

将图表数据瞬间增加十倍，确认帧时间和内存仍受控。

在渲染中抛出异常，确认错误边界不让其他功能失效。

切换路由再返回，确认没有旧监听器继续更新新页面。

## 10. 指标解释

Long Task 是定位信号，不是用户体验的唯一指标。

FPS 适合连续视觉更新，不解释表单提交等待。

Heap 增长需要结合 retained object 判断。

INP 必须按设备、路由和交互类型拆分。

平均值不能替代尾部用户体验。

优化记录收益、成本和适用范围。

## 11. 状态表

| 状态 | 进入条件 | 允许动作 | 退出条件 |
| --- | --- | --- | --- |
| idle | 没有任务 | 输入、开始订阅 | 创建任务 |
| running | 当前版本任务执行 | 取消、显示进度 | 成功、失败、取消 |
| stale | 条件已变更 | 丢弃结果 | 新任务完成 |
| failed | 可恢复错误 | 重试、返回 | 新任务开始 |
| disposed | 页面卸载 | 无 | 重新挂载 |

状态表要求每一个结果都归属一个仍有效的任务。页面不应因为异步回调晚到而从 disposed 回到 running。网络重连和组件重新挂载会创建新任务，不复用旧闭包中的可变引用。

## 12. 测量方案

开发环境先用 Performance 面板记录完整交互。录制前清除不相关扩展和后台任务，选择代表性 CPU 与网络限速；录制后标记输入事件、脚本火焰图、样式计算、布局、绘制和帧。若看到连续 Long Task，不要只查看函数名称，还要确认它是否由状态更新、序列化、图片解码或第三方脚本触发。

生产环境上报 INP、Long Task 摘要、路由、release、设备类别和匿名会话维度。不要上报用户输入、完整 URL query、token 或原始业务对象。将 p75 与样本量按移动端、桌面端、首次和重复访问分开；新 release 只覆盖少量用户时，不能把全站历史样本当作该 release 的结果。

内存测量使用多轮操作：进入页面、完成交互、离开页面、等待垃圾回收机会、重复相同循环。比较的是保留对象和增长趋势，而不是一次快照中的瞬时分配。若对象由全局缓存、订阅闭包或 DOM 引用持有，沿 retainer path 找到所有者，再决定应在何处取消或淘汰。

## 13. 方案取舍

Worker 增加并发能力，但引入消息协议、错误传递和复制成本。小计算放进 Worker 可能反而提高延迟；当 trace 已显示主线程被大规模计算占满，且计算不需要 DOM 时，Worker 才是合理候选。

虚拟化减少 DOM 工作，却增加高度测量、焦点管理和滚动定位复杂度。几百行简单列表通常直接渲染更可靠；数万行且用户主要查看局部窗口时，虚拟化的收益更明确。可访问性不是后置补丁，必须在选择虚拟化时定义屏幕阅读器、键盘和查找的行为。

缓存减少重复计算，却可能保存过期或敏感数据。缓存键必须包含影响结果的条件，缓存容量和 TTL 必须受限，失效时机由数据语义决定。没有可解释失效策略的缓存往往将性能问题换成一致性问题。

## 14. 集成练习

选择一个存在卡顿的页面，写出用户动作、当前 trace、最长任务、假设和变更范围。实现一种调度或隔离方案，并为取消、错误、快速连续输入和路由离开建立测试。若使用虚拟化，记录焦点、读屏和动态高度策略；若使用 Worker，记录消息 schema、传输大小与终止时机。

验收时提交优化前后 trace、生产指标分组、Heap 对比和回滚方案。若指标没有改善，应保留实验结果，撤销不必要复杂度，而不是把未证实的优化保留在代码中。

## 15. 容量边界

为每一类前端资源写出最大值。

列表定义最大可渲染行数与分页策略。

图表定义最长时间窗口与降采样规则。

缓存定义最大字节、条目数和过期时间。

消息队列定义最大积压和丢弃策略。

上传和解析定义最大文件大小与取消方式。

这些限制应向用户显示可理解的结果。

超限时不能静默卡死页面。

## 16. 常见误判

减少一次渲染不一定改善用户输入。

更少 DOM 不一定保持正确的键盘体验。

更小 bundle 不一定减少运行时计算。

更高缓存命中不一定改善首次访问。

一次 Heap 上升不一定是内存泄漏。

一次 benchmark 胜出不代表低端设备胜出。

每个判断需要对应的 trace 或真实数据。

## 17. 与安全的边界

Worker 和缓存不保存长期凭据。

前端取消不等于服务端取消写入。

页面隐藏权限入口不等于后端授权。

性能遥测字段经过脱敏和访问控制。

第三方性能脚本同样需要失败降级。

实现评审还应写明：哪一段代码拥有任务取消权，哪个状态代表最新结果，哪些资源可以跨页面复用，哪些资源必须在会话结束后清理。把这些责任写进接口和测试，性能优化才不会在后续功能迭代中重新失效。

- 用同一交互路径比较优化前后 trace、INP、Long Task、Heap 和 DOM 节点数。
- 所有异步结果有版本、取消和错误处理，旧任务不能覆盖当前状态。
- Worker、订阅、Blob URL、图表与缓存均有所有者、上限和释放时机。
- 虚拟化保留键盘导航、焦点恢复和辅助技术可理解的状态。
