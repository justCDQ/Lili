# 搜索索引的数据库同步、更新、删除与重建

搜索索引是数据库事实的异步投影。同步设计必须处理事务提交、重复、乱序、删除 tombstone、CDC 水位、全量快照与增量衔接，并能从事实源重建。

## 1. 能力边界与机制图

```mermaid
flowchart LR
    A["事实输入"] --> B["索引/处理机制"]
    B --> C["查询或派生输出"]
    C --> D["验证与观测"]
    D --> E["失败恢复/重建"]
```

图中的输出不自动成为事实；每次写入、查询和恢复都要保留可验证的源版本、权限范围和故障状态。

## 2. Dual write

机制：应用同时写DB和搜索存在不可原子的失败窗口。

实际使用：仅作低风险优化且有对账。

主要失败：把两次成功响应当事务。

验证方法：在每个崩溃点测试。

取舍：代码少但会漏/幽灵文档。

生产实现 Dual write 时必须把输入类型、处理上限、输出状态和失败类别写入接口契约；调用方不能根据一次成功响应推导未承诺的持久性、一致性或权限语义。

## 3. Outbox

机制：业务事务内写领域事件。

实际使用：可靠传递变更意图。

主要失败：publisher先标记再发送。

验证方法：监控oldest unpublished。

取舍：增加表和publisher运维。

生产实现 Outbox 时必须把输入类型、处理上限、输出状态和失败类别写入接口契约；调用方不能根据一次成功响应推导未承诺的持久性、一致性或权限语义。

## 4. CDC

机制：从WAL/binlog捕获提交变化。

实际使用：覆盖所有数据库写者。

主要失败：忽略复制槽积压和schema演进。

验证方法：观察LSN/slot lag。

取舍：低延迟但耦合变更日志。

生产实现 CDC 时必须把输入类型、处理上限、输出状态和失败类别写入接口契约；调用方不能根据一次成功响应推导未承诺的持久性、一致性或权限语义。

## 5. External version

机制：文档保存源aggregate version。

实际使用：拒绝旧事件覆盖新状态。

主要失败：使用墙钟排序并发事件。

验证方法：乱序回放验证最高版本。

取舍：需要事实源分配单调版本。

生产实现 External version 时必须把输入类型、处理上限、输出状态和失败类别写入接口契约；调用方不能根据一次成功响应推导未承诺的持久性、一致性或权限语义。

## 6. Tombstone

机制：删除事件携带删除version。

实际使用：防旧upsert复活。

主要失败：物理删除后不保留水位。

验证方法：发送旧事件验证不复活。

取舍：保留状态有存储成本。

生产实现 Tombstone 时必须把输入类型、处理上限、输出状态和失败类别写入接口契约；调用方不能根据一次成功响应推导未承诺的持久性、一致性或权限语义。

## 7. Bulk API

机制：批量索引提升吞吐。

实际使用：回填和追赶。

主要失败：只看HTTP 200忽略逐项错误。

验证方法：解析每项status并分类重试。

取舍：吞吐高但失败隔离复杂。

## 8. Refresh

机制：提交后near-real-time可见。

实际使用：控制读可见延迟。

主要失败：每条写refresh=true压垮集群。

验证方法：测refresh/segment/merge。

取舍：即时可见换资源成本。

生产实现 Refresh 时必须把输入类型、处理上限、输出状态和失败类别写入接口契约；调用方不能根据一次成功响应推导未承诺的持久性、一致性或权限语义。

## 9. Alias

机制：逻辑名原子切索引版本。

实际使用：零停机重建切换。

主要失败：未对账就切换。

验证方法：检查count/hash/version lag。

取舍：双索引占空间。

生产实现 Alias 时必须把输入类型、处理上限、输出状态和失败类别写入接口契约；调用方不能根据一次成功响应推导未承诺的持久性、一致性或权限语义。

## 10. Snapshot+stream

机制：固定快照后接水位增量。

实际使用：全量重建不漏更新。

主要失败：用墙钟拼接产生缺口。

验证方法：记录CDC位置并演练。

取舍：实现复杂但可恢复。

生产实现 Snapshot+stream 时必须把输入类型、处理上限、输出状态和失败类别写入接口契约；调用方不能根据一次成功响应推导未承诺的持久性、一致性或权限语义。

## 11. Reconciliation

机制：周期比较事实与投影。

实际使用：发现静默漏失。

主要失败：只比总count。

验证方法：按tenant/version/hash抽样。

取舍：增加读成本但提供可信度。

生产实现 Reconciliation 时必须把输入类型、处理上限、输出状态和失败类别写入接口契约；调用方不能根据一次成功响应推导未承诺的持久性、一致性或权限语义。

## 12. 方案比较

| 方案 | 主要收益 | 关键边界 |
|---|---|---|
| 应用双写 | 低复杂 | 存在漏写窗口 |
| Outbox | 领域语义清楚 | 至少一次重复 |
| CDC | 覆盖所有写 | 物理schema耦合 |
| 全量重建 | 修复彻底 | 资源与切换成本 |
| 增量修补 | 影响小 | 可能漏未知损坏 |

## 13. 完整案例：订单实时同步

### 输入与约束

PostgreSQL订单、Kafka、OpenSearch；30秒freshness；重复乱序；删除不能复活。

### 处理步骤

1. 事务写订单version和outbox。
2. 以order_id为key发布，consumer批量upsert。
3. 索引脚本只接受更高source_version。
4. delete写version tombstone或维护删除水位。
5. 逐项解析bulk响应，暂时重试、确定错误DLQ。

### 输出

索引最终为数据库最高版本，删除稳定。

### 验证

乱序100版本最终最高；commit后崩溃不漏；bulk部分失败可恢复。

### 失败分支

HTTP bulk 200中单项429被忽略导致静默缺文档；必须逐项检查并保留event ID。

### 完成判断

案例只有在正常路径、权限边界、容量上限和故障恢复都通过自动测试或可重复操作验证后才完成；单次手工命中不能替代持续观测。

## 14. 完整案例：全量重建v4

### 输入与约束

v3 mapping错误，2亿文档，在线写持续；可用额外磁盘1.5倍。

### 处理步骤

1. 创建v4 mapping和设置，记录CDC水位P。
2. 按数据库稳定keyset快照批量写v4。
3. 应用P后的事件并追到近实时。
4. 按tenant count/version/hash与查询golden对账。
5. 短时双读canary后原子切alias，v3保留回退窗口。

### 输出

在线切换且更新不丢，旧索引可回退。

### 验证

切换前lag为零/阈值内；随机文档版本一致；磁盘watermark安全。

### 失败分支

全量开始后只消费‘现在’事件会在快照边界重复/漏；必须使用一致快照与明确change position。

### 完成判断

案例只有在正常路径、权限边界、容量上限和故障恢复都通过自动测试或可重复操作验证后才完成；单次手工命中不能替代持续观测。

## 15. 失败注入矩阵

| 注入点 | 应观察的系统行为 | 不允许的结果 |
|---|---|---|
| 输入中加入未知字段/极端长度 | cdc_lag 出现可解释变化并触发受控降级 | 静默丢数据、跨权限返回或无限重试 |
| 暂停一个处理节点 | outbox_oldest_age 出现可解释变化并触发受控降级 | 静默丢数据、跨权限返回或无限重试 |
| 让下游返回429或503 | bulk_item_errors 出现可解释变化并触发受控降级 | 静默丢数据、跨权限返回或无限重试 |
| 制造重复与乱序版本 | version_reject_old 出现可解释变化并触发受控降级 | 静默丢数据、跨权限返回或无限重试 |
| 使磁盘达到高水位 | tombstone_count 出现可解释变化并触发受控降级 | 静默丢数据、跨权限返回或无限重试 |
| 让请求在提交后断线 | refresh_time 出现可解释变化并触发受控降级 | 静默丢数据、跨权限返回或无限重试 |
| 撤销一个租户权限 | segment_merge 出现可解释变化并触发受控降级 | 静默丢数据、跨权限返回或无限重试 |
| 使后台任务积压超过SLO | reindex_docs_per_sec 出现可解释变化并触发受控降级 | 静默丢数据、跨权限返回或无限重试 |
